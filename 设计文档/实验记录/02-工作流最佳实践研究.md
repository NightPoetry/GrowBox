## 来源(公开资料)

- 业界编码 agent 的公开工程实践——explore/plan/code/commit、
  evidence-based 验证、subagent 隔离+摘要返回、auto mode 分类器、adversarial review subagent。
- Anthropic《Effective harnesses for long-running agents》(anthropic.com/engineering)——commit 进度 + 写进度文件、
  让模型用 git 回退、context window 是首要约束。
- 业界 agent 子代理模式的公开实践——子代理"在独立上下文探索、向调用方简洁汇报摘要";写文件工具"先读再写"。
- destructive_command_guard 等开源破坏性命令防护实践——破坏性命令分类 + `bash -c`/`python -c` 递归分析 +
  引号内是数据非执行 + fail-open。
- 论文《Building Effective AI Coding Agents for the Terminal》(arxiv 2603.05344)——脚手架/上下文工程/防御纵深。

## 核心结论(与 GrowBox 栈函数 v2 的对应)

1. **context window 是首要约束**,性能随填满而退化 → GrowBox v2 的 **isolated 裁剪上下文 + 退出即丢 + digest 返回**
   正是业界公认"最强上下文工程模式"(subagent 隔离 + 摘要返回)。**最少充分信息原则**得到一手印证。
2. **explore → plan → code → commit**(四相,先探索再动手,避免解错问题);**小而清晰的改动跳过 plan**
   ("若一句话能描述 diff 就别 plan")。
3. **evidence-based 验证**:给一个能跑的检查(测试/构建/截图),让 AI 跑了给出**证据**而非断言成功;
   **针对根因不治标**;**对抗式 review**(新上下文只看 diff + 标准,只报影响正确性的缺口,别报风格)。
4. **命令安全 = 防御纵深 + 模式匹配 + 上下文感知**:递归看 `bash -c`/`python -c`/heredoc 内部真实命令;
   引号内是数据;高风险类别(rm -rf/git reset --hard/force push/DROP/TRUNCATE/FLUSHALL/kubectl delete/terraform destroy)。
5. **read before edit**(改前先读);**commit 描述性信息 + 进度文件**(便于回退/续跑)。

## 提炼:可实例化的工作流模板(经 define_workflow 创建,作用域据用途选)

> 通用的内置(已落地):`command_safety`(见 `workflow_store.rs`,旗舰示范,栈调用子函数)。
> 下列为**模板**(AI 据项目/用户意图经 `define_workflow` 自建,或后续按需内置)。节点 `tools` 子集物理收窄 = 结构性约束。

### 模板 A:auto_coding(探索→计划→实现→验证→提交)

强制顺序四相,每相只给该相工具 + 引导。phase 间靠"该相的特征工具调用"流转(本模型限制:推理相推进
需一个特征动作触发;explore→plan 用一次 file_read 后的 Always 或显式"plan 完成"动作)。建议节点:
- `explore`{tools:[file_read,file_list,shell(只读 grep/git log)],引导:"只读不改,读相关文件/历史搞清现状与模式,别动手"}
- `plan`{引导:"产出详细计划:要改哪些文件、顺序、边界、怎么验证。小改动可跳过本相"}
- `implement`{tools:[file_edit,file_write,file_read,shell],引导:"按计划写;改前先读现有文件(read before edit)"}
- `verify`{tools:[shell,file_read],引导:"跑测试/构建,给出证据(命令+输出),失败治根因不治标;可栈调用 review 子函数对抗复查"}
- `commit`{tools:[shell],引导:"描述性 commit 信息;执行任何 shell 前先栈调用 command_safety 审一遍"}

### 模板 B:read_before_write(改前必读)

固化"先读取什么再读取什么、干完写文档":
- `gather`{tools:[file_read,file_list],引导:"先读目标文件 + 同类既有实现(参照模式),再读相关接口/调用方"}
- `edit`{tools:[file_edit,file_write,file_read],next:[{on:file_edit→done}],引导:"据已读模式改"}
- `done`{引导:"改完写/更新文档(如该函数/模块的说明),再 finish"}

### 模板 C:verify / adversarial_review(子函数,栈调用)

调用方 `input=diff/改动描述 + 验收标准`,`context_mode=isolated`(只看 diff,不带写它的推理 = 无偏审查),
`return_spec="返回 {gaps:[...], blocking:bool}"`:
- `review`{tools:[file_read,shell(跑测试)],引导:"新上下文只看 input 的 diff+标准;只报**影响正确性/违背要求**的缺口,
  不报风格;给证据。完用 workflow_return 返回 {gaps, blocking}"}

## 提示词设计原则(从研究提炼,用于打磨节点引导)

- **直接、给具体上下文**:点名文件/接口/参照模式;描述症状+可能位置+"修好"的样子。
- **结构性优于劝诫**:能用节点工具子集物理锁死的,就别靠提示词"劝"(GrowBox 推论1)。
- **最少充分信息**:节点引导只放该步必需;返回只回错误/警告/结论(见 07 最高原则)。
- **要证据不要断言**:验证相要求贴命令+输出。
- **小事跳流程**:一句话能描述的改动直接做,别套工作流(一句话能描述 diff 就别 plan)。
