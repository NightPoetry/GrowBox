# 感知告知 · 双受众架构

> 真理来源。用户决策见 `用户决策/决策日志.md` 2026-06-02「所有提示类按自我感知原则,感知=对内、显示=对外」。
> 关联:`异常告知.md`(本系统的一个特例:严重异常)、`设计/00-交互层`(自我感知原则)、内部状态感知层(`Memory::perceive`)、`i18n` 两套语言轴。

## 原则

**任何"提示类"信息都有两个受众,必须同时送达,且来自同一事实源。**

- **对内(感知 / perceive)**:AI 必须能感知到一切发生的提示(自我感知原则)。用 **prompt_lang(zh/en)** 渲染,进 `Memory::perceive` → 内部状态块 + 时间线,LLM 下一回合即可见。
- **对外(显示 / display)**:用户看到的告知。用 **ui_lang(zh-CN/en/ja/zh-TW)** 渲染,进 toast / 常驻指示灯 / 横幅。

一句话:**感知的就是对内的,显示的就是对外的**。同一件事,两种渲染(口吻可不同:对外简洁安抚,对内含技术细节与可执行建议),但**绝不散落维护两套翻译** —— 一个 code,一处条目,co-locate 两受众全部语言,加守卫防退化。

> 这是把"异常告知"(`异常告知.md`)从"严重异常"推广到"一切提示";异常只是 `severity=error/fatal` 的特例。

## 本质:两个正交维度(2×2)

整套架构本质就是**两个互相独立的维度**的叉乘,每个格子 = 一条送达管道(同一个 `code`、同一组运行时 params):

| | **对外显示(human / ui_lang)** | **对内感知(llm / prompt_lang)** |
|---|---|---|
| **前端产生**(UX:校验/连接/项目/工具…) | 本地 `showToast` 即时弹(无往返) | `invoke("perceive_notice")` 回后端 → `Memory::perceive` |
| **后端产生**(失败/健康/工具结果…) | `emit "notice"` 事件 → 前端取 human 渲染 | 本地 `Memory::perceive` 直接渲染 |

- **维度 A = 受众**:对外(用户看,4 ui_lang,口吻安抚)/ 对内(LLM 感知,2 prompt_lang,口吻带情境+可执行)。
- **维度 B = 来源**:前端产生(就近显示、跨语言回后端感知)/ 后端产生(就近感知、跨进程发事件显示)。
- 四个格子**全部指向同一条 catalog 记录**——这就是"单一 code、一处维护、两受众全语言"的根因:维度再多,事实源只有一份。

**落地分期正好按格子推进(三块地,同一个原则):**
- **Phase 1** = 填【后端产生 × 对内感知】格(后端失败本地 perceive)。✅
- **Phase 2** = 填【前端产生 × 对外+对内】两格(前端 toast 显示 + 回后端感知)。✅
- **Phase 3** = 填**最后一格【后端产生 × 对外显示】**。✅ **已全部落地**:① health 部分=常驻指示灯四国化;② 瞬态 toast 子surface=`notify::emit_notice(app,code,params)` 发 `"notice"` 事件 + 前端 `startNoticeListener`→`displayNotice`(只显示不感知),**首个真实消费者 = 后台任务完成/失败**(`task.completed`/`task.failed`,Supervisor `drain_finished` 后逐条 emit;对内由 Supervisor 回合负责故 perceive=false)。**2×2 四格全闭合。**

## 单一事实源 = `prompts/notices.i18n.json`

镜像 `tools.i18n.json`(`include_str!` 内嵌后端 + 前端直接 import 同一文件,绕开打包路径坑)。每条提示按 **`code` 唯一**:

```jsonc
"project.switched": {
  "severity": "success",        // info|success|warn|error(显示样式 + 健康分级映射)
  "surface": "toast",           // toast(瞬态) | health(常驻指示灯) | silent(只感知不显示)
  "perceive": true,             // LLM 是否感知(默认 true=自我感知原则;纯 UI 噪音可显式 false)
  "human": {                    // 对外:4 ui_lang
    "zh-CN": "已切换到 {name}", "en": "Switched to {name}", "ja": "...", "zh-TW": "..."
  },
  "llm": {                      // 对内:2 prompt_lang(口吻给 LLM:含上下文/可执行)
    "zh": "用户切换到了项目「{name}」,你现在工作在该项目上下文中。",
    "en": "User switched to project \"{name}\"; you now operate in that project's context."
  }
}
```

- `human` = 4 ui_lang;`llm` = 2 prompt_lang。占位符 `{name}` 等在发出时以**同一组运行时参数**替换两受众。
- `perceive:false` 的纯 chrome(如"已复制")可省 `llm`。
- 翻译沿用 `scripts/translate-i18n.mjs`(deepseek-v4-pro):扩展为同时译本文件的 human(4)+ llm(2)。

## 一个原语 `notify(code, params)`,两条渲染

提示来源有二(后端事件 / 前端 UX),用**共享 code + 共享 catalog** 统一。每条提示都同时打到两受众:

```
后端原语 Notifier::notify(code, params):
  ① 若 perceive → 取 llm[prompt_lang] 填参 → Memory::perceive(code, text)   (对内)
  ② emit "notice" 事件 { code, params, severity, surface }                   (→ 前端显示对外)

前端 notify(code, params):
  ① 取 human[ui_lang] 填参 → toast / 健康指示灯                              (对外,本地即时,无往返延迟)
  ② 若 perceive → fire-and-forget invoke("perceive_notice", {code, params})  (→ 后端渲染 llm + 感知)

前端监听 "notice" 事件 → 取 human[ui_lang] 渲染(后端来源的提示走这条显示)
```

- **后端来源**(失败/健康/工具):后端感知(本地)+ 前端显示(经事件)。
- **前端来源**(校验/连接 UX/idle):前端显示(本地即时)+ 后端感知(经命令)。
- catalog 两端都可读(Rust `include_str!` / 前端 import),所以两端都能就地渲染各自受众,**单一文件、单一 code**。

## 完整性守卫(防退化,镜像 `tool_i18n_complete.rs`)

`crates/growbox-gui/tests/notice_i18n_complete.rs`:遍历所有 code,断言
- `human` 4 ui_lang 全非空;
- `perceive=true` 则 `llm` 2 prompt_lang 全非空;
- `severity` ∈ {info,success,warn,error};`surface` ∈ {toast,health,silent}。
缺任一 → CI 红。**新增提示漏任一受众/语言,编译不过。** 这就是"不散装维护"的硬保证:维护单元 = 一条提示(一处增改),不是一条翻译。

## 内部状态渲染 prompt_lang 化(配套修复)

现 `Memory::perceive` 渲染块头硬编码中文(memory.rs:386-394),且 `kind` 是自由中文串。改:
- 块头按 prompt_lang(zh/en)渲染;
- `kind` 用 **code**(语言中立稳定标识,LLM 跨回合可一致识别事件类型),message = llm 渲染文本。

## health 作为 notice 特例

`surface:"health"` = 常驻指示灯(`异常告知.md`);`surface:"toast"` = 瞬态。`health.set` 由"传硬编码中文 message"改为"传 code + params";前端 `HealthIssue` 携带 `{code, params, severity}`,显示时取 human[ui_lang](终于让健康消息四国本地化),感知时取 llm[prompt_lang]。

## 与旧 i18n 的关系(什么留、什么并入、什么退役)

本系统**不取代**两套语言轴(`i18n` 两轴:prompt_lang 对内 / ui_lang 对外),而是**复用 + 补全**它——`human`(4 ui_lang)与 `llm`(2 prompt_lang)就是这两条轴。

- **留**:`t()` + `locales/*.ts`(UI 字典)继续管静态 chrome(按钮/设置标签/面板标题等纯对外、LLM 无须感知的);`tools.i18n.json`(本系统就是抄它);`system.{zh,en}.md`。
- **并入(退役旧位置)**:现住在 `locales/` 的 ~40 条 **toast 提示文案**(只有对外一半),Phase 2 搬进 `notices.i18n.json` 补上对内 `llm` 半。退役的是"提示类只存对外"这个不完整做法,**不是双轴机制**。
- **补全**:`Memory::perceive` 里散装的硬编码中文(对内),并入 catalog 的 `llm` 渲染并改 prompt_lang 化。
- 结论:碎片更少而非更多。双轴原则此前只对"工具"落全,现在对"提示类"也落全。

## 边界(诚实记录)

- `perceive:false` 仅限真·纯 chrome(复制成功、jump-to-bottom 之类),且需在 catalog 里显式标注理由 —— 默认一切 perceive。
- 机器面向的既有边界(citation wrapper / ui_control ack / PANELS label,见 `i18n` 诚实记录)不属"提示类",不进本系统。
