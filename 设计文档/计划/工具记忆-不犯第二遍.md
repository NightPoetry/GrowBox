## 两条原则(用户原话提炼)

1. **工具具有记忆性**:每个工具(执行器)在每个项目维护一份"小本本"——用过没、何条件下成功/失败、**已知不可行**的事。不再对已知不可行的操作反复试(反例:内置 MCP 明知够不到当前项目内容仍一遍遍试)。
2. **任何问题不能犯第二遍**:失败做法默认不许原样重试;记下"在何**关键因素**下失败"。**唯一可重试 = 影响结果的关键因素变了**(环境变→结果可能变,可试 + 总结新经验)。**反复失败 = 对关键因素的判断错了**(以为变的不是真关键因素)→ 用重复失败反推、校正自己的因果模型。同精神 = 指针网**反 K 一票否决**,从记忆检索泛化到工具/任何行为,加"关键因素"这一可校正维度。

## 设计(复用记忆内核,不建平行子系统)

载体 = **受控 kind `tool_memory` 节点**(同 process/skill 族),项目级软隔离(project_id)。一条 = 一个工具在一种"情况(关键因素)"下的结论。复用 ingest/嵌入/检索/near-dup 取代,**不建独立存储**。

节点格式(`tool_memory_format`,镜像 `skill_format`):
```
# 工具记忆:<tool 名>
情况:<关键因素的自然语言描述>
结论:<infeasible | fails | works>

<detail:为什么 / 当时观察>
```

三件事:

### A. 写小本本 —— `note_tool_memory{tool, situation, verdict, detail}` 执行器(AI 的手)
AI 在判定"这事在此情况下不行/失败/可行"时主动记。脊柱拦截(同 `learn_skill`/`learn_process`)→ `crystallize_tool_memory`:即时嵌入 + **近重复取代旧版**(同工具 + 情况高相似 → 新版取代,反映"关键因素变了 → 覆盖旧结论",自校正)。**不自动记每次失败**(多数失败是瞬态、噪音);记是 AI 的**刻意总结**(呼应"总结经验")。

### B. 查小本本 —— 分发前自动会诊(脊柱钩子,"分发前查小本本")
Agent 循环在 `dispatch` 某工具调用**前**:
- **成本门**:本项目该工具无任何 tool_memory → 跳过(零开销,绝大多数工具如此)。有才嵌入当前"情况(工具+参数)"并取该工具最相似的一条记忆。
- **`infeasible` + 相似度 ≥ `tool_memory_veto_threshold`(默认 0.85)→ 反 K 一票否决**:不执行,把教训当工具结果返回("已知不可行:<detail>。仅当关键因素变了才重试;若确实变了,先 note_tool_memory 记录新情况覆盖旧的,再调用")。这是"不犯第二遍 + 反 K 否决"的硬闸,**仅对 AI 显式标 infeasible 的**(最强信号)+ 高相似 + AI 可经记录新情况解除(supersede→旧版被取代→不再否决)。
- **`fails` + 相似度 ≥ `tool_memory_warn_threshold`(默认 0.80)→ 软提醒**:照常执行,但在结果前置"提醒:你曾记录此工具在类似情况失败:<detail>。不犯第二遍——除非关键因素变了"。不阻断(尊重 build/test 类工具的关键因素是外部状态、可能已被其它工具改变)。
- `works` / 低于阈值 → 正常执行(v1 不注正面提示,保持精简)。

**为何 infeasible 硬否决、fails 仅软提醒**:`infeasible`=AI 断言"这条路根本走不通"(如 MCP 够不到项目)→ 重试纯属浪费,硬挡;`fails`=可能瞬态/外部状态依赖(关键因素=代码/环境,可能已被其它工具改变)→ 硬挡会误伤合理重试(改完代码再 build),故只提醒、由 AI 据关键因素自判。安全:硬闸只命中显式 infeasible + 高相似。

### C. 本回合即时守卫(零 LLM/嵌入)—— "犯第二遍"当场拦
Agent 循环记本回合**失败过的 (tool + 规范化参数) 指纹**;AI 原样重发同指纹 → 结果前置"不犯第二遍"提醒(本回合已相同调用失败:<原因>;除非关键因素变了别原样重试,换做法或说明哪个关键因素变了)。**软提醒非硬阻**(规避 build/test 这类"参数同但外部关键因素已变"的误伤);与持久小本本(A/B)互补,零成本当场强化原则。

## 旋钮(数值全可设,推论9)
- `tool_memory_enabled`(默认 true):总开关。关 = 完全不会诊不记不守卫(行为同今天)。
- `tool_memory_veto_threshold`(默认 0.85):infeasible 硬否决相似度阈。
- `tool_memory_warn_threshold`(默认 0.80):fails 软提醒相似度阈。
设置 → 高级 三项;关/低阈 → 退回旧行为。

## 接线点
- core `Settings`:3 字段 + 默认。
- memory:`node_kind::TOOL_MEMORY` + `tool_memory_format.rs` + `ingest_tool_memory`/`tool_memory_count_for`(成本门)/`crystallize_tool_memory`/`consult_tool_memory`(镜像 skill 那套)。
- gui executors:`note_tool_memory`(脊柱拦截,默认 deferred=低频)。

- 前端:高级 Tab 3 旋钮 + api + i18n 四语。

## v2(2026-06-13 用户点的★重要★方向):MCP 失败分析 = 带时间戳的工具笔记

> 决策原话见 `用户决策/决策日志.md` 2026-06-13 条。把工具记忆延伸到 MCP,并补"时间戳判过时"维。
> 背景:真机新建 VibeGame 项目,filesystem MCP 只授权 Desktop、够不到项目目录而失败 —— 不该只是"失败一次",应沉淀为可见笔记。

复用 A/B 的工具记忆内核,补三点:

1. **失败即分析**:MCP/工具失败 → LLM 分析出**失败原因 + 使用前提**(不是裸错误,而是"为什么失败 + 用它需要什么前提"),结晶成 tool_memory 笔记(verdict 多为 infeasible/fails,detail = 原因 + 前提)。比 v1"AI 刻意总结"再进一步:失败本身触发一次轻量分析(AI 收到失败结果顺手 note,或脊柱在 MCP 失败时提示 AI 总结)。
2. **选择期可见(不止"将调时拦")**:把 MCP/工具候选列给 LLM 选时,每个候选的描述**并入该工具历次笔记 + 每条笔记的时间戳**。LLM 选工具前就看到"这个 MCP 上次在什么前提下失败/可行",据**内容 + 使用前提**判"适不适合当前任务"——会诊从"将调时拦"前移到"选择时就知道"。
3. **时间戳交给 LLM 判过时**:每条笔记显式带 created 时间戳呈现。LLM 据**时间 + 内容**推断笔记是否过时(环境/MCP 版本/授权范围可能变了,旧失败结论未必仍成立)→ 不被旧笔记永久误导。区别于 v1 recency(现仅用于并列相似度 tiebreak,不交给 LLM 推理)。

落点(待实现,复用工具记忆内核):
- `tool_memory_format` 带 created_at;会诊/工具清单渲染时附时间戳。
- MCP/工具描述(definitions / tools_for / MCP 状态展示)旁挂该工具 tool_memory 摘要 = 选择期注入。
- MCP/工具失败结果回灌时,提示/引导 AI `note_tool_memory`(原因 + 前提);或轻量自动草拟待 AI 确认。
- 消费点从 v1"dispatch 前拦"扩到"工具选择/列举时并入描述"。
- 价值:失败一次即沉淀为"下次别踩 / 踩前看前提"的可见笔记,且随时间自然失效——基建不拖累模型,"小本本"做成选择期可翻、会过期的活笔记。

## 不做(留后续)
- 自动记录每次失败(噪音;靠 AI 刻意总结)。
- tool_memory 接指针网正反 K 学习(v1 用直接相似度 + verdict + supersede 已够;指针强化留 v2)。
- 工具记忆的设置 UI 只读浏览(同 skill 清单那样列出);v1 先把机制跑通。
