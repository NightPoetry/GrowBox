# 造物交互 v2 — 写造物进聊天可控 / 端口沟通不进 / 做好才展示 / 关闭硬机制

> 用户原话见 `用户决策/决策日志.md` 2026-06-04「造物交互 v2」条。本稿修正 0-OPUS21「造物全程不进聊天」。
> 背景:真机测五子棋,用户落子触发的 ArtifactSink 回合(静默 + max_turns=1000)反复 render 微调棋盘,
> 聊天看不到、停不下,用户恐慌。根因 = 写造物被静默 + 无终止 + 无收敛。

## 核心二分(用户定,灵魂)

> "造物的过程不就是在写代码吗?自然要进聊天。只有那些通过端口与造物双向沟通...太频繁了"

| 类别 | 例子 | 进聊天? | 可终止? |
|---|---|---|---|
| **写造物**(写 HTML/代码) | `render_artifact`、`selftest_artifact` | ✅ 进(写代码文件) | ✅ 终止按钮可停 |
| **端口双向沟通**(高频交互) | 落子回传、实时输入(gx-realtime) | ❌ 不进(太频繁刷屏) | — |

判据:**改造物结构/内容 = 写 = 进聊天**;**只读交互值/控制已有造物 = 端口沟通 = 不进**。
端口沟通"不可能出问题":造物关闭→端口不通→AI 自动知道(见 §4 硬机制)。

## 0. ★造物灵魂:自包含即时视觉 + LLM 是使用者(2026-06-04 用户点破,最重要)★

真因诊断:五子棋"慢/卡/没法点击"的根 = **无即时视觉反馈**(非 max 慢本身)。落子后画面没动→以为卡死。
**根治 = 造物自己即时反馈,LLM 只是使用者:**
- **造物自包含即时视觉**:用户点交叉点 → 造物自身 JS **立刻画黑子**(本地 0 延迟,不经 LLM/端口)。视觉是造物的事。
- **LLM 是"使用者",不控制视觉渲染**:LLM 像玩家用自己造的棋盘 —— 通过端口发"落白子(x,y)"**指令**,造物 JS 执行;LLM **不重画整棋盘、不控制视觉**。("延伸≠另一个程序"的真义:LLM 用造物如人用工具。)
- **造物显示 LLM 状态**(让用户确信"在想不是卡"):造物含 ① **思考计时区**("对手思考中 · Ns")② **流式思考小窗口**(LLM reasoning 流式推进造物)。
- **对 render_artifact 模型的修正**:不再"每次重画整个棋盘"。造物**维护自身状态**(棋盘 DOM),LLM 发**增量指令**(落子)+ **思考流**,造物自己更新视觉。
- **机制需求**:端口 **LLM→造物** 要能发结构化指令(落子)+ 流式 reasoning(推思考窗),不只现有 push_artifact_notice(单条文字覆盖层)。设 `artifact_command`(LLM→造物 JS,造物注册 handler 执行)+ reasoning 流式转发到造物思考窗。
- **提示词**:LLM 造棋盘 HTML 时必须做:即时交互(落子本地显示)+ 思考计时区 + 思考流小窗 + 注册 LLM 指令 handler(收"落白子"画子)。

## 1. 写造物进聊天 + 可终止

- artifact 回合的 `render_artifact`/`selftest_artifact` 的 ToolStart/ToolEnd + AI 的 reasoning/content
  **emit 进聊天**(改 ArtifactSink:不再全静默,写造物工具进度进聊天)。
- 端口输入 seed(落子值)**不显示**(不创建用户气泡)——但 AI 的写造物响应进聊天。
- 前端:离散造物交互(非 realtime)→ 走"进聊天的接收回合"(创建 assistant 气泡 + 监听 chat-chunk + present)。

## 2. ★可终止(新能力,GrowBox 当前没有)★

现状:chat 回合**无中断机制**(chat.ts 注释"前端不应 abort";只 supervisor 后台有 CancellationToken)。
需新建:
- 后端:`agent_loop` 每轮检查 `CancellationToken`(回合级,存 AppState 或独立 managed state)。命中即优雅收口(StopReason::Cancelled)。
- 命令:`cancel_chat()`(set token)。
- 前端:聊天「终止」按钮(sending 时显示)→ invoke cancel_chat → 后端下一轮检查点停。
- 同时治本失控:写造物进聊天后,用户看到反复 render 能主动终止(替代当前 max_turns.min(4) 硬限——可保留作兜底)。

## 3. 做好才展示(present)— ★已实现★

- `render_artifact` 渲染到**常驻 iframe**(草稿,display:none,可 selftest)但不弹给用户。
- 回合结束(chat.ts doStreamRound finally / artifactEvent then)→ `presentArtifactIfReady()` 展示最终态。
- `push_artifact_notice`(覆盖层=AI 主动提示)立即展示。
- as-built:ArtifactCanvas `artifactRendered`(渲染态)/`artifactCanvasOpen`(展示态)分离。

## 4. ★造物关闭硬机制(非 LLM,硬性)★

> "随着造物关闭端口不通 AI 就知道了,并且在聊天中告知用户窗口关闭了,并且自己关闭监听(不是 LLM 操作是硬性的)"

- 造物窗口关闭(用户点 × / closeArtifactCanvas)→ 前端检测 → 通知后端"造物已关闭"。
- 后端**硬性**(非 LLM 判断):标记造物端口失效 + 在聊天 emit 一条"造物窗口已关闭"(notice)+ **自动停止相关监听/造物回合**(若有在跑的造物响应回合,取消它)。
- AI 感知"端口不通"= 造物没了,不再尝试操作它。
- 对应"端口沟通不可能出问题":出问题(造物关了)是硬机制兜底,不靠 LLM 自己发现。

## 6. 造物文件夹(状态/记忆隔离,2026-06-04 用户洞察)

> 逐字原话+整理见 `用户决策/决策日志.md` 2026-06-04"造物有自己的文件夹"条。真机 diag 印证:AI 本能 `file_edit /root/.growbox/artifacts/main.html` 想把造物存文件被授权拦。

- **造物是实实在在的造物 → 有自己的文件夹**:每个造物一个,如 `<project>/.growbox/artifacts/<canvas_id>/`,**可写**(沙箱授权,免每次弹授权)。
- **封装造物的流记忆/持久状态**:棋盘 `board.json`、聊天机器人对话历史等。AI 用文件工具读写 = **比"前端回传完整状态"更持久可靠的"AI 的眼睛"**(关游戏/重启后还在)。
- **LLM 定生命周期**:持续一整盘(五子棋一局清)/ 持续整个造物存续(聊天机器人持久)。
- **隔离主记忆**:造物文件夹读写不进主时间线/RAG,不污染。**双赢**:复用 Agent 文件能力 + 不干扰主记忆。
- **与工作流的关系**:造物工作流(作用域=造物)也存这个文件夹(见 `设计/07-工作流机制.md`)。
- 落地:① 路径约定(canvas_id→文件夹)② 沙箱把造物文件夹纳入可写 ③ 提示词:AI 用造物文件夹存状态/记忆+自定生命周期 ④ 隔离读写不进主记忆。

## 7. selftest 模拟 UI 操作(造物自检硬验证,任务 #5)

> 现状:selftest_artifact 只枚举 `data-gx-callback`(用户点击回调),**没验证 AI 自己的落子通道 `gxOnCommand`**——真机暴露:AI 轮到落子却发 notice(可能因棋盘没实现 gxOnCommand,落子无门)。

- **自检应模拟 AI 操作**:selftest 不止枚举回调,还要**发一条测试指令经 gxOnCommand**,验证造物能正确执行 AI 的操作(如落子)→ AI finish 前确信落子通道通。
- **硬验证(非仅提示词)**:AI 没实现 gxOnCommand / 通道不通 → selftest 报告失败 → AI 必须补好再 finish(治"造物版虚假成功"在 AI 操作面的延伸)。
- 与工作流配合:对弈工作流的"AI落子"节点,其工具子集物理保证用 artifact_command(不会误用 notice);selftest 保证 gxOnCommand 真能执行。两者一上一下夹逼可靠。
