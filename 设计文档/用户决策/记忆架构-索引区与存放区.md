# 记忆架构:索引区 与 存放区(用户原话 · 2026-06-15)

> 单列文件(决策日志已长)。本文是用户对 GrowBox 临时记忆/置换架构的一次**清晰架构关系**澄清——逐字原话 + 提炼。
> 真理来源:与 `记忆置换系统-总纲.md`、`设计/02-记忆检索.md` 互为印证。

## ★用户原话(逐字,勿改写)★

**(一)RAG 与缓存是两码事,缓存=临时记忆:**
> "RAG 仅仅是检索方式它与缓存是两码事,只要检索到的不论是 RAG 还是 L2 都会进入缓存体系啊。因为缓存体系才是临时记忆体系。"
> "你没进缓存怎么记忆的?"

**(二)索引区 + 存放区,临时记忆只有一个存放区:**
> "还记得我们的 L2 缓存是怎么设计的吗?分为索引区和对应的存放区你看看设计。所以自始至终我们的临时记忆只有一个存放区。而且索引区是找到的记忆的索引指针,不仅仅是 LLM 还有 RAG,LLM(L2)和 RAG 只是索引的手段不同而已。"

**(三)存放区规则(两态铁律):**
> "还有存放区的规则知道吧,虽然队列一直在移动但是只要没有被淘汰出队列,那么存放区中对应的内容是不会移动的,从而保证缓存命中率。"

## 清晰的架构关系(原话提炼)

```
                  ┌─────────────── 索引区(找"该调入谁")───────────────┐
   一条 query ──▶ │  索引手段(可多种,只是手段不同):                  │
                  │    · RAG(向量近似,第一层)                         │
                  │    · L2/精确(LLM 读原文 + 指针图导航,第二层)      │
                  │  产出 = 命中记忆的【索引指针】                       │
                  │  (指针网/页表 + 磁盘指针图之上的 LFU 缓存=加速)    │
                  └───────────────────────┬───────────────────────────┘
                                          │ page_in(把指针指向的内容调入)
                                          ▼
                  ┌─────────── 存放区(临时记忆,自始至终只有一个)──────────┐
                  │  = 工作记忆区 = LLM 上下文里那块"在场"的记忆数据         │
                  │  ★两态铁律★:append-only 排列;信息只有"在/移除"两态;     │
                  │   只要没被淘汰出去,内容【原地不动、绝不随热度重排】      │
                  │   → 上下文字节稳定 → 命中 LLM 的 prompt KV 缓存(省钱)   │
                  └─────────────────────────────────────────────────────────┘
```

**四条要点:**
1. **临时记忆 = 缓存体系 = 存放区,有且只有一个。** 检索到的内容(不论 RAG 还是 L2 找到的)**都进这一个存放区** —— 这就是 AI "怎么记住"的:没进存放区就谈不上记忆。
2. **RAG 和 L2 只是"索引手段",不是两个缓存/两个存储。** 它们的差别只在"怎么找到记忆的索引指针"(RAG=向量近似;L2=LLM 读原文+指针图导航),找到后殊途同归、都调进同一个存放区。
3. **索引区 = 命中记忆的索引指针**(指针网/页表)。其上的 LFU"缓存"(曾叫邻域缓存/NeighborCache)是**索引区的加速器**,不是存储——它空/满都不代表记忆有没有进存放区。
4. **存放区两态铁律:队列(索引/热度)一直动,但存放区内容不随之移动**;只有"被淘汰出队列"才会让对应内容从存放区移除。幸存者原地不动 → 上下文前缀字节稳定 → 保证(LLM prompt)缓存命中率。

## 对实现 / 面板的含义(澄清以往误判)
- **面板"临时记忆/缓存"指标 = 存放区占用 = 「队列占用」**(工作区填充度)。这是唯一的"记忆缓存"指标。
- **「置换率」= 淘汰出存放区的频率**(唯一真正改变存放区的事件);队列/热度的来回移动不碰存放区、不计入。
- 曾经把 NeighborCache 当"记忆缓存"摆在面板上(显示 0/256)是**框架错误**——它是索引区的 LFU 加速器,非存储;0 不代表"记忆没进缓存"。
- 代码落点:存放区 = `growbox-memory/src/context.rs` `ContextWindow`(两态铁律就在其注释);索引区 = 指针网(`pointer.rs`/`store.rs` EDGES)+ RAG 向量索引(`index.rs`)+ LFU 缓存(`cache.rs`)。

## 指针与 L2:一次 / 二次指针的生命周期(用户原话续 · 2026-06-15)

### 用户原话(逐字)
> "还有指针和 L2 的关系你知道吧,L2 里面全是指针,就算索引到了原文也会建立一个指针。这也是指针创建的一个方式,被索引然后就出现了指针不是吗?第一次被索引的位置就会产生指向被索引内容的指针,然后再产生二次指针存放到 L2 中,然后这个二次指针随着被换出 L2 将会在换出的位置留下自己的二次指针,并随着再次被索引到,会将二次指针移动回 L2(当然为了防止崩溃增加鲁棒性二次指针在进入 L2 的时候的位置会有的,进入 L2 再出去的时候只是将二次指针更新了位置,而二次指针随着碎片整理会变成一次指针)。这些结合之前我写的技术文档理解一下。"

### 提炼(结合 `设计/02` 五件套的"指针 + 二级索引/碎片")
- **L2 里全是指针;指针的一种创建方式 = "被索引"** —— 任何一次索引(RAG 向量命中 或 L2 的 LLM 读原文判定相关)都会沉淀出指针(原则2:扫描副产品全沉淀,越用越快)。被索引 → 就出现指针。
- **一次指针(= 一级索引)**:第一次被索引时,在**索引发生的位置**产生一个指向被索引内容的指针(走过的路 / 捷径,下次直接跳)。
- **二次指针(= 二级索引 / 锚)的生命周期**:
  1. 索引时,除一次指针外,再产生一个**二次指针存进 L2**。
  2. 二次指针**被换出 L2** → 在**换出的位置**留下自己的二次指针(= `02` "缓存条目漂远 → 在当前位置建二级锚点指回原始位置")。
  3. **再次被索引到** → 把二次指针**移回 L2**。
  4. **鲁棒性(防崩溃)**:二次指针进 L2 时其位置就有记录;"进 L2 再出去"只是**更新**二次指针的位置(不新建、不丢)。
  5. **碎片整理(做梦 / Dream)** → 二次指针**升级成一次指针**(`02` "再待满 2K → 锚点改指一级索引、删除旧二级;永远只有(一级,二级)两层,中间空白即碎片")。
- **与 `02` 对应**:一次指针 ↔ 一级索引 / 指针;二次指针 ↔ 二级索引 / 锚点;换出留标记 · 再索引移回 · 碎片整理升级 ↔ "远处拉近 + 欠债记账 + 做梦补入"。L2 全是指针 ↔ 精确层在指针图上导航。

## 指针的纯净性:LLM 指针 vs RAG 指针 / 假指针(用户原话续 · 2026-06-15)

### 用户原话(逐字)
> "因为一次指针必然是扫描过去的,所以一定是前面都被直接或者间接扫过的所以没有碎片。到那时 RAG 的出现导致一次指针不纯净了。所以一次指针分为 LLM 指针和 RAG 指针,LLM 指针有碎片回收机制,RAG 看似是单点指针,但是 RAG 本质上也是用向量沿着从前往后扫的(除非 RAG 系统有自己的扫描方法),如果是从前往后扫的就能给 RAG 单独进行染色,指针和碎片系统分为 RAG 和 LLM 部分,从而加速 RAG 的检索,如果 RAG 系统有自己的检索方式,那么就不用了,只需要将 RAG 的指针作为假指针方便统一在队列和缓存中而已,假 RAG 指针离开了队列不会落入序列中。"

### 提炼
- **一次指针"无碎片"的本质 = 它必然是"扫描过去的"产物。** LLM 的 L2 顺序扫(从近往远读原文),所以到它为止的路径都被直接/间接扫过 → 底下天然没碎片。(比"底下已扫"更本质:它本就是扫出来的。)
- **RAG 打破纯净**:RAG 靠向量相似度**直接跳**到目标、没扫路径 → 这样得到的一次指针"不纯净"(底下没扫、可能有 gap)。
- **故一次指针分两类**:
  - **LLM 指针**:顺序扫的产物,**有碎片回收机制**(做梦扫 gap → 还债)。
  - **RAG 指针**:向量跳的产物,怎么处理取决于 RAG 是否顺序扫 ↓
- **RAG 处理两条路(取决于 RAG 的检索方式)**:
  - **(a) RAG 是"从前往后扫"(朴素/暴力向量)**:本质也是顺序扫(只是用向量算分而非 LLM 读)→ **可给 RAG 单独染色**(记它扫过哪些区)→ **指针/碎片系统分成 RAG 部分 + LLM 部分两套** → RAG 跳过已染色区 → **加速 RAG**。
  - **(b) RAG 有自己的检索方式(ANN/HNSW,不顺序扫)**:它不扫、没 gap 概念 → **不需要给它染色/碎片**;只把 **RAG 指针当"假指针"**,纯为**统一进队列和缓存**(让 RAG 命中也能进存放区、参与置换);**假 RAG 指针离开队列时不落入序列**(不像真指针那样在时间线位置留二次锚——它没有有意义的序列位置)。
- **与 GrowBox 实现**:`index.rs` = 可切换 ANN/暴力的向量索引,0-OPUS4"第一层 ANN 已落地" → 若用 ANN 则走 **(b)**:RAG 指针应作"假指针",统一进队列/缓存参与置换,但**不留序列二次锚、不做碎片回收**(ANN 无扫描 gap)。〔2026-06-15 核实(Opus):**设计基本落地、无功能偏差**。RAG=ANN(`index.rs` HnswIndex 内存 / ArroyIndex 磁盘)。LLM/L2 侧完整:`secondary.rs` 二级锚 K/2K 规则 + `retrieval.rs:503` 跳过中段记碎片 + 做梦回收(LLM 指针有碎片回收)。RAG 侧:命中只 `touch+persist+page_in` 进存放区、**不建真指针** → 天然无不纯指针 / 无幻象碎片 / 换出不留序列锚,行为完全符合 (b)。**唯一差异 = 概念/命名**:无显式"假指针"对象、无 LLM-vs-RAG 标签,但"RAG 统一进存放区吃置换"的目的已由 page_in 达成。可选小增强:把 RAG 命中物化成标了 fake 的假指针进指针队列(纯为统一视图/面板可数),不影响当前正确性。〕
