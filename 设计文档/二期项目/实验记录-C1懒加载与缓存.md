## 实验一:tools 数组对 KV 缓存的影响(`live_lazy_cache.rs`)

**方法**:同一固定大 system(8558 prompt_tokens)+ 同一 user,**只变 tools 数组**,非流式直读
`usage.prompt_cache_hit_tokens`。tools 用**真实 registry 载荷**(全量 25 / 节点子集 6 / 懒开核心 12)。

**结果**:

| 请求 | tools | prompt_tokens | cache_hit | 命中率 |
|------|-------|---------------|-----------|--------|
| 预热 | 全量 25 | 8558 | 0 | 0%(首次建缓存)|
| 基线·重复同一 tools | 全量 25 | 8558 | **8448** | **99%** |
| ★懒关·换 tools | 节点子集 6 | 3981 | **0** | **0%** |
| ★懒开·tools 恒定 | 核心 12 | 4386 | **4352** | **99%** |

**结论(铁证)**:
- **tools 数组在 deepseek 缓存前缀里,且排在最前**:system 一字未改,只换 tools → 命中从 8448 **掉到 0**
  (整个 prompt 全部 cache_miss 重算)。说明 tools 序列化在 system 之前,**任何 tools 变更 = 整条 prompt 缓存全毁**。
- **C1 的病真实存在**:工作流 v2 每进出节点换工具子集(懒关行为)= 每次都 0% 命中,最大化缓存破坏。
- **C1 的药有效**:懒开让 tools 恒定(核心常驻)→ 99% 命中保住,与"重复同一 tools"基线持平。

## 实验一·补:多轮工作流端到端省多少(`live_lazy_cache.rs::lazy_tools_cache_multiturn_workflow`)

**方法**:模拟 6 轮工作流,对话 append-only 增长;懒关在 3 处节点切换换 tools(full→node→node→full→node→full),
懒开 tools 恒定(core)。量累积 cache_hit/miss(= 每轮重算多少 token)+ 总耗时。两条会话 system 加不同 run_tag 防缓存互染。

**结果(6 轮 / 3 处切换)**:

| | 累积 cache_hit | 累积 cache_miss(重算) | 命中率 | 耗时 |
|---|---|---|---|---|
| 懒关 | 25344 | **13188** | 66% | 10866ms |
| 懒开 | 22272 | **4959** | 82% | 8629ms |

**结论**:懒开比懒关**少重算 8229 token**(miss 降 62%)、命中率 +16pt、耗时少 ~20%。
懒关每次节点切换把"已缓存的**增长前缀**"整段毁掉重算 → **任务越长、节点切换越多,差距越大**。
(懒关命中率没到 0% 是因 6 轮里只 3 轮换 tools,非切换轮仍命中;但 miss=真账,懒关多烧一倍。)

## 实验二:懒加载下 agent 的真实行为(`live_lazy_agent.rs`)

**方法**:懒加载开,`code_search` 是 deferred(只露名)。给真模型任务"用 code_search 找 greet 的调用处"。
观测它会不会自己 tool_search 加载 deferred 工具并完成,还是因 schema 不在场而卡住/变笨。

**结果**(3 轮,Completed):
1. 模型见露名 → `tool_search{query:"code_search"}` → 拿回完整 schema;
2. 立刻 `code_search{pattern:"greet", glob:"src/**/*"}` → 命中 src/lib.rs 2 处;
3. 正确报告 file:line + `finish`。

**结论**:真模型**自然地** discover→load→use deferred 工具,零卡顿、零困惑,一次成功。
懒加载不让 agent 变笨(成本侧通过)。

## 总结论

C1 收益(缓存:单次换 tools 0%→99%;6 轮工作流少重算 8229 token、miss 降 62%、快 ~20%)
+ 成本(agent 行为无退化)双侧真机证实。**默认仍保持关**(用户可在设置拨开关),但数据强烈支持:
**工作流密集 / 工具数膨胀(D 期 MCP)场景应开**。待用户拍板默认值(可按"是否常用工作流"分场景)。
