# 工具体系扩展 · 总览与路线图

> 把 `设计原理/00-工具体系扩展.md` 的原则落成四个可实施模块,定优先级与里程碑。
> 每模块详细设计另起文档(`01-LSP集成.md` 等);本篇是总览。

## 范围

只做"让 GrowBox 看得见代码 + 收编开放生态工具"四件事:**LSP 代码智能 / 代码搜索 / Web 查询 / MCP 客户端+懒加载**。
不做:不在编码闭环上的生态工具(日历/CRM/设计稿等,留用户按需连);不改内核认知架构(记忆/飞轮/安全公理不动)。

## 项目级智能凌驾于四模块之上(见 `01-流程与检索架构.md`)

下面四模块是**语言级**代码智能的底层事实源(找代码里的事实)。其上还有一层**项目级**智能——
"本项目改一个东西的全部涟漪面"——它语言无关、由学习型流程承载、当"第 0 层"、用下面四模块当手。
原理见 `设计原理/01-流程即一等公民.md`,架构见本层 `01-流程与检索架构.md`。

**这重排了二期重心**:项目级流程(自举 + 报告-纠正回路 + 飞轮结晶,且检索复用一期记忆 RAG)很可能比
"追着给每种语言接通用 LSP"更核心、更 GrowBox——因为它解决的是真正难的那部分(项目涟漪面),还语言无关。
故落地上**建议档流程(M1,只依赖一期记忆系统)可与 LSP 并行甚至更早起步**。

## 四模块一览

| # | 模块 | 类型 | 对编码价值 | 接原理 |
|---|------|------|-----------|--------|
| 1 | **LSP 代码智能** | 自建执行器 | ★★★ | 原则1 / 推论1·4 |
| 2 | **代码搜索(ripgrep)** | 自建执行器 | ★★ | 原则1 / 推论1 |
| 3 | **Web 查询** | 自建执行器(或走 MCP) | ★★ | 原则1 / 推论4 |
| 4 | **MCP 客户端 + 懒加载** | 桥 + 注册机制 | ★★★ | 原则2·3 / 推论2·3 |

### 1. LSP 代码智能(优先级最高)

- **范围**:接 rust-analyzer + tsserver(GrowBox = Rust + SolidJS),暴露 LSP 操作给 AI;诊断接感知层主动推送。
- **方案**:起一个 LSP 客户端子进程管理器(按文件类型起对应 server),一个 `lsp` 执行器封装操作;
  诊断订阅 → 经内核「内部状态感知层」推入上下文(像感知失败一样)。
- **接口草案**:`lsp{ op: goToDefinition|findReferences|hover|documentSymbol|workspaceSymbol|goToImplementation|incomingCalls|outgoingCalls, filePath, line, character, query? }`。
- **接原理**:推论1(自建核心代码智能、诊断接感知层)+ 推论4(补"看"与"验"两环)。
- **里程碑**:M1 起 rust-analyzer + hover/definition/references 三件 → M2 诊断推送接感知层 → M3 调用层级 + tsserver。
- **风险**:LSP server 生命周期/性能(大工程首次索引慢);位置是 1-based 行列,要和 file_read 的行号体系对齐。

### 2. 代码搜索(ripgrep 后端)

- **范围**:一个 `code_search` 执行器,正则 + glob/类型过滤 + 输出模式(行/文件/计数),替代 `shell grep`。
- **方案**:内嵌或调用 ripgrep;权限受可写目录约束、结果结构化回传。提示词层引导"搜代码用它,别用 shell grep"。
- **接原理**:推论1(代码搜索是"看得见代码"的物理基础之一)。
- **里程碑**:M1 基础 content/files/count 三模式 → M2 multiline + 类型过滤。
- **风险**:与 `file_list` 职责划清(list=列目录,search=按内容/模式找);大仓库结果上限(接内核工具输出上限旋钮)。

### 3. Web 查询

- **范围**:`web_search` + `web_fetch` 两执行器,用于解陌生报错、查 crate/库 API、查文档。
- **方案**:走一个搜索 API + 抓取转 markdown;或先用 MCP 生态里的 web server 顶上(见模块4),不急自建。
- **接原理**:推论4(补"看"——解报错也是看清问题的一环)。
- **里程碑**:可延后到 MCP 客户端就绪后,用现成 web MCP 顶替,省自建。
- **风险**:联网与隐私(发出去的内容即公开,接内核安全门 provenance 判定);需稳定网络/API key。

### 4. MCP 客户端 + 懒加载(战略级,大工程)

- **范围**:GrowBox 做成 MCP 客户端,连任意 MCP server,把其工具动态注册成执行器;配懒加载管理工具数。
- **方案**:① MCP 传输层(stdio/SSE/HTTP)+ 握手列工具;② 每个 MCP 工具 → 适配成 `Executor`(definition 来自 MCP schema)动态注册进 registry;③ 懒加载:核心工具常驻,MCP/工作流工具只露名,加一个 `tool_search` 执行器按需把 schema **追加**进上下文(不改前缀);④ 连接配置(类似 `.mcp.json`)落项目/全局作用域(复用工作流 P3 持久化的分桶思路)。
- **接口草案**:`tool_search{ query }`(select:名字 / 关键词 / 必含词)→ 返回匹配工具的完整 schema 块,之后即可直接调。
- **数据流**:`AI 想发 Slack → tool_search("slack send") → 拉回 slack_send schema 追加上下文 → AI 调 slack_send → MCP 客户端转发给 server → 结果经唯一脊柱回灌(安全门/感知/记忆一视同仁)`。
- **接原理**:推论2(MCP 工具=动态注册执行器)+ 推论3(懒加载保前缀 byte-stable,顺带消除工作流 v2 按节点换工具对缓存的破坏)。
- **里程碑**:M1 懒加载机制(tool_search + 名字清单注入 + append schema)先在**现有工具/工作流**上落地(不依赖 MCP)→ M2 MCP 传输 + 工具→执行器适配 + 单 server 打通 → M3 多 server + 连接配置持久化 + 安全门(MCP 工具结果当不可信输入,见内核安全)。
- **风险**:MCP 工具结果是外部不可信输入(prompt injection 面),必须过安全门、不可直接用于风险动作参数;工具数膨胀后 `tool_search` 召回质量;跨机/headless 下交互式鉴权的 server 不可用。

## 总里程碑(建议顺序)

1. **LSP(模块1)先行** —— 单点回报最高,IDE 本职,接感知层。
2. **代码搜索(模块2)** —— 便宜,补"看"。
3. **懒加载机制(模块4 的 M1)** —— 先在现有工具/工作流上落地,顺带修工作流 v2 的缓存破坏。
4. **MCP 客户端(模块4 的 M2/M3)** —— 收编生态,Web 查询用现成 web MCP 顶替。

> 每完成一个模块的设计细化,在本表对应行补"详见 `NN-xxx.md`"。落地仍按内核 phase-pacing 冷却节奏分 phase。
