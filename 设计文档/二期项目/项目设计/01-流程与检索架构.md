# 流程与检索架构(项目级智能的落地)

> 把 `设计原理/01-流程即一等公民.md` 落成可实施架构。核心结论:**不另起炉灶**——执行、检索两个引擎
> 一期都有,二期只补一座"桥"。

## 范围

只定流程的**存 / 检索 / 物化 / 执行**四件事怎么接到现成件上。
不做:不重做执行引擎(用工作流 v2)、不重做检索引擎(用记忆 RAG)、不改一期任何机制。

## 方案

**三机制汇流**(全是一期现成/在建件):

| 环节 | 用什么 | 来源 |
|------|--------|------|
| **执行容器**(流程怎么跑) | 工作流 v2(类型化 I/O、节点图、P3 持久) | 一期 `设计/07-工作流机制.md` |
| **检索**(现在该用哪个流程) | 记忆分层检索 RAG→精确层 + 学习型指针 | 一期 `设计/02-记忆检索.md` |
| **物化**(把选中的流程拉成可调工具) | 懒加载(只给召回的少数注入 schema) | 二期 `00-总览与路线图.md` 模块4 |

**两档流程**(别把什么都塞进刚性节点图):
- **建议型配方(多数)** = 当**记忆**存。召回后**直接注入上下文给 AI 读、照做**(不物化,像召回一条结论)。轻、灵活、过时好改。
- **可执行流程(少数,复杂且稳定)** = 当**工作流**存。召回后**物化**成可调工具去跑。

**桥(唯一新增)**:流程作为 `process` kind 入记忆 → 语义召回 → 建议档直接注入 / 可执行档物化。

## 物化(术语澄清)

**物化 = 把"只有名字/索引的流程"加载成"此刻可调用的实体工具"的那一下**——像 `import` 一个声明过却没引入的函数。
只有**可执行流程**需要物化;**建议型配方不需要**(召回即读)。流程生命周期:
`存(索引项)→ 检索(RAG 判相关、浮现)→ 物化(注入定义、可调)→ 调用`。

## 接口草案

- **受控 kind 表 + `process` kind**:节点 role 受控表加一个取值(非破坏,旧值天然合法)。
- **检索**:复用 `mind_search`/RAG,kind 过滤到 `process`,返回 top-k `{name, desc, executable?, refs}`。
- **物化**:可执行项经 `tool_search`(模块4)把工作流 schema **append** 进上下文(不改前缀,保 KV 缓存 byte-stable)。
- **结晶**:飞轮把"复发的报告-纠正序列"写成 `process` kind 记忆;够稳够复杂时经 `define_workflow` 升格为工作流。

## 数据流

```
飞轮结晶(process 记忆 / 工作流)
        │
        ▼
任务开始 RAG  或  文件模式触发(端口触发式,如要改 Settings.rs)
        │
        ▼  记忆分层检索(kind=process)→ top-k
  ┌─────┴─────┐
建议档              可执行档
直接注入上下文      tool_search 物化成工具
(AI 读了照做)     (AI 调用去跑)
        │                 │
        └────── 执行 ──────┘
                │
                ▼
        如实报告"改了哪几处"
                │
                ▼
        用户扫一眼纠正  →  飞轮更新该流程(正/反 K + 内容)  ──┐
                                                              └→(回到结晶,自愈)
```

## 接原理

兑现 `设计原理/01-流程即一等公民`:
- 推论1(项目级智能)→ 两档流程承载项目涟漪面知识。
- 推论3(流程即记忆)→ 检索复用 RAG,不另起引擎,解"太多没法检索"。
- 推论2(自举 + 报告-纠正)→ 数据流末段的报告→纠正→飞轮更新闭环,建即修。
- 推论4(纪律)→ 结晶门槛 / 合并淘汰落在飞轮的 consolidation/prune。

## 里程碑与风险

**两个真正要拍板的新决策**(其余都是接线):
1. **毕业门槛**:建议配方何时升格为可执行工作流?判据 = 复发次数 + 步骤稳定度 + 是否值得自动跑;否则一直当可检索记忆(更便宜)。
2. **检索触发点**:任务开始就 RAG 一把,还是"要碰某类文件时"按文件模式触发(端口触发式)?八成两者都要。

**风险**:
- **过时**:流程随架构演化失效 → 靠正/反 K + 用前先验(同记忆陈旧校验);报告-纠正回路兜底修。
- **过度结晶**:堆出废流程 → 靠 `设计原理/01` 推论4 的"复发才结晶 + 持续合并淘汰"。
- **召回不准**:触发了不相关流程 → 靠精确层学习型指针(误触加反 K)自我矫正。

**落地顺序**:
- M1 **建议档先通**:`process` kind 索引 + RAG 召回注入(只依赖一期记忆系统,**不依赖懒加载**,可先落地)。
- M2 **可执行档**:接懒加载物化(依赖模块4)+ `define_workflow` 升格路径。
- M3 **触发与毕业策略**调优 + 飞轮结晶/合并的旋钮化(同一期"一切数值可设")。
