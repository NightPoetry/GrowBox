# 二期 · 项目设计 · 写作规范

本文件夹是二期的**工程落地层**,回答"做什么、怎么落、怎么算对"。对应内核 `系统架构/`,但二期多为**尚未建造**
的前瞻方案,故格式从"描述已建系统"调整为"设计待建项目"。

> 与原理层(`设计原理/`,回答"为什么")分工不同。
> **铁律**:每个项目设计决策必须能回指 `设计原理/` 里的某条原则。指不回去的,要么漏了原则,要么是过度设计。

## 六段格式(为"待建"调整)

1. **范围** — 一句话划死边界:**只**做什么、**不**做什么(防膨胀)。
2. **方案** — 核心做法(模块怎么切、关键机制),够别人据此动手即可。
3. **接口草案** — 对外暴露的关键类型/工具签名(可随实现微调,但先定形)。
4. **数据流** — 一个真实场景从进到出怎么穿过(ASCII/箭头图)。
5. **接原理** — 兑现 `设计原理/` 哪条原则的哪条推论(每个决策可回指,否则即无源设计)。
6. **里程碑与风险** — 分步落地顺序 + 已知风险/待定项(替代"已知坑":尚未建造,先列雷区)。

## 模板

````
# <能力/模块>

## 范围
只做 <X>;不做 <Y>。

## 方案
<核心做法>

## 接口草案
<关键类型/工具签名>

## 数据流
<真实场景的箭头图>

## 接原理
本设计兑现 `设计原理/<文档>` 的 <原则X→推论Y>。

## 里程碑与风险
- M1 <...> / M2 <...>
- 风险:<...>
````
