# 工具体系扩展(代码智能 + 工具生态收编)

> 为什么 GrowBox 要在内核之上扩一层工具能力,以及凭哪条公理扩。核心:让它从"会跑命令的 agent"
> 升级为"看得见代码的 IDE",并能收编开放生态的成百上千工具——而不破坏内核的唯一脊柱与缓存稳定性。

## 原则

1. **IDE 必须看得见代码,而不是靠文本搜索猜。** 一个编程环境的核心能力,是像编辑器一样理解代码结构
   (类型、定义、引用、诊断),而非只能读写文件 + 跑命令后看输出。看不见代码的 agent 只能盲改 + 全量构建验证;
   看得见的 agent 改之前就知道类型对不对、谁会被影响。
2. **一切能力皆执行器,Agent 循环是唯一脊柱(← 内核公理)。** 任何新增能力——本地代码智能、外部生态工具——
   都收敛为执行器,经唯一注册表与分发路径接入。扩展不是开特例,是往同一根脊柱上挂更多执行器。
3. **能力来自开放生态时,工具数无界 → schema 必须懒加载。** 当工具不再是内置几十个、而是来自开放生态的成百上千个,
   不可能把全部 schema 注入每次请求。名字常驻、schema 按需拉取,且追加而非改写,保请求前缀 byte-stable
   (← 内核缓存铁证:byte-stable prefix 才命中 KV 缓存)。

## 推论

### 推论 1:自建核心代码智能,不外包(← 原则 1)

LSP(语言服务器)与代码搜索是 IDE 的**本职**能力,要第一方、可靠、低延迟,不依赖外部生态服务器。
LSP 提供:定义跳转、引用查找、悬停取类型/文档、文档/工作区符号、实现跳转、调用层级(谁调我/我调谁)。
代码搜索提供:正则 + 文件类型/glob 过滤 + 三种输出模式的快速检索。这两者是"看得见代码"的物理基础。

更进一步:LSP 的**诊断**(编译错误/警告)应接入内核「内部状态感知层」**主动推送**——AI 不必每次跑全量构建,
改完即被告知哪一行不过。这把"自我感知一切失败"从运行期失败延伸到**编辑期失败**。

**案例**:AI 要改一个 Rust 函数签名。改之前先 `findReferences` 看 12 处调用点、`hover` 确认相邻类型,
判断改动影响面;改完 LSP 诊断立刻推回"第 47 行 trait bound 不满足",当回合就修,无需等 `cargo build` 几十秒。
对一个不懂 Rust 的用户,这等于 AI 自己把借用/类型错误在编辑期就看清了。

### 推论 2:外部生态工具 = 动态注册的执行器(← 原则 2)

收编开放生态(MCP,Model Context Protocol,开放标准,社区有成千上万 server)不靠逐个开发,而是做一个
**MCP 客户端桥**:连上某 MCP server,把它暴露的每个工具**适配成一个 GrowBox 执行器**,动态注册进 registry,
之后与内置工具走完全相同的分发路径。零特例,完美贴合"一切能力皆执行器"。

**案例**:连上 GitHub MCP → 自动多出 `gh_pr_create`/`gh_issue_search` 等执行器;连上 Postgres MCP →
多出 `db_query` 执行器;连上 Playwright MCP → 多出浏览器操作执行器用于前端造物的真机验证。AI 调它们时,
和调 `file_read` 一样经唯一脊柱,安全门、感知、记忆一视同仁。

### 推论 3:懒加载分层,保缓存前缀稳定(← 原则 3 + 内核缓存铁证)

工具分两档:**稳定核心**(file_*/shell/workflow 等,常驻注入)与**扩展/MCP/工作流**(只露名字,经一个
`tool_search` 执行器按需拉 schema)。拉取到的 schema **追加**到上下文尾部、不改前缀,使系统提示词 + 核心工具
这段前缀始终 byte-stable,命中 deepseek KV 缓存。这也顺手解决内核已存在的隐患:工作流 v2 的"按节点换工具子集"
本会改动工具定义段、破坏前缀——改成"核心常驻 + 节点工具按需露名"即可消除该破坏。

**案例**:连了 5 个 MCP server 共 200 个工具,但每次请求只注入 ~20 个核心工具 + 200 个工具的名字清单(几百字节);
AI 要发 Slack 时先 `tool_search("slack send")` 拉回该工具 schema,追加在尾部,前缀缓存不破。

### 推论 4:工具体系要支撑"看→改→验"编码闭环,而非零散堆能力(← 原则 1)

扩展不是把能力一件件堆上去,而是补全一个闭环:**看**(LSP 取类型/引用 + 代码搜索定位)→ **改**(file_edit)→
**验**(LSP 诊断 + 构建/测试)。每加一个工具都问:它补的是闭环的哪一环?补不进闭环的、与编码主线无关的
(如日历、CRM),即便生态里有,也不优先收编。

**案例**:排优先级时,LSP(看+验)> 代码搜索(看)> Web 查文档(看,解陌生报错)> GitHub MCP(协作);
而 Figma/Salesforce 这类虽在 MCP 生态里,但不在编码闭环上,二期不主动收编,留给用户按需连。

### 推论 5:代码智能分层降级,且 AI 感知当前层(← 原则 1 + `内部状态感知.md`)

通用解析的强(编译器级精确)与弱(没工具就没能力)是同一枚硬币——自定义语言/非标脚本没有语言服务器。
故代码智能不是"有 LSP / 没 LSP"的二元,而是**精度递减、覆盖递增的阶梯**:
- **语义级** = LSP + 真语言服务器(最准:类型/跨文件引用/诊断;需该语言有服务器)。
- **结构级** = tree-sitter 语法树(大纲/文件内符号导航/结构化搜索;容错解析,代码写错也能解;给自创语言写一份 grammar 即可,比写 LSP server 便宜一个数量级)。
- **文本级** = file_read + 代码搜索 + LLM 自己读(对任何文本都行,零配置,无保证)。

`lsp` 执行器**必须优雅降级**:无语言服务器不报死,退 tree-sitter → 退文本+LLM,**永远有路走,只精度递减**;
且 **AI 须感知自己在第几层**(接 `内部状态感知.md`),免得对一个根本不存在的 `findReferences` 抱虚假信心。

> 注:本推论讲的是**语言级**代码智能的底层事实源。其上还有一层**项目级**智能(本项目改一个东西的涟漪面),
> 它语言无关、凌驾这三层当"第 0 层"、并用它们当事实源——见 `01-流程即一等公民.md`。

**案例**:`.xyz` 自创脚本无语言服务器 → `lsp` 返回"无服务器" → 自动退到代码搜索 + LLM 读;
AI 被告知"按文本模式处理 .xyz",不假装能精确找全引用。
