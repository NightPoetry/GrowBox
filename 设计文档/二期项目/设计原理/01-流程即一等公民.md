# 流程即一等公民

> 二期头号原则。做任何事的同时维护可复用流程;系统能力 = 从干活里长出来的活流程库。
> 这把一期飞轮(经验→知识→理解,见 `设计/04-飞轮学习.md`)从"压缩事实"扩展到"沉淀过程",
> 并解决一种通用解析(LSP)永远给不了、却最难的代码理解——**本项目改一个东西的全部涟漪面**。

## 原则

1. **做任何事的同时维护可复用流程。"完成" = 事做好 + 这次暴露/印证的流程已建立或更新。**
   系统能力 = 自举(LLM 草稿)→ 报告-纠正回路精修 → 飞轮结晶 → 按项目持久的**活流程库**。
   (← 一期 `设计/04-飞轮学习.md` 的延伸:飞轮从压缩"事实"扩展到沉淀"过程";"动态生长的 IDE"在此落地。)

## 推论

### 推论 1:项目级智能靠学习型流程,而非通用解析(← 原则 1 + `00-工具体系扩展.md` 推论5)

通用 LSP 知道**语言语义**(符号定义/引用),但不知道**项目约定**——"在本项目加 X 要碰 A→B→C、按什么顺序;
改 Y 会波及 P、Q、R"。这些是**约定不是代码引用**,任何 LSP 都找不到;却正是改对一个东西的全部涟漪面,
也是最难、最易漏的部分。项目框架稳定时这些约定是固定的 → 可被学习型流程捕获 → 形成项目专属的"LSP",
**语言无关**,顺带填平自定义语言/非标脚本的解析缺口(凌驾 `00` 推论5 的三层、当"第 0 层")。

**案例**:本项目"加一个数值设置" = 固定碰 13 处(core Settings → 命令 → tauri-api → state 信号 → 面板回显 →
组件控件 → 四国 locale)。LSP 的 `findReferences` 永远找不到那四个 locale 文件——它们与 Settings 结构体之间
没有任何语法引用,纯是约定。一条学习型流程能,且对任何语言的项目同理。

### 推论 2:自举 + 报告-纠正回路,使建流程边际成本≈0(← 原则 1 + `设计/00-交互层.md` 如实报告 + `内部状态感知.md`)

冷启动几乎不冷,靠两个免费助推:① **LLM 读项目即能起草八九不离十的流程草稿**,不是从零;
② **每次完成任务如实报"改了哪几处",用户扫一眼即纠正**——这个纠正信号是"好好沟通"的副产品,不额外花成本。
更关键:**同一个"报告→纠正"回路既建又修**——一条过时流程下次被用、被报出来,就在同一回路里被改对。
**建设回路即自愈回路**,不必另搞维护机制。

**案例**:第一次 AI 漏改 locale、报告里写"改了 9 处",用户指出漏了四国语言 → 飞轮把"加设置"流程补全为 13 处;
下次自动带上;某天 locale 结构被重构、流程过时,下次用到、报出来,又在同一回路里被纠正。

### 推论 3:流程是一种记忆,检索复用记忆系统而非另起引擎(← 原则 1 + `设计/02-记忆检索.md`)

流程太多则"枚举式工具表"撑不住(几十个就崩)。但"从大量条目里按当前上下文找出相关的少数"正是一期记忆
已解决的问题:**把流程作为一种记忆 kind 索引**,用同一套分层检索(RAG→精确层)按语义召回 top-k;
精确层**学习型指针(正/反 K)**让常对的流程越用越好找、误触的被压制——**流程会自己学会"何时该被想起"**。
受控 kind 表加一个 `process` kind 即可(非破坏,该表本为加 kind 而设)。

**案例**:库里几百条流程,要改 Settings 时 RAG 只冒出"加设置/改设置"相关那两三条,不枚举全部;
"加设置"反复命中 → 强正 K、优先召回;某次误召的 → 反 K 压制。几千条也只浮现最相关那几条,与记忆召回同理。

### 推论 4:纪律——复发才结晶、持续合并淘汰、报告质量是进料口(← 原则 1 + `设计/02-记忆检索.md` 排除/做梦整理)

"一等公民"不等于"急着建":一看见就立流程会堆出一堆半成品、相互重叠的废流程,**比没有更糟**。
- **复发才结晶**:第二三次发现"这我好像干过"才抽成流程;一次性的不立(同记忆"别存一次性的"那条纪律)。
- **持续合并/淘汰**:近重复合并、长期不命中淘汰(同记忆 consolidation/prune)。
- **报告质量卡死学习质量**:报得含糊("改了几个文件")→ 用户无从纠正 → 流程永不精修。如实、具体报清
  改了哪几处、什么顺序,是这套学习的**进料口**(← 接 `设计/00` 如实报告)。
- **不给用户加负担**:流程在后台/飞轮维护,用户只看本就要看的任务报告,纠正是顺手的、不是被要求的
  (← `设计/00` 别频繁打断用户)。

**案例**:一段一次性的临时迁移脚本——做完不立流程(不会再用);而"出测试包"这种第三次重复的多步操作
(重建前端 dist → 仓库根 cargo tauri build → 验证 .app 时间戳)——该结晶成可执行流程,免得每次重踩"必须在仓库根"的坑。
