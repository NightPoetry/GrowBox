# 二期 · 完整实施计划(让软件真正可用)

> 把 `设计原理/` + `项目设计/` 排成可执行的分阶段路线。**目标不是堆能力,是让 GrowBox 成为一个真正能用的编程 IDE**:
> 看得见代码、改完即知错、随交互掌握本项目、并能收编生态工具。
> 每个 Phase 自成一个"可用增量"(做完就比之前更能用),按一期 phase-pacing 冷却节奏推进,每 Phase 绿了再下一个。

## 一句话蓝图

```
一期(已有内核):记忆/检索/指针 · 飞轮 · 工作流 v2 · 安全 · 内部状态感知 · 执行器脊柱
                                  │ 二期全部"焊"在这些现成件上,不另起炉灶
   ┌──────────────┬──────────────┴──────────────┬──────────────────┐
 Phase A          Phase B                      Phase C            Phase D
 看得见代码        项目级智能(独有)             工具数治理          收编生态
 LSP+搜索          process kind+飞轮            懒加载+可执行流程    MCP+tsserver+tree-sitter
 (最高ROI,先做)  (与A并行)                   (修缓存,铺路D)     (广度扩展)
```

## 依赖图(谁挡谁)

- **Phase A**(LSP / 代码搜索):**零前置**,独立子进程 + 新执行器。今天就能起。
- **Phase B**(process 建议档 + 指针 + 飞轮结晶):只依赖一期记忆/飞轮,**与 A 完全并行**。
- **Phase C**(懒加载 + process 可执行档):依赖一期 registry;懒加载不依赖 MCP。**C1 懒加载是 B 的可执行档(02 M3)和 D 的 MCP 共同前置**。
- **Phase D**(MCP 多 server / tsserver / tree-sitter):MCP 部分依赖 **C1 懒加载**;LSP 扩展部分依赖 **A**。

> 关键路径:A、B 可立即并行起步;C1 懒加载是承上启下的枢纽;D 最后。

---

## Phase A — 看得见代码(最高回报,先做)

让 AI 从"盲改 + 全量构建验证"变成"改前看清、改完即知错"。详见 `项目设计/03-LSP集成.md` + `04-代码搜索与Web查询.md`。

| 步 | 内容 | 依赖 |
|----|------|------|
| A1 | LSP 客户端管理器 + `lsp` 执行器:rust-analyzer + hover/definition/references(客户端基于 `async-lsp`+`lsp-types`,真活=生命周期编排+诊断胶水,见 `项目设计/03`) | 无 |
| A2 | 诊断推感知层:`publishDiagnostics` → `perceive` 推入上下文(编辑期失败自我感知) | A1 + 一期感知层 |
| A3 | `code_search`(ripgrep)content/files/count 三模式 | 无 |

**可用增量(Gate A)**:AI 写 Rust 时能看类型/引用、改完即被告知编译错、能快速结构化搜代码——**编码质量一个台阶**。
这一步对一个不懂 Rust 的用户价值最大:AI 自己把类型/借用错误在编辑期就看清。

**退出硬门**:`cargo test --workspace` 绿 / clippy -D warnings 零 / 前端 tsc+vite 干净 / **真机目测**:AI 用 lsp 查引用、收到一条真实诊断并据此修。

## Phase B — 项目级智能(GrowBox 独有,与 A 并行)

让 AI 随交互掌握"本项目改一个东西的全部涟漪面"。详见 `设计原理/01-流程即一等公民.md` + `项目设计/02-process-kind落地.md`。

| 步 | 内容 | 依赖 |
|----|------|------|
| B1 | 加 `PROCESS` kind(`node_kind.rs`)+ `ingest_with_role` 写入 + `retrieve` 召回注入(**建议档**) | 一期记忆 |
| B2 | process 指针接通:命中记正 K / judge 拒记反 K(复用 `PointerNet`)→ 越用越准 | B1 + 一期指针 |
| B3 | 报告-纠正回路接飞轮:任务报告"改了哪" + 用户纠正 → 飞轮结晶/更新 process 节点 | B1 + 一期飞轮 |

**可用增量(Gate B)**:AI 自动带全本项目涟漪面——"加设置碰 13 处"这类不再漏;且**语言无关**,对自定义语言项目同样成立。
这是 GrowBox 区别于通用编码 agent 的独有能力。

**退出硬门**:全工作区绿 + **真机目测**:故意漏改一处 → 用户在任务报告里纠正 → 下次该流程自动带全;过时流程被纠正后更新。

## Phase C — 工具数治理(修缓存,为 D 铺路)

让流程/工作流/工具多了也不撑爆上下文,并消除工作流 v2 的缓存破坏。详见 `项目设计/05-MCP客户端与懒加载.md`(M1)+ `02`(M3)。

| 步 | 内容 | 依赖 |
|----|------|------|
| C1 | 懒加载:`tool_search` + 核心常驻 + deferred 名单注入 + append schema;**修工作流按节点换工具的缓存破坏** | 一期 registry |
| C2 | process 可执行档:`content` 的 `wf:` 约定 + `registry.workflow_defs` 按需物化过滤 + 升格路径 | C1 + B + 工作流 v2 |

**可用增量(Gate C)**:工具/流程库可无界生长而上下文不膨胀、缓存前缀稳定;复杂稳定的流程能被检索后物化成工具去跑。

**退出硬门**:全工作区绿 + 缓存命中实测不因工具列表变化而掉(对照一期缓存铁证)+ 真机:检索到一个可执行流程并物化栈调用成功。

## Phase D — 收编生态(广度扩展,完整可用)

一次性收编 GitHub/Sentry/DB/Playwright 全家桶;前端 SolidJS 也有代码智能;自定义语言有结构兜底。详见 `项目设计/05`(M2/M3)+ `03`(M3/M4)。

| 步 | 内容 | 依赖 |
|----|------|------|
| D1 | MCP 传输 + 工具→执行器适配 + 单 server 打通(优先 github / filesystem) | C1 懒加载 |
| D2 | 多 server + 连接配置持久化 + **安全门(MCP 结果当不可信输入)** + 浏览器/DB MCP(Playwright/Postgres);web 用现成 web MCP | D1 + 一期安全 |
| D3 | LSP 扩展:tsserver(SolidJS/TS)+ 调用层级(incoming/outgoing)+ tree-sitter 第2层(自定义语言兜底) | A |

**可用增量(Gate D · 完整)**:GrowBox 成为一个连通生态的完整编程 IDE——能查 PR/线上错误/数据库、能驱动浏览器做前端验证、对前端代码也有智能、对任何语言都有"语义→结构→文本"的分层降级。

**退出硬门**:全工作区绿 + 真机:连一个 MCP server 调通其工具(过安全门)+ tsserver 给前端一条诊断 + 一个无服务器的 `.xyz` 文件走文本降级且 AI 感知当前层。

## 贯穿全程的纪律

- **每 Phase 一次"爆发",Phase 间约 30 分钟随机非整点冷却**(一期 phase-pacing 铁律,防账号过热)。
- **每步硬门**:`cargo test --workspace` 绿 / clippy -D warnings 零 / 前端 tsc+vite 干净 / 新阈值做成可设旋钮。
- **真机目测**每 Gate 必走(需用户的手 + key 端点 192.168.x.x:8080/key;`scripts/build-test.sh`,`cargo tauri build` 必须仓库根)。
- **不干扰一期**:二期只引用一期文件/内容,不修改 `设计/`、`系统架构/` 任何文件。
- **改文档非增量先备份** `设计文档/History/`。

## 建议起步顺序

**A1 + B1 并行起步**(都零/低前置、回报最直接):LSP 三件套让 GrowBox 自我开发即受益;process 建议档只依赖记忆、当天可落。
之后 A2/A3、B2/B3 收口 → C1 懒加载枢纽 → C2/D 依次。**最小可用产品(MVP)= Gate A + Gate B**:一个看得见代码、且随交互掌握本项目的编码 IDE。
