# 二期项目

> **内核之外的扩展层。** 一期(内核)= GrowBox 自身的认知体系(交互/能力/记忆/安全/飞轮/工具/多 Agent/工作流),
> 文档在 `设计文档/设计`(原理)+ `设计文档/系统架构`(工程)。**二期 = 在内核之上扩展能力**(代码智能、
> 工具生态收编等),**不属于内核**,故单独放这里,避免稀释内核的纯粹性。

## 为什么单独成区

内核是"灵魂"——一套自洽的、最少前提的认知架构,改它要极慎重。二期是"装备"——往这套架构上挂更多能力
(LSP 代码智能、代码搜索、Web 查询、MCP 客户端收编开放生态工具等)。两者生命周期、风险、改动频率都不同:
内核求稳,二期求广。混在一起会让内核显得臃肿、也让扩展受内核评审节奏拖累。分开后各按各的节奏走。

**铁律不变**:二期的每个扩展仍要落在内核公理上——尤其"一切能力皆执行器,Agent 循环是唯一脊柱"。
扩展不是开特例,而是把新能力收敛成执行器接进唯一脊柱。

## 两层文档(同内核,换更直白的名字)

| 本区 | 角色 | 对应内核 | 写法 |
|------|------|----------|------|
| `设计原理/` | 为什么、凭什么 | 同 `设计/`(原理层) | 原则 → 推论 → 案例 |
| `项目设计/` | 做什么、怎么落 | 同 `系统架构/`(工程层) | 范围/方案/接口/里程碑/接原理/风险 |

- **`设计原理/`** 先行:任何二期能力先在这里立原则、推推论、举案例,想清"为什么值得做、凭哪条公理做"。
- **`项目设计/`** 落地:据原理出可实施方案(模块边界、接口草案、里程碑、风险),**每个决策回指 `设计原理/` 的某条原则**——指不回去的就是过度设计。

## 当前二期主题

工具体系扩展(代码智能 + 工具生态收编)——让 GrowBox 从"会跑命令的 agent"升级为"看得见代码的 IDE",
并能收编开放生态(MCP)的成百上千工具。原理见 `设计原理/`,落地见 `项目设计/`。

> **★完整分阶段实施计划见 `二期计划.md`★**(A 看得见代码 → B 项目级智能 → C 工具数治理 → D 收编生态;
> 每 Phase 自成可用增量,MVP = Gate A + Gate B)。先读它就懂二期全貌。
