# 系统架构 · 写作规范

本文件夹(`系统架构/`)是**工程层**,回答"什么在哪、怎么连、为什么这么切、怎么算对"。

> 与原理层(`设计/`,回答"为什么",用 原则→推论→案例)分工不同,格式也不同。
> **铁律**:每个架构决定都必须能回指到 `设计/` 里的某条原则(见第 5 段)。指不回去的,要么漏了原则,要么是过度设计。

## 六段格式

每篇架构文档统一六段:

1. **职责** — 一句话划死边界:**只**管什么、**不**管什么(防止模块膨胀成单体)。
2. **接口** — 对外暴露的关键类型与函数签名(别人靠它使用,内部实现可随意替换)。
3. **依赖** — 依赖谁、被谁依赖(单向图,杜绝循环依赖)。
4. **数据流** — 一个真实请求从进到出如何穿过它(ASCII/箭头图)。
5. **接原理** — 本模块兑现了 `设计/` 里哪条原则的哪条推论(每条决策可回指,否则即无源设计)。
6. **已知坑** — 旧实现在这块踩过的雷 + 这次怎么绕开(可指向 `Old/` 或旧 `crates/`)。

## 模板

````
# <模块/主题>

## 职责
只管 <X>;不管 <Y,归属某模块>。

## 接口
```rust
pub fn foo(...) -> ...
pub struct Bar { ... }
```

## 依赖
→ 依赖:<A, B>    ← 被依赖:<C>

## 数据流
<进> → <步骤> → <步骤> → <出>

## 接原理
- 兑现 `设计/02-记忆检索` 原则 1(分层下沉)。

## 已知坑
- 旧 `cmds.rs`:<什么雷> → 本次:<怎么绕>。
````

## 规矩

- **边界先于实现**:第 1 段写不清"不管什么",说明边界没想透,先别写代码。
- **依赖必单向**:出现循环依赖 = 切分错了,回去重切。
- **决策必溯原理**:第 5 段为空的模块不予通过。
- **坑必记**:旧代码每个反复出错点都该在第 6 段留痕,这是这次重做的核心价值。
