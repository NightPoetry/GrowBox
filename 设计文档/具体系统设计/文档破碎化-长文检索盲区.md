# 文档破碎化 —— 长文投喂的检索盲区修复(2026-06-17,Opus)

> as-built 实况。配套:原理见 `设计/02-记忆检索.md`;本系统总述见 `memory-记忆系统.md`;
> README 顶层原理「涌现式分层」见仓库根 `README.md`。真机验证在 `crates/growbox-gui/tests/live_chunk_doc.rs`。

## 病根(dream-board 投喂文档复现)

用户把整篇技术栈/约定文档粘进对话 → 它在时间线上是**一个大节点 → 一条 e5 向量**。
一整篇多主题长文被平均、且超嵌入窗(e5 约 512 token)的尾部被直接截断,**单条向量代表不了任一窄主题**:

- **RAG 第一层漏**:窄问(如"想法卡片 class 叫什么")与这条被稀释/截断的向量余弦上不了命中阈 `0.85` → RAG 不命中。
- **精确层也漏**:该节点被 idle 嵌入后(`has_embedding=true`),某次无关线性扫扫到它 → 按"有嵌入=有索引退路、可安全跳过"染 `Deep` → 精确层 `filter(stain != Deep)` 把它踢出候选。

→ 文档**两网皆漏、永不进上下文**,AI 只能凭现有代码/常见默认瞎答(实测把 `--db-accent` 答成 `--primary`、`better-sqlite3` 答成 `sql.js`)。

> 关键反讽:2026-06-15 那次"未嵌入契约被埋"的修复把"只染已嵌入"定为规则,其隐含前提是
> **"已嵌入 ⇒ RAG 能可靠兜底"**——这前提**对超嵌入窗的大节点根本不成立**,投喂文档正踩这块盲区。

## 解法:消灭大节点,而非给它打补丁

不保留大节点再想办法让它可检索(那是治标),而是**把它按句切成小块**:各块小、单主题、向量忠实索引自身。
于是:① 窄问由 RAG 第一层直接命中聚焦块(稀释解除);② "已嵌入=已被忠实索引"重新处处成立 →
染 `Deep` 跳过重新安全(**染色不变式恢复**)。**检索侧零改动**——病因消失,两条路同时自愈。

## 管线(7 步;同步只打标,重活全在 idle)

1. **入场打标**(`memory/mod.rs::ingest_with_role`):content 字符数 > `chunk_min_chars` 且角色可破 → 置 `needs_chunk=true`。同步、廉价、**不调 LLM、不卡回合**(铁律 `llm-calls-must-not-block-turn`)。新粘贴文档在 idle 处理前靠 64K 最近 ring 兜着,无检索空窗。
2. **idle 找待破**(`gui/idle.rs::chunk_while_idle`,排在补嵌之前):靠 `NodeMeta.needs_chunk` 不触盘筛出待破节点。
3. **确定性切原子句**(`memory/mod.rs::split_sentences`):按 `。！？!?；;\n` 切,分隔符归前一句,末尾残段成句。**保证 `各句拼接 == 原文`**(零丢字),给每句隐式编号。
4. **LLM 只判破点**(`Subconscious::chunk_doc`,真实现 `gui/bridge.rs`):把编号句喂 LLM,它回"每个新块的起始句下标"JSON 数组,如 `[0,3,5]`。**它不输出任何句子内容**。
5. **Rust 精确拼接成块**(`memory/retrieval.rs::assemble_chunks`):破点 = `LLM 破点 ∪ 尺寸上限`,单遍扫描,块 = 原子句精确拼接 → **零改写零丢字**,纯空白块丢弃。
6. **各块独立成节点**(`ingest_chunk`):继承父节点 role + 项目 tag,强制 `needs_chunk=false`(防病态单句超阈无限重切)。块的向量由随后的补嵌 pass 补。
7. **父节点退场**(`Timeline::mark_chunked`:`needs_chunk=false, chunked=true`):从向量索引剔除、不再补嵌、精确层线性扫跳过。append-only 保留作溯源,不删。

## 提示词(逐字,`gui/transpile.rs::CHUNK_DOC_ZH`,进自转译目录可改/换语言)

系统提示:

> 你是文档破碎器。下面是一篇长文档**按句切好的编号原子句**(从0起)。把它切成若干**语义连贯的块**:同一主题/同一条目的相邻句归一块,主题切换处另起一块。只输出每个**新块的起始句编号**的 JSON 数组(升序,例如 [0,3,7];0 可省)。不要解释、不要改写任何句子。

用户消息(`bridge.rs::chunk_doc` 拼):`编号原子句:\n` + 逐行 `{i}. {句子(换行替空格)}`。
解析:`extract_json` 抠首段 JSON → `Vec<usize>` → 过滤越界 → 要求至少一个 `>0` 破点,否则贪心兜底。

## 保真是硬保证,切点是 best-effort

- **保真(硬)**:LLM 能影响的只有"切在哪",**绝不可能改写/丢字/编造**——内容由 Rust 按原子句精确拼接重建。这是和"直接让 LLM 输出各块文本"那类做法的根本区别(后者会重写/漏内容)。`assemble_chunks` 对 `各块拼接 == 原文` 在真机测试里断言。
- **切点(best-effort)**:`assemble_chunks` 的破点是 `LLM 破点 ∪ 尺寸上限(size_break,累计超 target 强制断)`。于是:
  - LLM 给不出有效破点 / API 挂了 → 退化成**纯尺寸贪心**(`greedy_chunk_bounds`),仍能切、绝不拼回大节点;
  - LLM 想把整篇归一块 → 被尺寸上限拦住。
- **不可破时**(`assemble_chunks` 产出 ≤1 块,如整篇一个超长句):只 `clear_needs_chunk`(清待破标、**不**置 chunked),父节点仍作唯一载体留在索引,不反复重试。

## 检索侧配套(三处剔除 chunked 父节点)

- `rebuild_index`:`filter(!meta.chunked)` —— 父节点退出 RAG 向量索引,只让小块上场。
- 补嵌 `ensure_embeddings[_batch]`:`!needs_chunk && !chunked` —— 待破的先别嵌(等成块嵌块),已破父节点不嵌。
- 精确层线性扫 `retrieve_exact` 的 `scan_ids`:`!m.chunked` —— 父节点内容已由块表示,跳过免重复。

## 循环机制(idle 批处理,非 agent 精修)

- **每篇文档 = 一次性单发 LLM 调用**,不递归、不多轮自检精修。鲁棒性来自"确定性切句 + 破点∪尺寸上限",不靠跟 LLM 来回磨。
- **系统级 = idle 批抽干循环**(`chunk_while_idle`):每个 idle tick、排在补嵌前跑;取 `Priority::Flywheel`(前台/睡眠一来即让位)、可取消;每批 `CHUNK_BATCH=4` 篇,循环调 `chunk_pending_batch` 直到返回 0。与补嵌 `embed_while_idle` 同构。

## 旋钮(推论9 数值全可设)

- `RetrievalConfig.chunk_min_chars`(默认 `CHUNK_MIN_CHARS=1500`;`Settings.retrieval_chunk_min_chars` 透传,随 `scan_max` 先例不占前端命令参数)。0 = 关闭破碎。1500 的取法:正常对话/回复几十~几百字远低于,粘贴文档常 1000+ 字高于;e5 窗约 512 token,超此长度单条向量必稀释/截断。
- 豁免:`is_chunkable_role` 排除 `process / skill / tool_memory`——这些结构化 kind 靠整节点解析(`retrieve_processes`/`retrieve_skills`/`consult_tool_memory` 解析节点头),破开会毁掉对应系统。

## 真机验证(`tests/live_chunk_doc.rs`,真 deepseek-v4-flash + 真 e5,默认 `#[ignore]` 不计费)

- **切分质量**:600 字 dream-board 技术栈文档 → 真 LLM 破成 **7 块语义连贯**(把相邻同属"CSS 命名约定"的 `--db-accent` + `.idea-card` 归一块,非机械按长度);`各块拼接 == 原文`;三窄问均 RAG 第一层命中聚焦块。
- **before/after 对照**(同篇 14 事实点文档、同 LLM+e5,判据=召回里存在"含事实且长度<整篇一半"的聚焦块):
  - **破碎前**(整篇一条向量):**0/10** —— 窄问要么召回整篇 950 字干草堆(含事实但不聚焦)、要么彻底检索不到(「端口 3000」「pm2」两问 RAG 稀释 + 精确层 miss = 0 条 = 旧版凭空答的根因)。破碎前结构上不存在聚焦小块 → 必为 0,对照公平非循环。
  - **破碎后**(真 LLM 破成 7 块):**10/10** —— 每个窄问由 RAG 第一层命中 105~234 字聚焦块。
- 全工作区 `cargo test --workspace` 绿;memory crate 单测含破碎管线 4 项(dream-board 回归 / split_sentences 还原 / 结构化 kind 豁免 / greedy)。

## 已知边界(诚实)

- 真机只验过**结构清晰、句读分明的中文技术文档**(n=2)。日志/代码/长表格/无句末符的大段连写,`split_sentences` 切不出几句,破点基本靠尺寸上限兜,可能切在语义中间——未压测。
- LLM 那一发 best-effort:挂了/返回垃圾静默退贪心,结果仍正确(能切、不丢字)但**非语义切**,质量降一档。
- 无"切完回头自检/重切"环节(单发省钱);个别刁钻文档可能切得一般,靠尺寸兜底保正确、不保最优。
- 与「涌现式分层」原理的关系:破碎**只定粒度**(把大节点切成可独立命中的原子小块),**不强加跨记忆的分类/层级**;粒度之上"哪块连哪块、先看谁"仍只由查询涌现(指针网)。
