# 交互层

> 顶层目标:**用户最小心智负担**。交互层与能力层(见 `02`)共享这一个目标——交互层把"人要做的那一下"**做小**,能力层把它**取消**。

## 原则

1. **用户最小心智负担** —— 能替用户做的,绝不让用户自己做(包括 UI 操作,也包括"把意图打成字";对话/打字本身就是负担,所以纯聊天不是终点、是兜底)。把用户的**总**心智成本压到最小:输入端少说、消费端少看、决策端少决定。
   - **约束(安全边界)**:不可逆/高危的"裁决"必须留给用户。最小负担在此让位给安全。
2. **控制反转:IDE 是 AI 的手脚** —— 不是"传统轻量 IDE 外挂一个 AI",而是 **AI 为核心、IDE 为其身体**。常规 = 人开 IDE、偶尔问 AI;这里 = AI 开 IDE、偶尔问人。

## 结构:人站两端,AI 占满中间

```
用户:意图 ─→ [ AI:听懂 → 推导 → 预填 → 执行 ] ─→ 用户:裁决
   (最小输入)            全部劳动                  (最小确认)
```

两端是人"不可代劳"之处;最小心智负担 = 把两端尽量压窄。收窄**意图端**靠上下文与记忆(系统越懂你、越不用说),收窄**裁决端**靠可逆性与预填。

## 推论

### 推论 1:意图直达 UI(← 原则 1、2)
用户用自然语言说意图,AI 直接操作真实界面(弹对话框、切项目、填字段),而非在聊天框里指路。AI 是与用户共用同一套 UI 的"第二双手"。
- **案例**:用户说"在 /Users/x/blog 下建个个人博客" → AI 直接弹出新建项目 UI;随后说"进入那个项目" → AI 自动切换过去。

### 推论 2:预填可推断、留白不确定(← 原则 1)
凡能从上下文高把握推断的字段全部填好;拿不准的留空,绝不瞎猜。成本不对称:猜错 = 用户先发现再改(两步),留空 = 用户直接填(一步)。**留空是诚实的"我不知道",不是失败。**
- **案例**:建项目时名字预填"个人博客"、读写目录预填 `/Users/x/blog`;模板、语言等拿不准的字段留空,待用户选填。

### 推论 3:结构化选项 + 永远的"我们谈谈"门(← 原则 1)
AI 必须做决定但不该独断时,弹出竖排、可扫读的具体选项(A/B/C),让用户"选"而不是"想";并永远附一个固定项"我们谈谈"。有界选项保速度,开放对话兜底"AI 把选项、甚至问题本身框错了"的情况。用户永不被关进选择题。
- **案例**:写代码遇到分叉 → 弹"这里的错误处理怎么做?A. 返回 Result;B. 直接 panic;C. 暂时忽略" + 固定项"我们谈谈"。

### 推论 4:可逆性决定自主度(← 原则 1 的约束,接通安全层)
可逆、安全的操作 → AI 直接做(零裁决)或预填 + 一键确认;高危、不可逆的操作 → AI 把一切准备好,但把"最后那一下判断"郑重交还用户。同一条"可回滚 vs 不可回滚"既管安全,也管"AI 替用户做多少"。
- **案例**:改一处 CSS 颜色(可逆)→ 直接改并展示前后对比;删除整个目录(不可逆)→ 命令准备好,但弹确认请用户拍板。

### 推论 5:一键提交,决定权留人(← 原则 1)
AI 准备到位,收尾的"确定"由用户按。理想情况用户连填都不用填,看一眼、点确定。
- **案例**:建项目场景里,所有可推断字段已预填、不确定的留空,用户不补任何信息、直接点"确定"即创建成功。

### 推论 6:活着的 IDE = 在场 + 主动 + 可对话(← 原则 1、2)
该弹时弹、该填时填、听得懂"进入那个项目"这类自然语言导航。它像一个在场的协作者在操作同一个工作台,而非侧边栏里的聊天机器人——这是中间环节被 AI 占满后的体感。
- **案例**:整个"建项目 + 切项目"过程,用户只说了两句话、点了一次确定,UI 自己动了起来。

### 推论 7:活的 IDE = UI 操作也是执行器(← 推论 1、6,接通架构公理)
推论 1、6 讲"AI 操作真实界面"是**什么**;本推论讲它在工程上**靠什么兑现**:不为 UI 动作另开一条特殊通路,而是把每个 UI 操作(开/关面板、聚焦、滚动、toast、弹确认……)都做成与文件、shell 同形态的执行器,经**同一个注册表、同一条 Agent 循环**分发(见 `系统架构/00` 公理、`设计/05 工具系统`)。LLM 对自身界面的掌控,与它对文件系统的掌控是同一套机制——"IDE 是 AI 的手脚"由此从"对外部世界的手脚"推广到"对自身界面状态的手脚"。
- **前端被动渲染意图,不自作主张**:前端不决定 UI 怎么动,而是订阅 Agent 发出的"UI 意图"事件(种子:`AgentEvent::Intent`)被动渲染;UI 状态成为 Agent 可读可写的一等资源,而非前端私有。
- **边界不变**:可逆的 UI 操作(开关面板、滚动)AI 直接做(零裁决,推论 4);真要改用户数据/不可逆的仍按推论 4 留裁决。UI 操作天然多在"可逆"一侧,所以这一层几乎全可代劳。
- **不准按名拦截**:UI 执行器必须走唯一分发路径,不得在脊柱里按名特判(架构公理复位的同一条铁律,见 `系统架构/01-core` 已知坑)。
- **案例**:用户说"太挤了,把记忆面板关了" → LLM 不在聊天框回"点右上角的 X",而是直接发一个 `close_panel` 意图,前端关掉该面板。用户省去"找按钮 + 点击"两步,只说了一句话。

### 推论 8:特殊语义的内容必须有醒目的、可区分的样式(← 原则 1 的"消费端少看")
"消费端少看"不是少给信息,而是让信息一瞥即可分类。语义不同的内容(工具调用、命令输出、等待、告警、内部消息)必须有与普通正文截然不同的醒目样式 —— 同一种样式 = 同一类信息,用户靠模式识别完成分类,不必逐字读;样式本身在传递元信息。把语义不同的东西混成一片纯文本,等于把分类负担甩回用户的眼睛。
- **案例**:shell 调用渲染成终端代码框 `$ 终端命令 cd /x && node app.js [OK]`、输出在折叠码框,与 AI 散文一眼分开。反面:把工具进度发成普通 markdown 引用块和正文混在一起,用户一眼看不出哪是命令、哪是回答(用户原话"乱七八糟")。

### 推论 9:一切数值参数都可设(← 原则 1 的"决定权留人")
> **设计原话(用户 2026-06-02)**:同时L1-L3缓存容量建议也可以在控制面板设置,一切涉及数值的都应该能设置才可以。

凡是代码里写死、影响行为的**数值参数**(缓存容量 L1-L3 / 邻域缓存总容量、注意力间隔、上下文预算 working/ring chars、max_turns / max_tokens、退避间隔、各类阈值……)都应对用户暴露为**可设置项**(控制面板 / 设置),而非硬编码。数值是"行为的旋钮",把它藏死=替用户做了一个看不见的决定;决定权留人。即时生效优先,否则标注需重连。
- **案例**:max_turns / 上下文预算 working&ring chars 已可设,是先例;推广到 L1-L3 缓存(邻域缓存)容量等全部数值。

### 推论 10:被造物层——AI 现造 UI 作为自我延伸(← 推论 7,接通自我感知)
推论 7 讲 AI 操控自身**预置**界面;再推一步:AI 还能**现造 UI**(渲染 HTML:表单/图表/棋盘)作为**被造物**(AI 临时造出的自我延伸,不是另行开发的独立软件)。两层须分清——**宿主层**(GrowBox 本体的对话 AI)≠ **被造物层**(它现造的延伸);"实时输入响应""轮询"只活在被造物层,本体绝不轮询自己的聊天框。
- **延伸 ≠ 另一个程序(关键,本推论的灵魂)**:让 AI 用**被造物**陪你玩五子棋,对手**就是 GrowBox 本体那颗大脑**——同一套智力 + 记忆,落子由主循环现想,不接任何额外 AI;反例是"做一个独立五子棋 app,再给它塞/写一个下棋 AI",那是**另一个程序**——你重造了 AI 的轮子,而且那个 AI 没有 GrowBox 的记忆与智力。被造物的价值 = **轻松复用整套庞大 AI 体系**(呼应原则 2 控制反转、与"它本身就是 AI,别再去外接 AI"那句金句)。"要不要独立成另一个程序"是另一回事——见 `创世级更新计划/内核随行-被造物毕业为独立程序.md`。
- **可视化优先**:能用界面展示/交互的就别用纯文字。用户说"陪我玩五子棋" → 直接生成棋盘,而非文字描述棋局。
- **造物感知**(自我感知原则的再泛化:AI 对**自己造的东西及其上的用户操作**也可感知):造物里的按钮/输入,AI 看得见、会反应(点提交→处理整表,落子→回应棋局)。
- **轻造物不建后台**:前端只留操作接口(回调),回调穿回同一条 Agent 循环由 AI 手动响应——AI 本身就是后台(工程见 `05` 推论 8)。
- **非必要不读取**:轮询造物的实时输入用**指数退避**(无变化就睡、变化才看),目的是**主动帮忙**(看到用户卡壳/犹豫才出手),而非一直盯——防挂机把 token 读干。
- **案例**:用户做"模拟 AI 邮箱",输地址时犹豫 → AI 在造物覆盖层弹提示、替他拿主意(`05` 推论 8);整局的点击流走有界瞬态、不堆进聊天(`02` 备注)。
