# 安全审查

> 安全不是独立哲学,而是 交互层"可逆性决定自主度"(`00-交互层` 推论 4)+ 能力层"全链路自我感知"(`01-能力层` 原则 1)在**边界**上的具体化。

## 原则

1. **默认安全,越界即交还** —— AI 在授权范围内自主行动;一旦越过边界(范围外路径、敏感路径、危险命令),就**感知到拦截 → 说明原因 → 把带范围的裁决交还用户**,绝不静默失败、也不硬闯。**根因:破坏性/越权动作的决断权属于"被破坏物/被访问物的拥有者"(用户)——AI 是代理人不是所有者,只能请求;非破坏、在授权内的才自动跑。**
2. **用户主权:只有用户能放松,AI 不能** —— 安全标准默认从严;AI 与项目配置**只升不降**,但**用户可显式授权放松**(分范围、记在项目级)。决定权永远在人。

## 推论

### 推论 1:两套门 —— 文件按目录分级,shell 按命令危险性(← 原则 1、2)
读写文件和跑命令是两件事,各有各的门,不能混用一套白名单:
- **文件读写走文件工具**,受目录分级:**可读写目录**(能读能写)/ **只读目录**(能读不能写)/ **范围外**(碰都不能碰)。每个项目自带若干目录(用户在 UI 配)。防逃逸:路径先 canonicalize 再判断(`../` 逃不出)。
- **shell / 跑二进制走命令危险性检查**,**不被文件目录白名单约束**:两条永久黑名单 —— **敏感密钥路径**(`.ssh / .aws / .env / id_rsa …`)、**危险命令**(`sudo / dd if= / mkfs / shutdown / systemctl …`)。运行 `/usr/bin /opt/homebrew/bin` 下的 node/npm/python 属正常开发,不因路径被拦。
- **别把两套门搞混**:把"文件可写目录白名单"误用来限制 shell 能引用哪些二进制路径,会把跑 `/opt/homebrew/bin/node` 当越界 —— "访问文件"(产权裁决)和"运行程序"(危险检查)是两类事。
- **案例**:AI 想写 `/etc/hosts` → 不在任何可写目录、`/etc/` 属系统区 → 拦;AI 跑 `cd /proj && node app.js` → 不是文件越界、命令不危险 → 放行。

### 推论 2:拦截可感知 → 弹窗说明 → 请求授权(← 原则 1 + 能力层感知)
被拦时 AI **知道自己被拦了**(不是静默失败,也不瞎试别的路),于是弹窗:说明"我想做什么、为什么需要、被哪条规则拦了",请求授权。
- **案例**:AI 需要写某个项目外的资源目录 → 被沙箱拦 → 弹窗"我想把图片写到 `/Users/x/assets` 以供本项目引用,该目录不在可写范围,是否授权?"

### 推论 3:三种授权范围,用户点选(← 原则 2 + 交互层最小心智负担)
用户看完原因后点一个范围(选择不填空,按影响面从窄到宽):

| 模式 | 范围 | 失效 |
|---|---|---|
| **单次授权** | 仅这一次这个操作 | 用完即失效 |
| **此项目此路径授权** | 本项目内这个路径/这条命令,持久 | 跟随项目 |
| **此项目授权** | 本项目内放开此类,持久(最宽) | 跟随项目 |

- 授权记录在**项目级**安全配置里;换项目不带过去。
- **案例**:AI 请求写 `/Users/x/assets` → 用户点"此项目此路径授权" → 此后本项目对该路径放行,别的项目仍拦。

### 推论 4:shell 命令必须经审核 —— 手动逐条 / 自动 LLM 二次审(← 原则 1、2)
凡可能破坏数据/越权的 shell 命令,执行前必须有一道"代拥有者"的审核,默认手动:
- **手动模式(默认)**:每条 shell 交用户逐条批准,批准可记忆(允许这次 / 本命令总是允许 / 信任本项目所有 shell)——拥有者裁决一次可授权后续同类。
- **自动模式(用户开启)**:LLM 安全审核员对每条命令二次审(数据安全 / 隐私 / 是否触及个人文件夹),尽量全自动;但判定触及隐私时**仍把决断交回拥有者**(弹窗)。审核员是代拥有者预筛,不替拥有者拍板。
- **硬底线先于一切**:危险命令、敏感密钥两模式都先拦,自动/信任都越不过。
- **案例**:手动模式跑 `npm run build` → 弹窗;用户"总是允许这条" → 后续不再问。自动模式跑 `node app.js` → 审核员判 safe → 自动跑;`cat ~/Documents/x.xlsx` → 判触及个人文件夹 → 弹窗。

### 推论 5:最小权限 + 隐私文件夹的加重确认(← 原则 2 + 交互层最小心智负担)
- **最小权限**:只要读就先申请读,不够再升级写;授权范围匹配实际需要,不预先多给。
- **隐私文件夹**:用户可在设置里标记若干隐私文件夹。命中(且未授权)时**必然弹窗、着重强调"这是你设置的隐私文件夹"、需二次确认**才加入项目读写——比普通授权权重更高、更难误点同意,两种模式都不绕过。
- **案例**:自动模式触及 `~/Documents` → 只申请读;若该目录是用户标记的隐私文件夹 → 红框 + 二次确认才放行。

### 推论 6:工具可见性闸 —— 可用工具集是闸门级的动态权限边界(← 原则 1、2 + `07-工作流机制` 节点工具子集)
推论 1 的两套门是**资源维度**(文件按目录、shell 按命令)的静态门;其上还有一道**调用方维度**的动态门:任何带"工具调用列表"的作用域(工作流节点,及未来的受限子作用域),其内 AI **只能调用该列表内的工具**;列表外的——哪怕已注册、哪怕本身安全——在**唯一执行闸门**(dispatch)被拒。
- **为什么在闸门**:与两套安全门同坐 dispatch 一处(架构公理:分发路径与安全门各收回唯一一处)。"节点工具子集"由此从"提示词劝 / 只是不展示在菜单里"升级成"调了也执行不了"——是 `07` 推论 1"物理锁死"的兑现层。
- **与模式无关**:不挂在懒加载等任何开关上,无条件生效。菜单裁剪(尽量只把列表内工具暴露给模型)是第一道(防呈现);闸门校验是权威 backstop(无论模式都防幻觉、防绕过):模型幻觉出列表外工具名 → 闸门挡;代码绕过脊柱循环直接调 dispatch → 闸门按其拿到的作用域同样挡。
- **主智能体 = 无作用域 = 全工具**:不在任何受限作用域时,恒放行,主链行为不变;收窄只发生在显式进入一个带工具列表的作用域(如工作流节点)之后。
- **双重身份**:对内是**降错**(AI 选不到无关工具,专注当前步),对外是**安全边界**(受限子作用域 = 只能用授权工具面的子 Agent,即便不可信也越不了界)——这是"子代理 = 受限工具子集"的落点。**并发跑多个只读子代理(调查员并行扇出)是此落点的并发形态**,见 `07-附录-并行子代理.md`。
- **案例**:一个"只读巡检"工作流节点,工具列表只列 `file_read`/`file_list`/`grep`;节点内 AI 调 `file_write`/`shell` → 闸门拒(不在列表内),即便目标路径本在可写区。把"这一步只能看"从提示词约束变成执行约束。
- **三旋钮之一**:工具域只是子 Agent 三旋钮(看什么 / 能做什么 / 返回什么)中的"能做什么";与上下文域(`07` 的 `context_mode`:inherit / isolated / fork)、返回契约(digest / full)正交组合,凑成 fork、调查员、审核员等预设——详见 `07-附录-子Agent三旋钮.md`。

## 风险评级与确认
危险程度由 `risk.rs` 评级;需要用户拍板的走 `l3ui.rs` 的 L3 确认弹窗(即上面的授权弹窗)。可逆、低危的操作直接放行,不打扰用户——呼应"最小心智负担"。
