# 记忆检索

> 核心不是 Entry/Node(那是管道,可换),而是下面的检索结构。

## 原则

1. **分层检索、反馈驱动下沉** —— 先用廉价的 RAG;RAG 找到所需、且用户反馈好(= 确实找对了)就收工,只有 RAG 不够时才下沉到昂贵的精确检索。成本最优级联:便宜的先试,贵的按需。
2. **精确层是自加速飞轮** —— 精确检索的唯一手段是"LLM 读原文判断相关"(贵),所以每次扫描的副产品全部沉淀成结构、复用到下一次——检索越用越快。

## 推论

### 推论 1:两层检索流程(← 原则 1)
- **第一层 · RAG(向量近似)**:先上。够用 + 反馈好 → 停,不下沉。
- **下沉触发**:RAG 没找到 / 用户反馈不好 / 用户显式引用历史。
- **第二层 · 精确**:先查缓存命中 → 否则沿对话时间线从近往远一段段翻,每段让 LLM 读原文判断相关(翻到最早 = 全量扫描)。

- **案例**:用户问"上次那个配置" → RAG 命中、AI 用了、用户满意 → 全程没进精确层;若用户说"不对,是更早那次" → 下沉精确层,并在当前位置建一条指向那段历史的强制跳转指针。

### 推论 2:精确层"五件套"都是沉淀下来的扫描副产品(← 原则 2)
| 机制 | 沉淀的是什么 | 一句话 |
|---|---|---|
| **指针**(原子,单向 source→target) | 走过的路 | 扫到相关 → 建捷径,下次直接跳(演进中:边带正/负 K 学习,见 `计划/指针-学习型边.md`) |
| **染色** | 扫过的区(指针的反面) | 深绿(确信无漏)/浅绿(跟指针快扫、可能漏)/无色(没到过),别重扫 |
| **缓存** | 最近的热度 | 磁盘指针图之上的**平铺单 LFU 缓存**(HashMap,按 heat 淘汰最冷,查找 O(1)) |
| **二级索引 / 碎片** | 远处拉近 + 欠债记账 | 缓存条目漂远 → 建锚点拉回;跳过的中段 = 碎片 = 债 |
| **历史引用 / 强制跳转** | 用户显式指认的位置 | 用户引用历史 → 建强制跳转指针,遍历到此必跳 |

- **案例(染色)**:全量扫描时遇到深绿区直接跳过,只扫无色/浅绿区,避免重复 LLM 读。
- **案例(二级索引)**:某节点在缓存待满 K 个窗口 → 在当前位置建二级锚点指回其原始位置;再待满 2K → 锚点改指一级索引、删除旧二级。永远只有(一级,二级)两层,中间空白即碎片。

## 维护(精确层的债务处理 ← 原则 2)
- **做梦(Dream)**:扫碎片——读间隙内容 → 潜意识 LLM 判断有无遗漏 → 补入索引 → 清碎片标记。
- **睡眠(Sleep)**:做梦 + 推演 交替,直到碎片归零 / 被用户唤醒 / 超时。推演 = 趁记忆网络干净时自问自答"用户若问 X,该检索到什么",提前建索引(会生新碎片,是好事;超量则切回做梦)。
- **疲劳度**:不看 CPU/内存,看三指标加权——缓存命中率低 + 淘汰频繁 + 碎片占比大。
- **小息(Nap,用户手动)**:清当前对话 + 邻域缓存;保留长期记忆/指针/碎片/索引/设置。"擦黑板,不格式化硬盘。"

> **缓存为何是平铺的(2026-06-02 修订,退役"三级 1:2:4")**:早期借 CPU 缓存 L1/L2/L3 的"热/温/冷分层"比喻写了"三级缓存 1:2:4",但实现是一个 `HashMap` LFU 单缓存,查找本就 O(1)——真分层对查找零收益,"最热级先查"在实现里已退化为纯展示切分。这是**比喻带来的设计惯性**:概念抄了 CPU 三级,实现发现没必要。故概念与代码统一为**平铺单 LFU 缓存**(容量可在控制面板设置,见 `00-交互层` 推论9);面板只显示 占用/容量/命中率/淘汰数。
- 对话检索、做梦、飞轮提炼**共用一个潜意识 LLM**,同时刻只一个用,优先级 **Agent > 睡眠 > 飞轮**。

## 备注:向量与精确层的关系
向量不是"要不要"的问题:**RAG = 第一层,精确扫描 = 第二层**,两层互补。旧文档说"不做向量搜索"只是在描述第二层,漏了上面的 RAG 层,故读着像矛盾。

## 备注:造物交互的记忆(默认丢 + 主动留)
造物(AI 现造的延伸 UI,见 `00` 推论 10 / `05` 推论 8)的 UI 交互**不直接进永久记忆**:
- **默认丢**——点击/输入流水进**有界瞬态**(定容环形缓冲 + 不落库,满了自动滚;同 `perceive_transient`),AI 每回合看的是**当前态**(好感度多少、在哪个场景)而非流水账。默认丢不花 LLM 判断,省 token,合"非必要不读取"。
- **主动留**——只有 AI 判定"值得永久记的结论"(定下约定、解锁结局)才 `perceive` 落时间线,打受控 kind **`artifact`**(造物,加入 `node_kind` 受控表)+ **自描述 content**(哪个造物、当时情形、虚拟时间/名字等)。content 被 embedding → 既可语义检索、又可按 kind 筛;"造物日志"查看机制后补,标签先行零迁移。
