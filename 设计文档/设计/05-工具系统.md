# 工具系统(执行器)

> "一切能力皆执行器"(`系统架构/00` 架构公理)的原理面。
> 工具系统是控制反转(`00-交互层`)的兑现:每个 IDE 能力都是 AI 可驱动、可被安全审查、可被学习的**统一执行器**。

## 原则

1. **统一执行器** —— 每个能力 = 同一形态的执行器,经**一个注册表、一条分发路径**调用。否则分发逻辑会分裂、互相打架。
2. **执行器是四个系统的交汇点** —— 它同时携带:给 LLM 的定义、安全等级、可选预填 UI、经验产出。一个点上接通 LLM、交互、安全、学习。

## 推论

### 推论 1:统一形态,一条分发路径(← 原则 1)
执行器 trait:`name` + LLM 定义(schema) + 可逆/风险等级 + `execute()` + 可选 `ui_intent()`。注册表登记全部;Agent 循环只走一条分发路径。
- **案例**:新增工具 = 实现 trait + 注册一行,不碰循环、不碰别处。对比旧代码三套 `match` 分发打架(`系统架构/00` 已知坑)。

### 推论 2:空参 = 截断,不是"调了空工具"(← 原则 1 + `实验记录/00`)
推理模型 token 不足时,工具 `arguments` 会截断成空。分发器识别空参 / `finish_reason=length` → 判为**截断错误** → 重试或加预算,绝不当"模型调用了空工具"。
- **案例**:见 `实验记录/00` 记录 7——旧代码"空参友好提示"治错了病,本次按截断处理。

### 推论 3:交互类执行器——预填 UI,选择不填空(← 原则 2 + `00-交互层`)
建项目、选项弹窗等执行器,`execute()` 不直接动手,而是 `ui_intent()` 弹出**预填好的 UI**,把"裁决那一下"交给用户。
- **案例**:`create_project` 执行器 → 弹新建 UI,预填名字/目录、留空不确定字段,用户点确定即建。

### 推论 4:每个执行器自动产出经验(← 原则 2)
执行完吐一条客观经验(工具名/参数/结果/耗时)给飞轮,不需 LLM 额外总结。
- **案例**:`file_edit` 执行 → 自动记录 path/diff/结果 → 进经验池,成为飞轮原料。

### 推论 5:安全门内建于分发(← 原则 2 + `03-安全审查`)
分发前过安全门:按执行器风险等级 + 路径分级判定——可逆直接跑,越界弹授权。
- **案例**:`shell` 执行器要跑 `rm` → 高危 → 把裁决交还用户。

### 推论 6:后台 shell 自带说明,可被直接观测(← 原则 2 + `00-交互层`)
> **设计原话(用户 2026-06-02)**:shell 具有 shell 说明,所有后台执行的 shell 都会被 LLM 打上一个 tag 以及记录原始 shell 命令,从而可以在不使用其他手段的情况下知道每个 shell 在干什么。

后台 shell(`shell_bg`)启动时,LLM 必须同时给出一个简短 tag(这条 shell 在干什么)和原始 shell 命令一并落库;两者随后台任务全程可见(状态栏计数 / 任务列表 / 审计时间线)。
- **目的**:不靠 `ps`/`grep` 等旁路手段,光看 tag + 原命令就知道每个后台 shell 的用途——避免"我怎么有两个 shell、会不会跑重了"这类困惑。
- **案例**:翻译脚本后台跑 → tag「pro 翻译四国语言」+ 原命令 `node scripts/translate-i18n.mjs`,与"等待轮询循环"那条 tag 区分开,一眼可辨,不会误判跑重。
- **双受众(用户 2026-06-02 细化,原话)**:"不仅仅是用户可见,实际上也是对内的,因为状态栏上实际上只有 shell k 就足够了,但是内部需要 tag/原命令/状态 来进行感知,因为 LLM 需要对自身及其衍生的一切具有绝对感知能力"。落点:**对外**(状态栏)**只露 `shell k` 运行计数即足够**;tag/原始命令/状态主要服务**对内感知**——`list_tasks` 工具 + perceive 让 LLM 随时拿到每个后台任务在干什么。这是自我感知原则(内部状态感知层 / `感知告知-双受众`)在 **AI 衍生物**(它 spawn 出的后台 shell)上的延伸:LLM 对自身及其派生的一切都要可感知。

### 推论 7:触发式工具——后台任务 = 你派生的软件子 Agent(← 原则 1 + 推论 6 + `00-交互层`)
> **设计原话(用户 2026-06-03,逐字见 `用户决策/决策日志.md`)**:"让AI构造后台任务的时候天然就是触发式的……即让这些东西对于LLM来说就是触发式的""相当于是LLM派生了一个纯软件的Agent，并开放接口等待它汇报""这个接口也是LLM扫描了本机接口之后……找个最不会对系统产生影响的空闲接口开放""它知道这个接口被自己开放且占用并且这个知识常驻上下文……从而让它未来编写测试软件的时候有意的避开自己使用的端口"。

**核心:后台任务对 LLM 必须是"触发式"的,而非 LLM 阻塞轮询。** 把一个后台任务看作 **LLM 派生出的一个纯软件子 Agent**——LLM 起它、开一个接口等它汇报,任务**完成时主动唤醒 LLM**(事件触发),而不是 LLM 守在那儿一遍遍问"好了没"。CPU 的那点轮询开销无所谓,**LLM token 才是主要成本**;关键是把"等待"从 LLM 身上卸到事件上。

- **完成判据 = LLM 声明的触发方式**(`DoneWhen`):`exit`(进程退出,纯 await 事件、零轮询)/ `port:N`(任务起好开端口)/ `file:p`(完成写标记文件)/ `probe:cmd`(纯软件检查指令)。**长任务/服务优先 `port`/`file`/`probe`**(让任务"完成即通知"),少用被动 `exit`;短命令 `exit` 已是天然事件。这就是"让工具对 LLM 触发式"的兑现。
- **退避是兜底,不是主路**:主触发是任务结束 `notify` → Supervisor 醒来跑一回合让 LLM 感知。LLM 仅在**工作间隔 / 自动模式**下用**指数退避**偶尔瞥一眼(`reap_hung` 查卡死/崩溃),确认子 Agent 还活着——**很符合人类盯长任务的方式**(退避节奏本身可设,见 `numeric-settable` D 组旋钮)。
- **端口选择 + 自知(承推论6 双受众/自我感知)**:接口端口由 LLM **用端口前必须先 shell 检查本机占用**(`lsof`/`ss`/`netstat`),挑空闲、不与系统服务冲突的高位端口;**经 `list_tasks` 即可辨认自己开了哪些监听端口/子 Agent**(每任务的 tag/原命令都在),据此区别对待——选新端口/写测试时避开、**别误杀自己正在监听的端口**。**无需新建记忆区**(用户 2026-06-03 拍板:list_tasks 已暴露这些信息、够用,不为此造常驻记忆)。
- **→ 多 Agent 协作方向**:"能通知 LLM 的端口形式"本质是 **Agent 间通信原语**——子 Agent 经端口向母 Agent 汇报,变相实现 LLM 间沟通,指向 **多机 Agent 协作**。这是触发式工具原则**指向**的前瞻方向,独立成篇见 [`06-多Agent协作.md`](06-多Agent协作.md)。
- **案例**:`spawn_task` 的 llm_desc 已写入上述指导(派生软件子 Agent + 开口等汇报 / 完成即通知优先 port·file·probe / **用端口前必须 shell 检查占用** / 经 `list_tasks` 辨认自占端口、区别对待别误杀);`list_tasks` 的 llm_desc 亦点明可据此辨认自己的监听端口/子 Agent。

### 推论 8:造物——现造 UI 是执行器,交互走独立通道(← 推论 7 + `00-交互层` 推论 10)
造物(AI 现造的延伸 UI,见 `00` 推论 10)的"过程半"(计算/定时/自激活)**复用已有积木**——`spawn_task` + 端口自激活(写 Python 定时、监听端口唤醒自己,见推论 7 与 `06`),无需新建;**新增的只有 UI 半**:
- **`render_artifact` 执行器**:收一段沙箱 HTML,经唯一脊柱**往返验证**(返回前端确认的渲染态,不 fire-and-forget 撒谎——否则飞轮把假成功学进记忆,是项目反复打的坑)。
- **造物交互通道**(造物 UI 的点击/输入回传给 AI 的专用消息道):与 用户/助手/内部/perceive **平级的独立道**——**不进聊天记录**(否则一局棋把坐标刷满聊天)、有界瞬态(定容易失,见 `02` 备注)、**显示在造物画布**(棋盘就动棋盘)。
- **造物覆盖层**(每个造物都有的、GrowBox 控制的 AI→用户消息槽):造物是 HTML,可永久预留一个槽(如右上角)推主动消息,**任何造物通用**、能推"造物之外的消息"。**守沙箱**:它是 GrowBox 可信层**浮在沙箱内容之上**,而非真往不可信造物 DOM 注入——接口真·永久,注入点是可信层。主动提示走**双受众**(显示在覆盖层 + perceive 进永久记忆,见 `感知告知-双受众`)。
- **边界不绕门**:造物回调只把数据回传给 AI,**不能直接触发文件/shell 执行器**——那些仍走安全门(推论 5)。
- **案例**:用户"陪我玩五子棋" → AI `render_artifact` 渲棋盘;用户落子 → 走造物交互通道(不进聊天);AI 在覆盖层说"该我了",并把"和用户约好再来一局"这种结论 perceive 进永久记忆。

## 初始工具集(从原则推导,不预设死)
文件:`file_read / file_write / file_edit / file_list`;命令:`shell / shell_bg`;记忆:`recall / memory_*`;探索:`explore_*`;项目:`create_project / switch_project`;交互:`ui_action`;调试:`debug_ui`;设置:`settings_*`。
> 工具是执行器实例,可随飞轮的 P3(最大化自动化)增删——不是写死的清单。

> **工作流即动态工具**:一个工作流注册为一个动态工具,LLM 调它即进入工作流(强制顺序的节点流转,每节点限定工具子集),让复杂/多步/易错任务结构性降错,且复用唯一注册表+唯一分发路径、零侵入。详见 `07-工作流机制.md`。
