# 能力层

> 这些原则会**写入初始飞轮**,作为种子级"理解"——系统出厂即据此行动。
> 与交互层(见 `01`)共享顶层目标(用户最小心智负担):交互层把"人那一下"做小,能力层把它**取消**。

## 原则

1. **全链路自我感知(对内 · 飞轮地基)** —— 模型始终清楚"我在做什么、得到了什么反馈"。无感知 → 无反馈 → 无经验 → 无从学习。**盲人不能进化。**
2. **自动化到闭环(对外 · 承接最小心智负担)** —— 能闭环的环节绝不让人插手;人那一下不是被做小,而是被取消。**依赖原则 1**——先看得见,才干得完。

## 推论

### 推论 1:三圈感知(← 原则 1)
模型感知三个同心圈:

- **外部世界**:每条消息带时间(最基础的感知)、每次操作的结果。
- **自身内部**:索引碎片多了、记忆压力大了、检索命中率低了——自监控。
- **自己的造物**:凡它所造(文件、UI、调试闭环……),皆可不依赖人类地感知。

- **案例(时间)**:读到一条三天前的结论 → 判断"可能已过期" → 主动安排复验。
- **案例(内部)**:检索命中率下降、碎片占比上升 → 自标"该整理索引" → idle 时后台重建,不等用户提醒。

### 推论 2:全自动闭环调试(← 原则 2,依赖原则 1)
改 → build → run → **自动截图 / 读日志 / 点击 UI / 喂输入** → 看输出 → 分析 → 修 → 复验,全链路无人值守(CLI 更简单:自动喂入、自动收出)。能无人,正因 AI 能自感知截图/日志/输出(故依赖原则 1)。
- **案例**:用户说"这个按钮点了没反应" → AI 自动截当前 UI 图 → 远程注入点击 → 截图对比 + 读控制台日志 → 定位到事件未绑定 → 改码 → 重新 build/run → 自动点击复验 → 回报"修好了,这是前后对比图"。用户全程没动手。
