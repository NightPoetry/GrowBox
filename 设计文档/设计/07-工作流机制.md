# 07 · 工作流机制(GrowBox 底层执行机制)

> 来源:用户 2026-06-04 洞察(逐字原话见 `用户决策/决策日志.md` 同日"自定义工作流机制"条:① 自定义工作流+节点限定工具子集 ② 底层机制/三层作用域/强制顺序核心 ③ 工作流即动态工具)。
> 起因:真机玩五子棋时 AI 反复工具误用(轮到落子却发 push_artifact_notice、困惑反复 render 致 CPU 345% 失控)。提示词只能"劝",治标不治本。

## 定位

工作流**不是造物的附属,而是 GrowBox 的底层执行机制**:任何"应当按既定流程做"的事都是工作流。它统一了"复杂/多步/易错任务"的可靠执行——造物对弈、造物创建、打包、回复,皆可工作流化。

## 原则

1. **★强制顺序是核心★**。工作流 = 有向节点图,节点间**强制顺序流转**(不能跳步)。每个节点 = `{引导词 + 可选工具子集 + 流转规则}`。把"该怎么做"从对 AI 的"建议"变成结构上的"约束"。
2. **工作流即动态工具**(与架构公理"一切能力皆执行器、Agent 循环唯一脊柱"融合)。`define_workflow` 把一个工作流**注册成一个动态工具**(name = 工作流名);LLM 像调普通工具一样调用它即进入工作流,**复用唯一的工具分发路径,零新机制**。

## 设计的巧妙之处:灵活 · 可靠 · 不侵入,三者兼得

通常这三者互相矛盾——越灵活越难保证可靠,越想可靠越要约束、约束又容易侵入既有系统。工作流机制**同时拿到三者**,关键全在"工作流即动态工具"这一个统一抽象:

- **足够灵活**:LLM 自定义节点 / 工具子集 / 流转 / 触发,叠加三层作用域 + 可嵌套 → 能表达**任意**"既定流程"(对弈、创建、打包、回复……),而不是写死的几条内置套路。
- **足够可靠**:强制顺序 + 节点工具子集**物理锁死**选择空间 → 把"劝 AI 别用错工具"变成"AI 根本选不到错的"。**灵活不以牺牲可靠为代价**(自由度在"定义工作流"时释放,执行时被节点收紧)。
- **不破坏其他系统**:工作流本质是一个**动态注册的执行器**,复用唯一注册表 + 唯一分发路径;Agent 脊柱只多一个 `current_node` 状态做工具过滤——**不碰任何现有执行器、记忆系统、安全门**。零侵入。

一句话:**灵活来自"LLM 自定义",可靠来自"节点约束",不侵入来自"复用工具机制"——三者由"工作流即动态工具"这一抽象统一。** 这是本机制最值得珍视的地方。

## 推论

1. **结构性降错**(原则1 的直接结果)。节点的工具子集**物理锁死**了选择空间:五子棋"AI 落子"节点的工具表里只有 `artifact_command`,**没有** `push_artifact_notice`/`render_artifact` → AI 想误用也选不到。这比提示词"劝它别用错"根本——提示词是建议,工具子集是约束。

   ★"物理锁死"锁在哪一层★:① **菜单**(`tools_for`)是第一道——默认(懒加载关)直接把暴露给模型的工具裁成节点子集,模型连别的都看不到;懒加载开时菜单取稳定前缀(缓存优先)不随节点裁。② **唯一执行闸门**(`dispatch`)是权威层——无论哪种模式,都按当前作用域的可用集校验 membership,列表外的工具调了也拒(防模型幻觉工具名、防代码绕过脊柱循环)。闸门校验**与懒加载等模式开关无关**,是**基础设施**:任何带工具列表的作用域(节点、未来受限子作用域)同走一条规则(见 `03-安全审查` 推论 6)。由此节点工具子集有**双重身份**:对内"降错"(选不到无关工具),对外"安全边界"(受限工具面 = 子代理隔离)。
2. **三层作用域**:
   - **全局工作流**:系统内置的各种默认工作流(如"造物创建工作流""回复工作流"),所有项目可用。
   - **项目工作流**:项目特定(如"打包工作流":既定顺序+既定上下文+既定工具集+既定指导)。
   - **造物工作流**:某造物专属(如"五子棋对弈工作流"),存进**造物文件夹**(见造物文件夹设计),随造物生灭。
3. **统一触发(因"工作流即动态工具")**:① **工具调用**——LLM 主动调"工作流工具",走现成的唯一分发路径;② **端口/事件**——如造物交互回调(用户落子)触发进入工作流某节点。一触发跑整套强制流转。
4. **可嵌套**。某节点的工具子集里可以包含**另一个工作流工具** → 工作流调工作流(如"造物创建工作流"的某步调"造物自检工作流")。天然组合。
5. **省 token + 稳缓存**。每轮只暴露节点的工具子集(而非全集)→ prompt 更小、前缀更稳 → deepseek KV 缓存命中更好(见记忆系统缓存设计)。
6. **融入脊柱不破坏公理**。Agent 循环只多一个 `current_node` 状态:节点内每轮 → `tools_for` 把暴露给模型的工具裁成 `node.tools` 子集(始终保留 finish/ask_user 防卡死)+ system 注入 `node.prompt`;AI 调工具 → **脊柱循环派发前 + 唯一执行闸门 `dispatch` 两处按节点可用集校验(列表外即拒,见推论1 与 `03-安全审查` 推论6)** → 按 `next` 强制流转;到 END/finish → 退出工作流回普通模式。仍是同一条 Agent 循环、同一个注册表、同一条分发路径。

## 案例

- **五子棋对弈工作流**(造物作用域):`节点[等用户落子] --(端口:落子)--> 节点[AI落子]{引导:"轮到你(白),据造物状态分析下最佳一手", 工具:仅 artifact_command} --(调了 artifact_command)--> 节点[判胜负]{工具:artifact_command,finish}`。"AI落子"节点无 notice/render → 不会再发通知卡住、不会反复重画失控。
- **造物创建工作流**(全局作用域,强制顺序的典范):`先设计好工作流 → 再设计造物 → 据反馈修改工作流 → 反复,直到造物无法被继续优化`。即"创建造物"这件事本身被一个工作流约束,每步只给该步的工具+引导。
- **打包工作流**(项目作用域):`清理 → cd frontend 建前端 dist → 回仓库根 cargo tauri build → 验证产物`。强制顺序 + 每步既定指导,正好根治"cargo tauri build 必须在仓库根、不在 frontend 子目录"这类反复踩的坑(见 CLAUDE.md 出包已知坑)。

## 数据结构与接口(工程提要)

```
Workflow { id, name, scope: Global|Project|Artifact, nodes, entry, triggers }
Node     { id, prompt, tools: Vec<String>, next: Vec<Transition> }
Transition { on: ToolCalled(name)|Always, to: NodeId }
Trigger    { port: String, to: NodeId }
```
- `define_workflow{name, scope, nodes, entry, triggers}`(执行器):LLM 定义并注册为动态工具;内置全局工作流随系统出厂。

工作流落地后,以下"治标补丁"由工作流的结构约束接管(逐项见 `用户决策/决策日志.md` 及代码注释):

|---|---|
| 提示词"劝"分工(command 操作 vs notice 评论) | 节点工具子集**无 notice** → 物理选不到(推论1) |
| 造物回合 `max_turns.min(4)` 硬限轮 | 工作流有限节点**强制流转**,天然收敛(原则1) |

# 加强版:栈函数工作流(v2)

> 来源:用户 2026-06-05 洞察(逐字原话见 `用户决策/决策日志.md` 同日"加强版工作流机制"条)。
> 一句话:**把工作流从「共享上下文的嵌套」升级成「带类型化输入/输出契约的栈函数」**——上层像调用函数一样调用下层,
> 调用前 LLM 决定喂全量还是裁剪的上下文,下层必返回上层规定格式的结构化结果。于是大量**写死的内部机制**
> (自动命令安全检测、写码先看什么……)都能解耦成工作流 → **动态生长的 IDE**(AI 自创作工作流、自调用工作流)。

## v1 与 v2 的差距(为什么要升级)

P1-P3 的嵌套(`wf_stack`)已经能"工作流调工作流",但子工作流与父**共享同一份 messages/memory 上下文**,
"返回"只是控制流(父续流转),**不携带任何值、无类型化输入、无上下文隔离**。这对"五子棋对弈"够用,但要把
"命令安全判断"这种**子程序**做成工作流就不够:子程序需要(a)被喂**特定**输入(全量或裁剪),(b)产出**结构化结论**
回灌给调用者。v2 补上这两点 = 把"工作流即动态工具"升级为"**工作流即动态函数**"。

## ★最高设计原则:最少充分信息(2026-06-05 用户定调)★

> 让工作流**尽可能少用巨量上下文**,把各种复杂的无用内容**隔离在自己的上下文里、退出即丢弃**。
> 例:用 shell 装程序时大量成功打印,只给上层返回**失败/报错/警告**;除非上层(经用户意图)明确需要看全量,
> 才要求分支**全量返回**——全量返回是**特殊状态**,不经分支 LLM 整理(避免无价值的"LLM 搬运"),原始内容直通,
> **慎用**(易污染主上下文)。大部分情况靠"给分支额外上下文 + 让分支自己判断"交互即可。

这条原则统辖 v2 全部机制:isolated 裁剪输入、退出截断、digest 默认返回、full 特殊直通、分支不写主记忆——都是它的推论。

## 原则(在 v1 之上新增)

3. **工作流即动态函数**(原则2 的强化)。入口工具的**签名**新增可选参数,LLM 调用时填:
   - `input`:喂给被调工作流的输入(LLM 当场撰写——**裁剪**只给关键信息是常态,最少充分信息)。
   - `return_spec`:**预留的返回指令**——告诉被调工作流"我要你带回什么、什么格式"。
   - `context_mode`:`inherit`(默认,看得到父全量上下文、分支工作留在父)| `isolated`(只看 `input` + 系统提示,父对话被隐藏,退出即丢)| `fork`(看得到父全量上下文**但**分支工作不回灌、退出只回摘要——适合需要你全部背景的繁重机械活;见下方「子 Agent 三旋钮」与附录)。
   - `direct`(原则6)、`max_loops`(原则8)见下。
4. **结构化返回原语 `workflow_return{value, full}`**(新控制信号,可读属性的来源)。被调工作流收尾调它产出返回值,
   脊柱据此**出栈** + 把结果回灌进父上下文(父下一轮即读到)= "结构化调用必然有一个可被读取的属性"。两种返回:
   - **digest(默认 `full=false`)**:isolated 帧**截断分支工作消息**(退出即丢),只回灌 `value`(最少充分信息=错误/警告/结论)。
   - **full(`full=true`,特殊状态)**:**不截断**,分支原始工作上下文**直通回父**(零 LLM 搬运)。慎用——污染主上下文。
   (`workflow_return` 仅在工作流内可见;普通模式过滤掉。)
5. **函数式契约**:上层据 `return_spec` 规定格式,下层 `workflow_return` 兑现;格式遵从由**节点引导 + return_spec**
   软约束(LLM 友好,不上死 schema,与全系统"提示词驱动"哲学一致;需要硬校验时可后加 JSON Schema)。
6. **栈调用 vs 直接调用**(2026-06-05)。两种调用方式:
   - **栈调用**(`direct=false`,默认):压栈进入,被调**完成后返回到调用方**继续——函数调用语义。
   - **直接调用**(`direct=true`):**尾调用**,顺流而下,**替换栈顶而非压栈**(栈深恒定),**不返回调用方**——
     用于"一直往下走/循环"。**主智能体循环本身就是直接调用一直循环**(若压栈早溢出)。实现=进入前先优雅出栈当前帧再压新帧。
7. **记忆模型:只有主智能体被全量索引**(2026-06-05)。**直接调用链永远只有一条 = 主智能体**(高感知、涉用户交互)→
   **全量索引进主记忆**;**派生分支(栈调用帧)的逐步工作转录不自动写主记忆**(噪音隔离、退出即丢),它对主记忆的贡献=
   **返回值**(主智能体读到即记)。分支也可**主动写**记忆,但**编织成"动作"而非语言**("我探索了 X 发现 Y""我编译 X 报错 Z");
   除非主智能体在 `input`/`return_spec` 里灌输"需记住什么"。落地=`WfFrame.is_branch` + 脊柱 `in_branch` 门控 ingest。
8. **分支内循环 + 循环上限**(2026-06-05)。分支(栈调用进入)内可用**直接调用循环**处理某事,直到达标 OR 达到
   主 LLM 设的 `max_loops`(-1=无限,慎用),然后**返回自己的入口栈**(workflow_return)完成整次调用。
   直接调用保持栈深 + 链身份,故 `workflow_return` 正确弹回调用方;超 `max_loops` 引擎**强制返回**(防失控)。
9. **分支不与用户对话 + 项目级日志**(2026-06-05)。分支链**原则上不与用户对话**——展示给用户的思考**只有主链**内容,
   分支再长也不是主链,其复杂性被**内部包装**。但分支的**全部调用信息原样存入项目级日志文件**(含调用时上下文),
   随项目文件夹走,**环形覆盖**,**默认上限 25G,-1=无限**(可设)。即:主记忆只存摘要,原始转录入可轮替日志,细节不丢。

## 栈帧升级(`WfFrame`)

`wf_stack: Vec<(String,String)>` → `Vec<WfFrame>`(as-built,`crates/growbox-gui/src/agent/mod.rs`):

```
WfFrame {
    wf: String,            // 工作流名
    node: String,          // 当前节点
    isolated: bool,        // context_mode=isolated(隐藏父上下文 + 退出截断)
    fork: bool,            // context_mode=fork(看得到父、但退出截断分支工作=只回摘要;与 isolated 共用"截断"、不共用"隐藏")
    msg_base: usize,       // 进入本帧时 messages 的长度(隔离切片的地板 + 返回截断的锚点)
    prev_floor: usize,     // 进入前的 context_floor(出栈时恢复,支持嵌套隔离)
    is_branch: bool,       // 栈调用=true(派生分支,记忆门控) / 直接调用·主链续=false
    max_loops: i64,        // 直接调用循环上限(-1=无限),栈调用时主 LLM 设
    loops_used: i64,       // 已用循环次数(尾调用替换帧时继承+1),达 max_loops 强制返回
}
```
> `input`/`return_spec` 不存帧上,而是**进入时作为一条 system 消息注入 messages**(节点引导 + `[调用方输入]` + `[要求返回]`),
> 这样它自然进入 isolated 切片区、append-only 缓存稳。`EnteredWf` 只是入口工具调用参数的临时解析结果(传给压栈逻辑)。

## 上下文三态(inherit / isolated / fork)的实现 —— 两个正交开关

关键:`context_mode` 其实是**两个正交二元开关**的组合,不是一个三选一的整体:
- **是否隐藏父上下文**(进入时 `context_floor` 抬不抬):隐藏 = isolated;不隐藏 = inherit / fork。
- **是否退出截断分支工作**(返回/出栈时丢不丢分支 append 的消息):截断 = isolated / fork;保留 = inherit。

三态 = 这两个开关的三个有意义组合(第四个"隐藏父 + 保留工作"无意义,不暴露):

| 模式 | 看父上下文 | 分支工作退出后 | 适用 |
|---|---|---|---|
| `inherit`(默认) | 看得到(不抬 floor) | 留在父(不截断) | 轻量续做,分支工作就是想留下的延续 |
| `isolated` | 看不到(抬 floor) | 截断丢弃(只回摘要) | 自洽的调查/审核子任务,裁噪音、可并行 |
| `fork` | 看得到(不抬 floor) | 截断丢弃(只回摘要) | 需父全部背景的繁重机械活,什么都知道、只把结论交回 |

每轮请求按**栈顶帧 + context_floor** 切片:`messages[0..SYSTEM_PREFIX_LEN]` ++ `messages[context_floor..]`。
- `context_floor` = 仍在栈上的**最近一个 isolated 帧**的 `msg_base`(没有则 = 1)。**fork/inherit 帧不抬 floor** → 看得到父全量。
- **出栈/返回**:`isolated || fork` 帧 → `messages.truncate(msg_base)` 丢弃分支工作(父不被分支内部污染),push "工作流「X」返回:`value`";`inherit` 帧 → 不截断(分支消息留在历史)。三者都 `context_floor = prev_floor` 复位。`full=true` 是对截断的特例豁免(原始直通,慎用)。

> **fork = inherit 的"看" + isolated 的"丢"** —— 继承全量上下文(它什么都知道),但退出只回摘要(机械噪音不回灌父)。
> 这正是"繁重子任务要全背景、但别拿过程噪音污染我"的解。落地 = 解耦"隐藏父"与"退出截断"两个开关(`fork` 帧 isolated=false、fork=true)。

> 这套切片**零侵入** append-only KV 缓存:`messages[0..1]` 与 `messages[floor..]` 都是 byte-stable 前缀的拼接;
> isolated 子工作流自成一段稳定前缀,返回后父恢复其原前缀,缓存按段命中(见 [[deepseek-cache-and-reasoning]])。

## 子 Agent = 三旋钮的函数调用(综合)

> 把 v2 的上下文模式、`03` 的工具可见性闸、返回契约**收成一个统一视角**:**一个派生子 Agent = 一次类型化函数调用,由三个正交旋钮定义**——

1. **看什么(上下文域)**:`context_mode` = inherit / isolated / fork(上一节)。
2. **能做什么(工具域)**:节点工具子集 = 闸门级可用工具集(见 `03-安全审查` 推论6 工具可见性闸)。
3. **返回什么(返回契约)**:`return_spec` + `workflow_return{value, full}`(digest 摘要 / full 直通)。

(隐含第四旋钮:**用哪个模型**——自洽的调查/审核分支可挂更便宜的潜意识模型槽,fork 用主模型干你的活。)

常用的两个**预设**只是这些旋钮的两组取值:
- **fork**(繁重续做)= 看:inherit/fork(继承全境)· 做:全工具或按需 · 回:digest。需要我全部背景的机械活(大范围重构、反复编译),它什么都知道、只把结论交回、过程噪音不回灌。
- **调查员 / 审核员**(自洽子任务)= 看:isolated(只看 input)· 做:只读工具子集 · 回:digest。给精确简报、只回结论;可并行扇出、可用便宜模型。
  两个预设的差别**只在旋钮①**(看不看得到父)——这说明它们不是两个机制,是同一函数调用的两组取值。

**何时派生 · 用哪个预设(教给模型的判断规则)**:
1. 值不值得外派?——会产生大量工具噪音 / 能并行 / 有干净交付物 → 外派;否则就地做。
2. 需不需要我此刻积累的全部上下文(计划、已定决策)?——需要(我工作的延续)→ **fork**;不需要(自洽查询,看一眼代码 + 一份简报就能答)→ **调查员/审核员**。
3. 提醒:① 成本不对称,**默认调查员**(fork 每轮背全上下文,贵);② **欠简报是调查员头号失败**——要写巨长简报才够就是该改 fork 的信号;③ 调查员可并行(一回合发多个 isolated 调用 = 并发扇出,机制见附录 `07-附录-并行子代理`),fork 通常串行。

完整能力矩阵(两正交开关 × 三返回 × 工具域 × 模型)+ as-built 映射见附录 **`07-附录-子Agent三旋钮.md`**;调查员**并发扇出**(一回合多个 isolated 调用 → 并行跑、汇聚摘要)的机制见附录 **`07-附录-并行子代理.md`**。

## 分支(已天然支持,v2 只是用法)

"分支类型、各分支有自己的顺序、分支的分支" = 现有 `Node.next: Vec<Transition>`(`ToolCalled|Always`)的条件流转
+ 嵌套工作流调用的组合。一个节点可据"调了哪个工具/进了哪个子工作流"分流到不同节点/子工作流,每条分支是一段
自己的强制顺序;子工作流内部又可再分支 = 分支的分支。**v2 没有新增分支原语,新增的是"分支可作为带 I/O 契约的函数调用"**。

## 案例:Auto 编程工作流末尾的命令安全判断(用户原例)

```
[Auto编程...] --(要执行命令)--> [命令安全判断]{调子工作流 command_safety,
    input=待执行命令+必要环境信息, return_spec="返回 {safe:bool, reason, needs_full_ctx:bool}"}
command_safety 工作流(子函数):
  入口[分流]: 涉及脚本? --是--> [脚本审查]{context_mode=inherit/喂全量, 工具:file_read,shell(dry)} 
                        --否--> [直接判断]{context_mode=isolated/裁剪, 仅据 input}
  [脚本审查]/[直接判断] --> [判环境]{用户环境? 常规环境(Python)? 会损坏当前环境?} --> workflow_return{ {safe,reason} }
```

上层据返回的 `{safe:bool,...}` 决定是否真执行——**像调函数读返回值一样**。`command_safety` 是可复用子函数,
任何工作流要"判命令安不安全"都调它,**写死的命令检测逻辑被这一个工作流取代**(解耦 + 复用)。
