# 设计原理 · 写作规范

本文件夹(`设计/`)是**原理层**,回答"为什么"。所有文档用「**原则 → 推论 → 案例**」三层,自上而下、由抽象到具体。

这与系统自身的**压缩谱**同构:

- **原则 = 理解**(最少前提、最广适用)
- **推论 = 知识**(可操作的模式)
- **案例 = 经验**(具体事实)

读文档(原则→推论→案例)正好是系统压缩认知(经验→知识→理解)的逆过程。**写原理文档 = 给飞轮播种**——这些原则会作为种子级"理解"写入初始飞轮,系统开机即据此行动。

> 工程层(`系统架构/`)问的是"什么在哪、怎么连",用另一套格式,见该文件夹的 README。

## 三层

1. **宏观原则(K 条,越少越好,通常 1–3)** — 本主题不可再约简的根本立场,每条一句话能说清。
2. **推论(若干)** — 从原则机械推导出的具体主张,每条必须回指它来自哪条原则。
3. **案例(挂在每条推论下)** — 用真实场景把推论坐实。一条推论至少一个案例;没有案例的推论是空话。

## 模板

````
# <主题>

## 原则
1. <一句话原则>

## 推论
### 推论 1:<标题>(← 原则 1)
<一两句说明>
- **案例**:<场景 → 行为 → 结果>
````

## 规矩

- **先原则后细节**:扫完"原则"就抓住全局,细节可跳读。
- **推论必挂原则**:推不出来的"推论",要么补原则,要么删。
- **案例必具体**:可复现、可指认,不用"好/坏/成功/失败"这类评价词。
- **精炼**:能压则压;重复的推论合并;过期的删,不留尸。
