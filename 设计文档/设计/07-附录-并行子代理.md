# 附录 · 并行子代理(调查员并发扇出)

> 本附录是 `07-工作流机制.md`「子 Agent = 三旋钮的函数调用」一节中**旋钮①的 `isolated` 取值的并发实现**展开。
> 一句话:**一回合发出多个 `context_mode=isolated` 的调查员调用 → 它们各自作为一次性子运行并发跑、各回一份摘要 → 汇聚回灌主线**。这把 07 主文与 `07-附录-子Agent三旋钮` 里一直写着的"调查员可并行扇出"从**设计意图**落成**真机制**。

## 缺口:设计早写了"可并行",实现一直没给

07 上下文三态表(`isolated` 行「可并行」)、三旋钮决策规则(提醒③「调查员可并行,fork 通常串行」)、子 Agent 附录预设表都写着调查员**可并行扇出**。但 as-built 一直是:

- 主循环是**单脊柱**,`wf_stack` 是**顺序栈**,一次只有一帧活着;
- 一回合即便模型(DeepSeek 本就支持并行 `tool_calls`)真发出多个工作流入口调用,脊柱里 `entered_wf` 也是**单个 `Option`**(后者覆盖前者)→ 多个被压成一个、其余被丢。

于是模型照着"可并行"去派多个调查员,机制收成一个 → 控制流错乱(尾调用转过去没干净返回)→ 模型只好自己手动顺序做、嘴上还演一个并行小队。**这道缺口就是本附录要补的。**

## 核心机制:一次性子运行 + 单任务并发 I/O

判据:**一回合的 `tool_calls` 里有 ≥2 个 `context_mode=isolated` 的工作流入口调用** → 进入并行批,而非单帧栈路径。

每个调查员 = 一次**一次性子运行**(ephemeral sub-run),run-to-completion:

- **自带空的一次性 `Memory`**(`Memory::new()`):它不读、不写主记忆。隔离由此天然成立 —— 空记忆 → `assemble_context` 检索不到东西 → 上下文 = [系统提示 + 简报 + 它那个工作流的节点引导],父对话根本没传进去。跑完即弃。
- **工具按它的工作流节点收窄**(只读子集),经唯一执行闸门 `tool_in_scope` 锁死(见 `03-安全审查` 推论6)—— 它写不了、也派生不了再下一层(防 fork-bomb,见「边界」)。
- **静默**:用 muted sink,思考/正文不展示给用户(同分支展示门控);细节进分支日志。
- **贡献 = 返回值**:跑到 finish / `workflow_return` / END,`AgentOutcome.final_text` 即这份调查员的摘要(digest)。

主循环把这一批 N 个一次性子运行并发跑、收齐,再把**每份摘要按各自的 `tool_call_id` 回灌成 `tool_result`**,主线下一轮就看到 N 份结论、继续。

> **为什么能真并行、却不需要线程 / Arc / Sync**:N 个子运行 future 在**同一个异步任务**上 join(限流并发)—— 协作式并发。某个子运行 `await` LLM 的 HTTP 响应时,其它子运行继续推进 → **多个 LLM 请求同时在飞**(LLM 调用是 I/O 密集,这正是协作并发的甜点)。共享的 `&Registry` / `&dyn LlmDriver` / `&Sandbox` 是**只读借用**,N 个 future 同时不可变借用合法;各自的 `&mut 一次性 Memory` 是各自的局部,互不冲突。**没有跨线程 → 不需要 Send/Sync,不需要 Arc/Mutex。** 这是整件事低风险的根因(对照:若让 isolated 调查员也读写主记忆,就得 N 个 `&mut Memory`,根本编译不过 —— 是"isolated 不碰主记忆"这条既有铁律解锁了并发)。

## 单一脊柱公理:没破,反而更纯

架构公理 = "Agent 循环是唯一脊柱"。并行调查员**不违背**它:

- 调查员是**一次性的、无记忆的、立即 join 回来的 worker**(等价于一次并发 `map`),不是常驻的第二条脊柱;
- 它们**不写主记忆、不与用户对话、不持久、不递归**(不能再派下一层并行);
- 跑完即弃,主线仍是**唯一**持久脊柱。

这与 fork/inherit 的区别正好对齐三旋钮:**fork/inherit 是"我工作的延续"** → 必须在主线**就地**(in-line 栈帧,看得到 / 接着写父上下文),天然串行;**isolated 是"自洽子任务、只回结论"** → 可外派、可并发。**所以"isolated 可并行、fork 串行"不是规定,是两种语义的必然推论。**

## 边界与旋钮

- **并发上限 `parallel_max`**(默认 4,**可设** —— 铁律"一切涉及数值的都应能设置",见 `numeric-settable-and-cache-flat`):一回合最多几个调查员同时在飞,其余排队跑完。控成本 / 控速率限制。
- **只 `isolated` 进并行**:inherit/fork 仍是既有 in-line 栈帧(零改动、向后兼容)。`command_safety` 等既有 isolated **单**调用也不变 —— **只有"一回合 ≥2 个 isolated"** 才走新并行批。
- **无嵌套并行(防 fork-bomb)**:调查员的工具子集(只读)里没有派生工作流的能力 → 它发不出 isolated 入口调用 → 不会再扇出一层。纵深上再加一道:子运行内禁用并行批。
- **取消传播**:主线「终止」→ muted sink 的 `cancel_flag`(= 主线那个)→ 各调查员的 `drive_one` 流式循环秒级收手。
- **展示**:用户看到的是聚合状态(如"N 个调查员并发勘探中…" → "调查员已返回 N 份结论"),不是 N 路刷屏。

## 内置 `investigate` 工作流(给临时勘探一个可调对象)

真机那个 bug 的场景是"勘探一个代码库",模型手上没有现成的"调查工作流"可调。补一个内置 `investigate` 工作流(单节点、只读工具子集、引导 = 按 `input` 调查并据 `return_spec` 回结论)。模型在一回合里调它 N 次、各喂不同 `input`、`context_mode=isolated` → 并发 N 个勘探。**一个机制(带旋钮的工作流调用)同时覆盖结构化调查员(command_safety 式)和临时勘探(investigate)**,不引入新原语。

## as-built 映射(Phase 2 落地点,`crates/growbox-gui/src/agent/mod.rs`)

- 入口 `tool_calls` **预扫**:数出本轮 `isolated` 工作流入口数;≥2 → 收集 `Vec<(call_id, EnteredWf)>` 走并行批(跳过单帧 push 与即时 ack);否则既有单帧路径**逐字不变**。
- `run_investigator(ent, 共享服务, cancel) -> String`:空 `Memory` + **直接调 `run_agent_loop`**(同模块私有脊柱;`seed=简报`、`seed_internal=false`=必须执行才停、`initial_wf=Some((wf,entry))`=带工作流节点工具收窄、`muted_sink`、`transient=true`=不碰记忆机器)→ 返回 `final_text`。**异步递归**(脊柱 → 调查员 → 脊柱)用 `Box::pin` 在调用点打断,免无限大 future。
- `MutedSink`:实现 `EventSink`,展示类方法 no-op、`cancel_flag` 透传父、`open_terminal` / `request_decision` / `ui_round_trip` 一律否定(调查员只读,不该触发授权 / UI / 终端)。
- 并发限流:`futures` 的 `buffer_unordered(cfg.parallel_max)`(若未引入 `futures`,等价 = 按 `parallel_max` 分块 `join_all`,或 `tokio::sync::Semaphore` + `join_all`);输出带 `call_id`,收齐后逐个 `messages.push(tool_result(id, digest))`。
- 配置:`AgentConfig.parallel_max`(core `Settings` 默认 4)+ 设置 UI 旋钮(复用 `set/get_misc_config` 模式)+ 四语。
- 技能 `delegate-subagent` 措辞点明"同一回合发多个 `isolated` 调用 = 并发"(旧措辞"可并行扇出"在新机制下本就成真,这只是讲清怎么触发)。
- (交付细节,非阻塞)`skills::ensure_seed_nodes`:已物化的种子若正文变了应刷新,使技能修订能到达**既有库**(不覆盖用户 `learn_skill` 同名版)。非阻塞 —— 旧措辞已成真,刷新只是让指引更清。

## 测试矩阵(`agent/loop_tests.rs`,Phase 2)

- 一回合 2 个 isolated 调用 → 两份摘要都回灌为各自 `tool_result`(并发批生效)。
- `parallel_max=1` → 仍出两份(退化为顺序,结果不丢)。
- 主线 cancel → 调查员全停、不挂死。
- 调查员**不写主记忆**(批跑完主记忆节点数不变)、父上下文不被分支噪音污染。
- 单个 isolated 调用 → 仍走既有单帧路径(向后兼容,既有 isolated 测试不变)。

## 隐含第四旋钮(未接线)
