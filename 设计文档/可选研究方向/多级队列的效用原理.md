# 多级队列的效用原理

> 状态:**只记录原理,不实现**(2026-06-15)。三级队列的实现已取消;但"多级队列为什么有用"
> 这个道理值得留存,免得将来又想用却忘了它的用途。
> 性质:热索引 / 工作集淘汰策略的原理笔记。本文只讲"为何",不含落地代码。

---

## 1. 背景:为什么补这一篇

- 本项目早先**半设计过一个多级(三级)队列**用于热索引,但只设计了一半,后来**用途被遗忘、实现也取消了**。
- 用户重新口述了这套队列的机制与目的,这里把**效用原理**固化下来,作为将来若要给"热索引 / RAM 工作集"
  做"抗冲刷保留"时的参考。
- ⚠️ 别和 2026-06-02 退役的那个"**三级缓存 1:2:4**"混淆(见 `计划/precision-layer.md`):
  那次退役的是"按 heat 的**展示分层**"(HashMap 已 O(1),分层零收益)。**本文说的是另一回事 ——
  分段队列的"二次机会"淘汰机制**,价值在淘汰策略,不在展示。

## 2. 要解决的问题:单级 LRU 只认"近",不认"反复"

一个长度固定的单级 LRU,只按**最近性(recency)**淘汰。它的毛病:

- 一阵**一次性的近期流量**(例如一次大查询 / "大水漫灌"全量检索带进来的一堆只用一次的条目),
  会把队列冲满,**把那些"跨度长、但反复在用"的重要条目挤出去**。
- 即:单级 LRU 会被突发流量"冲刷(scan / burst pollution)",丢掉真正高频但间隔长的条目。

## 3. 多级队列的机制(用户原话)

- 热索引 = **一个长度 32 的索引队列**(排的是**指针 / 索引,不是信息本身** —— 与总纲
  "队列里排指针、信息只有在/移除两态"一致)。所谓"命中三次才进"之类的**计数阈值是错的**,
  它就是 LRU:
  - **最近命中的进队头**;新命中把旧的往队尾挤;**重新命中刷回队头**。
- **双队列(二次机会)**:
  - 从**队列①尾部**被挤出去的,不直接丢,而是**进入队列②**。
  - 队列②里的条目若**再次被命中**,就在**①、②中同时刷新到队头**(晋升回热区)。
  - 效果:**跨度很长、但被反复命中的条目能很好地保留下来**,不会被一阵近期一次性命中冲掉。

## 4. 效用原理(为什么有用)

- **同时利用"最近性"和"频率"两个信号**:单级 LRU 只认"近";多级队列用第二队列充当
  "**缓刑区 / 二次机会**",把"曾经热过、现在偶尔还热"的条目从一次性流量里**救回来**。
- **抗冲刷(scan/burst resistance)**:突发的大批一次性条目只会冲掉队列①;真正反复用的留在队列②,
  一旦再被命中即"满血复活"晋升回①。一次性流量因此**进不了、也留不住**热区。
- **本质**:用分层结构在 **recency 与 frequency 之间折中** —— 思路与经典的
  **2Q / SLRU(分段 LRU)/ LRU-K / MQ** 一脉相承(冷队列保护高频项,免遭纯 recency 淘汰)。

## 5. 关键参数 / 语义(将来落地用)

- 队列① 长度 = 32(用户给定);队列② 长度待定。
- 刷新规则:①命中 → 移到①头;②命中 → **①、②同时移到头(晋升)**。
- 溢出规则:①尾溢出 → 进②尾;②尾溢出 → **真正淘汰**(移出热索引)。
- 排队对象 = **指针 / 索引**,不是数据(数据在"在/移除"两态,原位不动)。
