# 技能:optimize-by-measuring
触发:觉得某处慢/想做性能优化时

性能靠测量,不靠猜。先证明哪里慢、慢多少,再动手——盲目优化常优化错地方还引入 bug。

1. **先确认真有问题**:有没有可观测的慢(实测耗时/profile/用户感知)?没有就别优化(过早优化是浪费 + 复杂度债)。
2. **测出热点再下手**:用计时/profile/日志定位**真正**占时间的那一段(往往不是你以为的那段)。说得出"X 占了 N% / 耗了 M 秒"再优化它。
3. **正确性优先于速度**:任何优化都不能换来错误结果或更难维护的代码;优化后跑同一组测试必须仍绿(行为不变)。
4. **找数量级,不抠常数**:先看算法/架构层面的数量级问题(O(n²)、重复 IO、缓存全失效),那才是大头;微优化(省几个分配)放最后,通常不值。
5. **本项目最具体的一类**:KV 缓存命中——任何破坏 byte-stable 前缀的改动(动 tools 数组、把变内容塞请求前部)= 整条 prompt 重算(实测 hit 640→128);优化上下文/缓存时,先想"这改动会不会破前缀"。
6. **测量优化前后对照**:给出"优化前 X → 优化后 Y"的实测数(像终止 296s→151ms 那样),别只说"应该更快了"(见 [[verify-by-running]])。

要点:优化的纪律是"测量→定位热点→改→再测对照"。没有数字支撑的优化,既可能没用、又可能引入 bug。
