# 技能:name-and-shape-apis
触发:设计一个函数/类型/模块的接口、或给东西命名时

接口和命名是给未来读者(包括未来的自己)的契约,值得当场想清楚。

1. **名字揭示意图,不揭示实现**:`retrieve_skills` 而非 `do_query2`;布尔用肯定式(`is_active` 而非 `not_disabled`);避免缩写黑话。读名字就知道它干什么、返回什么。
2. **跟库里的惯例走**:命名风格、错误返回方式、模块组织都对齐现有代码(GrowBox:Rust snake_case、`Result` 传播、执行器统一 trait;前端 SolidJS signal)。一致性 > 个人偏好。
3. **接口面最小**:只暴露调用方真需要的;能私有就私有。少一个 pub 就少一处未来要维护的契约。
4. **错误在边界处理、类型表达约束**:用类型把非法状态变得不可表示(`Option`/枚举/newtype)胜过运行时校验;校验只放系统边界(见 [[disciplled-change 的边界校验]])。
5. **一个东西一个职责**:函数做一件事、名字就能完整概括;要用"и/and"才能描述 = 该拆。
6. **对称与可预测**:成对操作命名对称(open/close、push/pop、set/get);同类东西签名形状一致,让调用方一通百通。

要点:好接口让调用方"猜得到怎么用、不会用错";名字是最廉价的文档,当场起对,省下无数次解释。
