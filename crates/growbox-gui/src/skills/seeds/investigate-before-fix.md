# 技能:investigate-before-fix
触发:遇到真机/运行期的 bug 或异常,要先定位根因而不是急着试改时

"全面勘探"胜过"单点试错"(本项目血泪:curl 博客响应头一次抓到 CSP 根因,而试 ATS/cors 全没中)。

1. **复现 + 看真证据**:别信任何人(含自己)的转述,去看一手证据——日志、响应头、redb 里的真节点、
   报错全文、退出码。用 shell / file_read 把现场原样捞出来。
2. **列候选根因,逐个证伪**:写下 2~4 个可能原因,各找一条能**排除**它的观察,而不是只找支持
   某个猜想的证据(防确认偏误)。
3. **定位到具体那一行/那个值**再动手:说得出"是 X 文件 Y 处的 Z 值导致",才算找到根因;
   说不出就继续勘探,别开始改。
4. **改完验真**:改完要么跑出能复现→不复现的对照,要么补一条专门测它的用例。
   "应该好了"不算修好。

要点:急着改 = 拿试错当勘探,常常治错了病(本项目反复踩"虚假成功"的坑)。先看清,再下手。
