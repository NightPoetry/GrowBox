# 技能:commit-in-logical-units
触发:把改动提交进版本控制时

提交是给未来 review/回退/排错用的历史。一个提交一件逻辑事,历史才有用。

1. **一个提交 = 一个逻辑改动**:一个 bug 修复、一个功能、一次重构,各自成提交;别把五件无关的事塞一个提交(将来要回退其一就连累其余)。
2. **重构与改行为分开提交**:纯搬迁/改结构(行为不变)单独一个提交,改逻辑另一个——这样 review 一眼看清哪是搬、哪是改(见 [[safe-refactor]])。
3. **提交信息说清"为什么"**:标题一句话说改了什么;正文说为什么这么改、踩了什么坑/根因(what 看 diff 就知道,why 才是历史的价值)。
4. **每个提交都应可编译/可跑**:不提交跑不通的中间态——这样历史里任一点都能 checkout、二分排错(git bisect)才有效。
5. **大改前打检查点**:动架构/大批改动前先 tag 或备份,出问题能干净回退(本项目铁律:大改前备份)。
6. **只提交该提交的**:别把临时调试代码、本地配置、生成产物混进去;敏感信息绝不入库。

要点:好历史让"哪次改动引入的、怎么回退、当时为什么这么做"都查得到。一提交一逻辑事 + why 写清,是这一切的前提。
