# 技能:review-your-diff
触发:改完代码、收尾前自查这次 diff 有没有引入 bug 时

收尾前把自己的改动当别人的 PR 审一遍,按已知高频 bug 类别逐 hunk 扫:

1. **逐 hunk 读 + 读整个被改函数**:被改函数里**没改的行**也在范围内(这次改动重新暴露/没修到它们)。
2. **对每处问:什么输入/状态/时序/平台会让这行错?** 重点查这些高频类别:
   - 条件写反 / 边界差一(off-by-one);
   - 空解引用(None/null/undefined 没判);
   - **漏 await / 漏 `?`**(异步没等、Result 没传播);
   - **falsy-zero 陷阱**:`x || 默认` 当 x 合法地为 0/""/false 时被吞掉(用"是否为 None"判而非真值判);
   - 复制粘贴错变量(改了一处忘改对应的另一处);
   - **catch / Err 把错误吞掉**(静默失败——本项目反复踩,失败必须可感知);
   - 正则元字符没转义、字符串拼接没转义。
3. **核对涟漪面**:这处改动牵连的地方都同步了吗?(改设置碰 Settings→命令→tauri-api→state→面板→i18n 四国;
   加工具碰注册→i18n→懒加载名单——用 code_search 找全调用方)。
4. **跑最相关的测试 / lsp 看诊断**,别只靠眼睛。

要点:大多数回归是这十来类里的某一类。带着这张清单扫一遍,比"看着没问题"可靠得多。
