# 技能:disciplined-change
触发:要落地一处代码改动时(决定改多大范围、怎么不引入新问题)

改动越克制,成功率越高。范围扩张和过度工程是 bug 的主要来源。守这几条:

1. **改现有文件优先于新建**:能在原处改就别另起文件;新建文件要有真正的理由(新模块/新职责)。
2. **只做任务要求的改动**:不顺手重构、不加用不到的抽象、不加"以后可能用"的功能。
   **三行相似代码好过过早抽象**;确定无用就彻底删,别留 `// 旧的`/不 re-export/不改名成 `_var`(兼容 hack 会腐烂)。
3. **不加多余的错误处理**:只在**系统边界**校验(用户输入 / 外部 API / LLM 返回 / 文件 IO);
   内部代码与框架保证信任它,别给"不可能发生"的场景套防御分支(Rust 里尤其别滥用 unwrap 之外的防御)。
4. **像周围代码那样写**:命名、注释密度、惯用法都跟着该文件的风格走,别带进自己的风格。
   **注释只写 why(不显然时)、不写 what,不写任务上下文**(`// 为 X 流程加的`/`// 见 issue#123` 属 PR 描述、会随代码烂掉)。
5. **小步 + 不留半成品**:一次一个意图;不提交跑不通/编不过的中间态。
6. **大改/不可逆前先对设计**:动架构、删数据、改公共契约前,先给用户 2~3 句方案 + 取舍、等同意再动
   (本项目铁律:别屎山、大改前对设计)。

要点:最好的改动是"看不出多余动作"的改动——只改该改的,读起来像本来就在那。
