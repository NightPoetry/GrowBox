# 技能:orient-in-unfamiliar-code
触发:进入一个不熟悉的代码库/模块,动手前要先建立全局认识时

改不熟的代码,先用最少的阅读建立"地图",别一头扎进细节。

1. **找入口与脊柱**:先定位程序入口(main / lib 根 / 路由表 / 命令注册处)和"主流程"——大多数系统有一条核心循环或一条主调用链,顺它走一遍就懂了七成。
2. **由宽到窄**:先看目录结构和模块边界(谁依赖谁),再钻到要动的那块;别从一个细节函数倒推全局。
3. **读测试与类型当文档**:测试用例说明"这东西该怎么用、边界在哪";类型签名/接口说明契约。比读实现更快抓到意图。
4. **搜用法而非只读定义**:用 code_search 找某函数/结构的**调用方**,看真实怎么用——用法比定义更能说明设计意图和约束。
5. **找同类先例**:要加的东西,库里多半有类似的——找到它,照着它的模式写(抄惯例比自创风格更安全、更易被接受)。
6. **建立不变式清单**:边读边记这个模块维护什么约束(顺序/缓存前缀/单一所有权/必经某门),改动时不能破坏(见 [[read-before-write]] [[disciplined-change]])。

要点:先花十分钟画地图(入口→主链→边界→不变式),比直接改省下几小时的返工。理解不能外包,也不能跳过。
