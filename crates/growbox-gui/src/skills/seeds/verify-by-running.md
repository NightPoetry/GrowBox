# 技能:verify-by-running
触发:改完一件事、要向用户汇报"做好了"之前

自我负责:验证 = **证明它真能跑**,不是确认代码存在、不是"应该没问题"。不信自己的自述,去核真。

1. **跑真的那条路,不只跑测试**:能跑就跑——`cargo test` 跑最相关的、UI 改动用全自动调试模式真驱动(见 [[self-test-with-auto-debug]])、命令改动真执行一次、网页改动真刷新看渲染。
2. **看真产物,不看意图**:改了文件就 file_read 复看真落了字;说"写到了 X"就去确认 X 在;说"修了 bug"就拿出复现→不复现的对照。AI 的自述描述的是"我想做什么",不一定是"真做成了什么"。
3. **核对你即将说的每句话**:收尾汇报前,把"我做了 A、B、C"逐条对着证据核——证据不支持的改口或标注"未验证",别把推测说成事实(本项目血泪:核 redb 戳破过"嵌入在跑/历史隔离"的假象,催生了 self_verify 机制)。
4. **失败照实说**:测试红了就说红、贴输出;跳过了某步就说跳过;没法验证的明说"无法验证"。报喜不报忧会把假成功喂进飞轮、越学越歪。
5. **本项目的核真手段**:cargo test/clippy、全自动调试模式(注入操作+内部截图+事件埋点)、读 redb 真节点、curl 看真响应头/退出码。

要点:"应该好了"不算好了。证明它跑通,再说做好了——这是自我负责,也是不让虚假成功污染记忆的前提。
