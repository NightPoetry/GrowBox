# 技能:write-tests-that-matter
触发:为一处改动写测试、或判断现有测试够不够时

测试的价值不在数量,在"能不能在出错时失败"。写能抓真问题的测试:

1. **测行为,不测实现**:断言"输入 X 得到结果 Y"(可观察的契约),别断言内部调了几次某函数——后者一重构就假红。
2. **必覆盖失败/边界路径**:happy path 谁都会写;真正抓 bug 的是空输入、0/""/边界值、越界、超时、取消、并发、错误分支。本项目高频回归就藏在这些(见 [[review-your-diff 的 bug 类别]])。
3. **一个测试一个关注点**:名字说清测什么;失败时一眼知道哪坏了。一个测试塞五件事 = 红了也不知道哪错。
4. **能失败才有意义**:写完先想"什么改动会让它红?"——如果想不出,这测试是噪音(永远绿=没测到东西)。可临时把被测逻辑改坏一下,确认它真会红,再改回来。
5. **本项目语境**:Rust 用 `cargo test -p <crate>`;真机/API 用例标 `#[ignore]` 单独跑;UI 行为用全自动调试模式驱动(见 [[self-test-with-auto-debug]])。i18n/防散装这类"完整性守卫"测试要随新增条目一起更新(否则它会红)。
6. **修 bug 必配一条复现测试**:先写一条能复现该 bug 的测试(此刻它红),再改代码让它绿——这条测试从此守住这个回归。

要点:可靠的测试是"改坏代码它就红"的测试。写完反问自己这一句,胜过堆十个永远绿的。
