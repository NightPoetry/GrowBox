# 技能:async-cancellation-discipline
触发:写异步/长任务/流式循环,要保证能被「终止」+ 不挂死时

长任务和流式循环最容易出两类真 bug:**点了终止不停**、**对方不返回就永久挂死**。本项目都踩过(终止失效 296s→151ms;shell 后台进程攥住管道 445s 不返回)。守这几条:

1. **永远不要无逃生地阻塞等待**:`recv().await` / `output()` / 读管道这类"等对方"的调用,必须配**超时**或**取消**逃生口,别裸 await。否则对方一卡,你就永久挂。
2. **取消要能在等待中途生效,不只在循环顶**:若取消标志只在 `loop` 开头查、而循环体卡在一个长 `recv().await` 里,点终止要等这次 await 结束才生效。修法=把长等待切成 ≤1s 的 poll 分片,每片重查取消(累计静默到阈值才真超时)——这正是 296s→151ms 的修法(`drive.rs`)。
3. **取消标志要穿到执行器深处**:不止主循环,长命令执行器(shell 等)也要据 `ctx.is_cancelled()` 中途收手(core `ExecCtx.cancel` + `dispatch_with_cancel`)。
4. **管理子进程的生命周期**:spawn 的子进程/后台服务要独立进程组 + 完成/超时/取消三者 select + 任一出口 killpg 杀整组(否则 `cmd &` 的子进程攥住 stdout 管道,捕获方等 EOF 等不到)。
5. **失败/超时必可感知**:挂死或超时不能静默——emit/perceive 告知,让 AI 与用户都知道(失败必可感知铁律)。

要点:异步代码的正确性 = 任何"等别人"的点都要能被超时或取消打断,且打断在 ≤1s 内可见。裸 await 是挂死的种子。
