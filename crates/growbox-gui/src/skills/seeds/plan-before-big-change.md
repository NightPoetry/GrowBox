# 技能:plan-before-big-change
触发:要做一个多步/跨文件的功能或较大改动,动手前要先定方案时

大改动先规划再下手,边界/方向问题在写代码前暴露最便宜。

1. **拆成可独立验证的小步**:把目标分解成一串"每步都能编译/跑/验"的增量,排出顺序(被依赖的先做)。一串小绿步胜过一个大红堆。
2. **定位关键文件**:先列出这次要碰的 3~5 个核心文件 + 各自改什么(用 code_search/orient 摸清),让范围具体、可估。
3. **先解决最大不确定性**:哪一步最可能推翻整个方案(技术可行性/接口形状),先做它的最小验证(spike),别把它留到最后才发现走不通。
4. **开放性决策先给方案、用户同意前不实现**:遇到方向/取舍岔路(用哪种错误处理、要不要拆抽象),给 2~3 句推荐 + 主要取舍,呈现为"可改向"而非已定;用户拍了再写——避免做错方向再返工。本项目铁律:大改/不可逆前先对设计。
5. **留出验证步**:方案末尾明确"怎么证明做成了"(跑哪些测试 / 真机走哪条路,见 [[verify-by-running]]),别做完才想怎么验。

要点:规划不是仪式,是把"返工风险"从写代码后移到写代码前。最贵的 bug 是方向错了还写了一大堆。
