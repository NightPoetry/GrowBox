# 技能:artifact-ui-craft
触发:用 render_artifact 现造一个 UI(棋盘/表单/图表/小工具)给用户交互时

造物是你的自我延伸(同一颗大脑 + 记忆,不外接别的 AI)。造得好不好,看这几条(都来自本项目五子棋真机血泪):

1. **可视化优先**:能用界面展示/交互的就别用纯文字。"陪我玩五子棋"→直接渲棋盘,别文字描述棋局。
2. **★单次渲染 + 之后用指令驱动,别每次交互都重画★**:第一次 render_artifact 渲出完整 HTML,**在造物里注册 `window.gxOnCommand(cmd)`**;之后用户交互/你要更新画面,走 `artifact_command` 发指令让造物自己改 DOM——**不要反复 render_artifact 整段重画**(五子棋失控就是反复重画把 CPU 干到 345%)。
3. **回调上报**:普通元素挂 `data-gx-callback="名"`;canvas/按坐标算的(如落子)在 JS 里调 `window.gx.emit("名", 值)` 上报给你(别手写易错的 postMessage 协议)。realtime 高频输入用 `gx.emitRealtime`。
4. **做好了才展示**:中间多次渲染只更新草稿,回合结束由 present 展示最终态——别"一边改一边弹"打断用户。
5. **覆盖层说话**:要主动给用户提示/吐槽,用 push_artifact_notice 推到造物覆盖层(GrowBox 可信层,浮在沙箱内容之上),不是往不可信 DOM 注入。
6. **守沙箱边界**:造物是 `sandbox=allow-scripts` 的 null-origin iframe——脚本能跑,够不到本机/网络/父页;回调只把数据传回给你,**不能直接触发文件/shell**(那些仍走安全门)。
7. **自包含**:HTML 里把样式/脚本都带齐(沙箱里加载不了外链);别依赖外部 CDN/资源。

要点:造物的灵魂是"渲一次、之后发指令操控",不是"每帧重画"。重画失控是这里最大的坑。
