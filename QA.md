## 一、整体定位

**Q:一句话,GrowBox 是什么?**
A:一个"越用越强"的持久学习型 AI 编程 Agent。它把 IDE 当作 AI 的手脚,把**记忆**当作 AI 的本体——不是每次对话从零开始的助手,而是把每一次检索、每一次判断的副产品沉淀成结构、下一次直接复用。仓库的真正核心不是又一个 Agent 循环,而是底层的 **AI 记忆机制**。

**Q:它和"又一个套壳 Agent"有什么本质不同?**
A:多数 Agent 把记忆做成"塞更多东西进上下文"或"挂一个 RAG"。GrowBox 把问题重述为操作系统的**虚拟内存**问题:记忆无界(磁盘),上下文有限(物理内存),于是需要一套**置换系统**——把"现在该看的"换进上下文、"暂时不看的"换出冷归档,用索引/指针网做地址映射。**置换是本质,缓存是加速。**

**Q:为什么用操作系统来类比?**
A:因为矛盾同构:无界虚拟地址空间 vs 有限物理内存 = 无界记忆 vs 有限上下文窗口。对应关系:全部记忆=虚拟地址空间(redb 磁盘);LLM 上下文窗口=物理内存 RAM;指针网=页表/地址翻译;检索 miss→沿指针召回=缺页→从 swap 调页;冷归档=swap。这套类比不是修辞,是直接指导了架构(见 `记忆置换系统-总纲.md`)。

## 二、缓存机制:上下文里放什么、怎么放才省钱 【已实现】

**Q:持久 Agent 用久了,上下文会不会无限膨胀?**
A:不会。上下文不是一个持续增长的字符串,而是一个**有预算的工作集队列**(`context.rs::ContextWindow`)。每条记忆只有"在/移除"两态,超预算就按热度淘汰最冷的(`enforce_budget`)。所以上下文大小恒定可控,真相在磁盘、工作集在内存。

**Q:你强调"排队的是指针,内容原位不动",为什么?**
A:为了命中 LLM 的 **prompt 缓存**——它只对"逐字不变的稳定前缀"生效。如果每次某条记忆被访问就把它在上下文里挪到队首,整个前缀字节就变了,缓存全失效。所以 GrowBox 把"排队"(指针,在上下文之外,随热度重排)和"数据"(在上下文之内,append-only、原位不动)彻底分开。命中率高时,这一整段都能被 prompt 缓存复用。

**Q:那最近发生的事怎么保证一定在场?**
A:上下文最末挂一段固定容量的**最近记忆 ring**(默认 64K 字符),每轮都覆盖最旧。它有意放在最末:每轮都打破缓存,但只损它自己这一小段,不波及前面的稳定前缀。

## 三、索引机制:怎么快速找到"该换入的那块" 【已实现】

**Q:RAG 和"LLM 精确检索"是上下两层吗?**
A:**不是层级,是两种平级的索引手段**,区别只在"怎么找到那条指针":RAG=向量近似(直接跳);精确层=LLM 读原文 + 指针图导航。找到之后殊途同归——都把内容调进同一个工作集。成本上便宜的 RAG 先试、贵的 LLM 精确按需补,但这是**成本顺序,不是层级**。

**Q:第一层 RAG 怎么做?命中阈多少?**
A:向量相似度召回(磁盘原生 ANN,arroy),首条余弦 ≥ `RAG_HIT_THRESHOLD=0.85`(按 e5 实测分布定)即"找到了",直接返回不下沉。够用且反馈好就停。

**Q:"越用越快"具体怎么实现?**
A:核心是把软件工程的**记忆化搜索(memoization)**迁移到检索上:算过的结果记下来,下次同样的子问题直接查表。落在记忆上,这张"表"就是**学习型指针**(`pointer.rs` + EDGES 表)。每次精确层线性扫一旦扫到相关,就在当前位置沉淀一条指针;下次走过的路直接抄近道。这就是"扫描越用越快"。

**Q:指针的 K 是什么?Q 是什么?(这是最关键的点)**
A:一条指针的键(**K**)= **过去某次成功找到这段记忆时所用的查询**;新的一次查找(**Q**)= 当前查询。"一个曾经成功命中目标的提问"比目标原文本身更能预测"下次类似提问要什么"。所以本质是 **Q↔K**:K 是过去的 Q,Q 是现在的 Q,一旦当前 Q 被判定和某个过去的 Q 相似,就沿指针直接复用那次的搜索结果,省掉一整轮线性扫。

**Q:Q 和 K 的匹配,是用 embedding 相似度,还是用 LLM 判断?**
A:两档可选(`PointerMatchMode`,见 `config.rs`):
- **WeightedCosine**(默认,廉价):当前 Q 向量与指针上各 K 向量做加权余弦,过门即走快车道。
- **LlmJudge**(精确,贵):把这条边的正/负 K 原文 + 目标内容一起喂 LLM,**生成式判断**该不该跳(`sub.judge_edge` 返回 bool)。
LLM 这档是"有关/无关"的二分判断,不是阈值调参;而且因为是生成式,**检索任务可以携带关键上下文、做得很复杂**——它是"拿着 Q 去真正查找 K",不是单纯算两句话的相似度。

**Q:如果误跳到一个不相关的记忆怎么办?(失败跳转)**
A:系统**不仅记成功跳转,也记失败跳转**。judge 拒绝时记一条**反 K**(`record_negative_edge`);以后新 Q 若和这条反 K 高度相似(余弦 ≥ `neg_block_threshold=0.90`),直接**一票否决**这条边、不再跳(`neg_veto`)。这正像 CPU 分支预测——既记"预测命中"也记"预测失败"来优化未来决策。成功的则记**正 K**(`record_positive_edge`,累积去重 + 升热度),越用越准。

**Q:同一个问题在不同上下文下答案不同,怎么不串味?**
A:这考验"查找提示词"。因为是生成式检索(LlmJudge),judge_edge 看的不只是 Q 和 K 像不像,而是带着这条边的**正负 K 历史 + 目标内容**综合判断,可以纳入关键上下文。工程上还有项目软隔离 tag(命中本项目加权)、强制跳转(用户显式钉的历史位置必跳)等配套。

## 四、染色 / 碎片 / 做梦:扫描副产品 + 后台维护 【已实现】

**Q:染色是什么?几种颜色?**
A:染色标记"一段区域被扫描的程度",好让"翻到最早=全量扫描"时跳过已确认区、不重复读。代码里实际是**三级**(`Stain`):`None`(没到过)/ `Light`(跟指针快扫过、可能漏)/ `Deep`(仔细扫过、确信无漏)。本质上对线性扫的意义是二元的:**`Deep` 可安全跳过,`None`/`Light` 还需扫**。所以"区分扫够了的区 vs 还要扫的区"这个直觉是对的,只是代码里多留了 Light 这个"快扫过但没确认"的中间态。

**Q:碎片是什么?它和染色什么关系?**
A:走指针/二级锚点是"一跳直达",中间那段被整个跳过、**没经 LLM 判断**——可那段里可能还藏着与某个 K 相关的内容。这块"明知可能有货、本次跳过、未经判断"的区间就是**碎片(碎片=债)**,记进一本有界台账(`fragments.rs`)。染色是"标记扫过没",碎片是"标记哪段欠扫"——一体两面。

**Q:碎片整理什么时候做?会阻塞主流程吗?**
A:**在闲暇时做,完全非阻塞,不做也不影响正确性**(只影响速度)。它是"做梦"(`maintenance.rs::dream_once`):空闲时取一笔碎片债→读那段被跳过的中段→判断有无遗漏→相关的补一条指针→销账。触发靠 idle 飞轮(`gui/idle.rs`),取最低优先级、可被前台随时打断。

**Q:做梦/睡眠/疲劳怎么回事?**
A:**做梦**=还一笔碎片债。**睡眠**=做梦+推演交替直到债清/被唤醒。**推演**=趁网干净时自问自答"用户若问 X 该检索到什么",提前建好索引。**疲劳度**不看 CPU/内存,看三项加权(缓存命中率低 + 淘汰频繁 + 碎片占比大),疲劳高就该睡。一个**潜意识仲裁器**按"前台 Agent > 睡眠 > 飞轮"调度,保证后台维护不和前台抢资源。

**Q:缓存/索引会不会爆炸?**
A:**上下文不会**——它是有预算的队列,旧的按热度淘汰出去(见第二节)。真正的风险是**索引数量暴涨**(沉淀的 K 太多)。这不是核心算法问题、属工程细节,**目前**的做法是淘汰低价值索引(长期没被命中、也没被负反馈标记过的),最优策略还在探索。

## 五、RAG 的"假指针":为什么 RAG 命中要特殊对待 【已实现】

**Q:RAG 命中产出的指针和精确层的指针一样吗?**
A:不一样。染色/碎片有个隐含前提:**沉淀的指针必须是"扫过去"的产物**。精确层天然满足(从近往远顺序读,路径无 gap)。但 GrowBox 的 RAG 是**非线性 ANN**(直接跳到目标,没有"路径"和 gap 的概念),所以它命中产出的指针只当**假指针(RagFake)**:纯粹为了让 RAG 命中也能统一进工作集、参与置换,但**换出时不留二级锚、不进碎片系统**(没扫描 gap,没债可还)。真/假指针在 `context.rs::Origin` 里显式区分。

## 六、文档破碎化:长文投喂为什么会"检索不到" 【已实现,2026-06-17】

**Q:用户把整篇技术文档粘进来,后面问里头某个细节却答不上,为什么?**
A:整篇长文在时间线上是**一个大节点 → 一条向量**。多主题长文被平均、超嵌入窗的尾部被截断,单条向量代表不了任一窄主题 → 窄问 RAG 必漏;它一旦被染 Deep,精确层也跳过 → 两网皆漏,AI 只能凭空答(实测把 `--db-accent` 答成 `--primary`、`better-sqlite3` 答成 `sql.js`)。

**Q:怎么解决?**
A:**消灭大节点**——入场按尺寸阈标记,idle 时按句把它破成语义小块,各块独立成节点、独立向量(`memory-记忆系统.md` + 专页 `文档破碎化-长文检索盲区.md`)。关键安全设计:**LLM 只判"在哪几句之间破",不碰文本**;Rust 按原子句精确拼接,零改写零丢字。真机 before/after:同篇文档 10 个窄问,破碎前 0/10 聚焦命中(2 个完全检索不到),破碎后 10/10 由 RAG 第一层命中聚焦块。

## 七、涌现式分层:为什么不让 LLM 主动分类 【已实现 + 设计哲学】

**Q:组织无界记忆,常见做法是让 LLM 读文档、提炼标题、建分类树。你为什么不这么做?**
A:**LLM 主动分层是主观的,基于用户查找涌现的分层是客观的。** LLM 按"它以为的"主题归类,和使用者实际怎么找常常错位,一旦归错就固化进结构、之后每次查找被带偏。GrowBox 的分层不是预先规定的,而是从真实查找里长出来的:每条指针的 K 是一次真实成功的查询,每条边是一次真实发生的联想。于是涌现的分层+图结构直接编码使用者的客观分类习惯,高频路径被反复加固成最短捷径,**最大化加速真实查找**。(破碎化只定粒度,不强加分类;粒度之上的组织仍由查询涌现。)

## 八、【探索中】用 KV 缓存进一步加速查找(新方向,当前未落地)

> ★诚实声明★:**这一节是新构想,当前代码尚未实现。** 现有的"越用越快"靠的是上面第三节的
> 学习型指针网 + 染色 + 上下文置换;下面这层 KV 缓存复用是在它之上更激进的加速设想。

**Q:你提到 KV 缓存,这是个什么想法?**
A:思路:把上下文片段做成 `[内容, 预置 Q]` 形式的可检索单元,并**连同这段内容在 LLM 里算出的 KV 缓存一起保留**。检索时做 **Q-to-Q 匹配**(当前用户 Q vs 片段的预置 Q),命中就直接复用那个片段的**完整 KV 缓存**——既不像 RAG 那样丢信息(chunk 截断),又省掉对这段内容**重新跑一遍注意力**的计算。

**Q:它和现有的 Q/K 指针有什么区别?**
A:层次不同。现有指针复用的是**"检索结果/导航路径"**(找到该看哪段记忆);KV 缓存进一步想复用**"推理的中间态"**(这段内容在模型里已经算好的 Key/Value,不必重算)。前者省检索,后者再省一截前向计算。两者可叠加:指针先定位到片段,KV 缓存让这个片段"零重算"地进场。

**Q:为什么说它能补 RAG 的不精确?**
A:RAG 是"用户 Q → 内容块"的相似度匹配,块可能截断、可能选错。这套是"用户 Q → 历史 Q"的匹配 + 复用完整 KV,信息零损失。本质上把"每次重新读一遍书"换成"记得以前读过、连当时的理解状态一起调回来"。

**Q:用现在的托管 API(比如 DeepSeek 的 chat 接口)能直接做这套 KV 缓存复用吗?**
A:**不能直接做**,这是它最关键的前置条件。典型托管 chat API **不暴露模型内部的 KV 缓存**——你既拿不到那段已算好的 Key/Value,也无法把它注入到下一次请求里。所以这套加速**对部署形态有前置要求**,只有两条落地路径:
- **① 上游配合**:模型服务方在 API 层支持 KV 缓存的导出/注入(即比"prefix caching 前缀缓存"更进一步、可寻址复用任意片段的能力)。
- **② 可控的本地模型**:自己跑开源/自托管模型,**完全掌控 KV 缓存的读写**,才能把"复用某片段的 KV"真正实现出来。
换句话说,它不是纯应用层能独立交付的优化——要么等上游开口子,要么把模型握在自己手里。这也是它目前仍属"探索中"而非"已实现"的根本原因之一。

**Q:难点/未决?**
A:KV 缓存的存储成本、跨上下文位置的 KV 复用是否数值有效(位置编码/前缀依赖)、预置 Q 的生成与命中率、缓存一致性——都还在概念阶段。**当前版本不依赖它也能跑**,它是未来的加速选项,不是已交付能力。

## 九、边界与诚实(哪些还没做好)

- **索引爆炸的最优淘汰**:当前是"淘汰长期未命中且无负反馈的低价值索引",最优策略仍在探索。
- **文档破碎化**:真机只验过结构清晰、句读分明的中文技术文档;日志/代码/无句末符的大段连写,切点可能不理想(靠尺寸兜底保正确、不保最优)。
- **KV 缓存加速**(第八节):构想,未实现。
- **前端面板 / 打包 / 跨平台**:后端记忆机制已全落地、`cargo test --workspace` 全绿;面板接真数据、出包流程、跨平台仍在推进。
- 一个原则:**一切加速结构都是"这次的副产品",失效了不影响正确性,只影响速度**——最差退化为 LLM 一段段读原文的暴力全量扫描。

## 十、架构 / 技术栈速览

6 个 crate,单向依赖,公理"一切能力皆执行器、Agent 循环是唯一脊柱":

```
app(gui)  Agent 循环脊柱 + 执行器注册表 + Tauri + 前端(SolidJS)
├── memory   分层检索(RAG → 精确层飞轮)+ 上下文置换   <- 本文主角
├── learn    飞轮:收集 → 聚类压缩 + 永久目标调度
├── safety   沙箱 / 路径分级 / 风险 / 三种授权
├── llm      LLM 通信:流式 / reasoning / 工具解析 / 嵌入
└── core     共享类型:结论模型 + 执行器 trait(零依赖)
```

记忆技术栈:**redb**(单文件持久化,节点/边 write-through)、**arroy + LMDB**(磁盘原生向量索引,mmap 不全驻 RAM)、**candle 跑 multilingual-e5-small**(本地嵌入,另留远程 OpenAI 兼容槽)。正文与向量两半都磁盘原生,内存占用=工作集、与总记忆量解耦——这是"无界记忆"成立的前提。
