# GrowBox QA —— 一问一答，把这套 AI 记忆机制讲清楚

这是一份用对话形式写的项目说明。问题是面试式的追问，答案尽量讲人话，并且都对着仓库里的真实代码和 `设计文档/` 说，不是空想。

每个回答会标一下它的状态：**【已实现】** 表示代码里真有、也有测试；**【探索中】** 表示这是个想法或方向，目前还没落地。想往下挖细节，可以看 `设计文档/设计/02-记忆检索.md`(原理)、`设计文档/具体系统设计/memory-记忆系统.md`(实际怎么写的)，以及仓库根的 `README.md`。

## 一、先搞清楚这是个什么东西

**Q：一句话，GrowBox 是什么？**

一个越用越强的 AI 编程助手。它和普通助手最大的区别是：普通助手每开一轮对话都从零开始，而 GrowBox 把每一次检索、每一次判断的结果都沉淀下来，下次能直接复用。所以这个仓库真正值钱的不是又一个 Agent 循环，而是底下那套**记忆机制**——它把"记忆"当成 AI 的本体，把 IDE 当成 AI 的手脚。

**Q：现在到处都是套壳 Agent，你这个凭什么不一样？**

多数 Agent 处理记忆的办法，要么是"往上下文里塞更多东西"，要么是"外挂一个 RAG"。GrowBox 换了个角度看这件事：它发现这其实就是操作系统早就解决过的老问题——**虚拟内存**。

你的记忆是无限增长的(都在磁盘上)，但模型的上下文窗口是有限的(就那么多 token)。这跟"无限大的虚拟地址空间 vs 有限的物理内存"是同一个矛盾。操作系统的答案是分页和置换：把现在要用的那一小块换进内存，暂时不用的换出去，再用一张页表记住"谁在哪"。GrowBox 就照着这套来：磁盘是全部记忆，上下文窗口是物理内存，指针网是页表。所以它的核心是**置换**，缓存只是顺带加速——这个主次关系一直贯穿整个设计。

## 二、上下文里到底放什么、怎么放才省钱 【已实现】

**Q：一个 Agent 用久了，上下文不会越堆越大、最后爆掉吗？**

不会，因为上下文根本不是一个一直在变长的字符串。它是一个**有预算上限的工作集**(代码里叫 `ContextWindow`)。每条记忆在里面只有两种状态：在场，或者被移出去。一旦总量超过预算，就把最冷门的那条挤出去。所以上下文的大小是恒定可控的——真正的全量记忆躺在磁盘上，内存里只留当前这一小撮工作集。

**Q：你特意提到"排队的是指针，内容本身不动"，这有什么讲究？**

这是为了省钱。模型供应商的 prompt 缓存有个前提：只有开头那段内容**逐字不变**，它才能复用上一轮的计算、给你省下延迟和费用。问题来了——如果每次某条记忆被用到，我就把它在上下文里挪到最前面，那开头这段就被打乱了，缓存全作废。

所以 GrowBox 把"谁热谁冷的排序"和"内容本身"彻底分开了：排序是在上下文**之外**对一堆指针做的，爱怎么重排怎么重排；而上下文**里面**的内容是只增不挪的，一条记忆只要还没被挤出去，它的位置就纹丝不动。这样开头那段始终稳定，缓存命中率自然就高。

**Q：那"刚刚发生的事"怎么保证一定在场？**

上下文最末尾挂了一小段固定大小的"最近记忆"窗口(默认 64K 字符)，每轮都把最旧的覆盖掉。它故意放在最末尾：每轮都会打破缓存没错，但只损它自己这一小段，碰不到前面那段稳定的开头。

## 三、怎么快速找到"现在该换进来的"那块记忆 【已实现】

**Q：RAG 和你说的"LLM 精确检索"，是上下两层的关系吗？**

不是上下层，是**两种平起平坐的找东西的办法**，区别只在于"怎么定位到那条记忆"。RAG 是靠向量相似度一步跳过去；精确层是让 LLM 真的去读原文、顺着指针图导航。但找到之后是一样的——都把内容搬进同一个工作集。只不过 RAG 便宜，先试它；不行了再上贵的 LLM。这是个先后顺序，不是高低层级。

**Q：第一层 RAG 具体怎么做？什么时候算"找到了"？**

用向量相似度去召回(底层是磁盘原生的近似最近邻索引 arroy)。最相似那条的余弦值如果到了 0.85(这个阈值是按 e5 模型的实测分布定的)，就认为"找到了"，直接返回、不再往下沉。够用就停。

**Q：你反复说"越用越快"，到底是怎么变快的？**

灵感来自软件工程里一个很朴素的东西：**记忆化搜索**。算过一遍的结果记下来，下次碰到同样的子问题直接查表，不重算。

搬到记忆上，这张"表"就是**学习型指针**。精确层每次老老实实线性扫，一旦扫到一条相关的记忆，就在当前位置留下一条指针，记住"从这儿能到那儿"。下次再走到这附近，直接抄这条近道，不用重新扫一遍。走的路越多，近道越多，于是越用越快。

**Q：这指针上记的 K 是什么？Q 又是什么？(这块我最想听明白)**

关键就在这。一条指针上的键(K)，存的是**过去某次成功找到这段记忆时用的那个问题**。为什么存问题而不是存内容？因为"一个曾经成功命中目标的提问"，比目标原文本身更能预测"下次类似的提问想要什么"。

所以本质是 **Q 和 K 都是问题**：K 是过去的问题，Q 是你现在的问题。当系统判断你当前这个 Q 和历史上某个 K 很像，就顺着那条指针，把上次的搜索结果直接拿来用，省掉一整轮线性扫。

**Q：判断 Q 和 K 像不像，是用向量算相似度，还是让 LLM 来判？**

两种都行，是个可切换的档(代码里叫 `PointerMatchMode`)：

- 便宜那档(默认)：拿当前问题的向量和指针上的 K 向量算个加权余弦，过线就走。
- 精确那档：把这条指针的正反例和目标内容一股脑喂给 LLM，让它**生成式地判断**该不该跳，只回"该跳/不该跳"。

精确这档有意思的地方在于：它不是在算两句话像不像，而是"拿着你的问题去真正地查找"。因为是生成式的，你可以把检索任务做得很复杂、带上关键上下文，远不止相似度匹配那么简单。

**Q：万一跳错了、找到一条不相关的记忆，怎么办？**

系统不光记成功的跳转，也记失败的。某次判断"这俩不相关"，就在这条指针上记一个**反例**。以后再来一个问题，如果它和这个反例高度相似(余弦超过 0.90)，系统直接一票否决、不走这条路了。这套思路跟 CPU 的分支预测一模一样——既记"猜对了"，也记"猜错了"，拿失败的经验去优化以后的决策。成功的那边则累积正例、加热度，所以是越用越准。

**Q：同一个问题，在不同上下文里答案不一样，会不会串味？**

这就考验"查找时的提示词"功夫了。因为精确档是生成式检索，它看的不只是问题和键像不像，而是连着这条指针的正反例历史、加上目标内容一起综合判断，可以把关键上下文带进去。工程上还有几道配套：按项目打软标签(命中本项目的加权)、用户显式钉过的历史位置强制必跳，等等。

## 四、染色、碎片、做梦——扫描的副产品和后台维护 【已实现】

**Q：染色是干什么的？有几种颜色？**

染色是给"这段区域被扫得多仔细"做个标记，这样下次再"从头翻到最早"做全量扫描时，能跳过已经确认过的地方，不重复读。

代码里实际上是**三档**：没到过、跟着指针快扫过(可能有漏)、仔细扫过(确信没漏)。对线性扫来说，意义其实是二元的——"仔细扫过"的可以放心跳过，另外两种还得再扫。所以你把它理解成"区分扫够了的区和还要扫的区"是对的，只是代码里多留了个"快扫过但没完全确认"的中间档。

**Q：碎片又是什么？它和染色什么关系？**

走指针或者二级锚点是"一跳直达"，中间那段被整个跨过去了，**没经过 LLM 判断**。可那段里很可能还藏着和某个键相关的东西，只是这次没去读它。这块"明知道可能有货、但本次跳过了、还没判断过"的区间，就是**碎片**——本质是一笔债。系统把它记在一本有上限的台账里。

说白了，染色管的是"扫过没扫过"，碎片管的是"哪段还欠着扫"，是一体两面。

**Q：碎片整理什么时候做？会卡住正常流程吗？**

不会卡，它是闲下来才做的，而且**不做也不影响正确性**，只影响速度。具体就是"做梦"：空闲时取一笔旧债，把那段被跳过的中间读一遍，看有没有漏，有相关的就补一条指针、然后销账。触发它的是后台的 idle 飞轮，优先级压得很低，前台一有动静立刻让路。

**Q：做梦、睡眠、疲劳这几个词是怎么回事？**

做梦就是还一笔碎片债。睡眠是做梦和"推演"轮着来，直到债还清、或者被叫醒。推演是趁记忆网干净的时候自问自答"用户要是问 X，我该找到什么"，提前把索引建好。疲劳度不看 CPU 和内存，而是看三个信号加权——缓存命中率低不低、淘汰频不频繁、碎片占比大不大——累了就该睡。这几件事和前台检索共用同一个"潜意识"模型，由一个仲裁器按"前台优先、其次睡眠、最后才是后台飞轮"来调度，保证后台维护不跟前台抢资源。

**Q：缓存或者索引会不会无限膨胀？**

上下文不会，前面说过，它是有预算的队列，旧的按热度淘汰掉。真正的隐患是**索引数量暴涨**——你沉淀的"键"太多了。这个我得诚实说：它不算核心算法问题，更像工程细节，目前的做法就是淘汰那些长期没被命中、也没被打过反例标记的低价值索引，最优方案还在摸索。

## 五、为什么 RAG 命中的指针要特殊对待 【已实现】

前面那套染色、碎片机制，有个藏着的前提：它沉淀的指针必须是"一路扫过来"的产物。精确层天然满足——它从近到远顺着读，路上没有跳过的窟窿。

但 RAG 不一样，它是靠向量**直接跳**到目标的，中间那段路根本没走过，没有"路径"也没有"窟窿"这个概念。所以 RAG 命中产出的指针，GrowBox 只把它当**"假指针"**：它存在的唯一目的，是让 RAG 命中的东西也能统一进工作集、参与置换；但它被换出去的时候，不留二级锚点、也不进碎片系统——因为它压根没有扫描留下的债要还。代码里用一个 `Origin` 标签把真假指针明确分开。

## 六、投喂整篇长文档，为什么反而搜不到 【已实现，2026-06-17】

**Q：我把一整篇技术文档粘进来，后面问里头某个具体细节，它却答不上，怎么回事？**

问题出在：整篇长文进来后，在时间线上是**一个大节点，对应一条向量**。一篇覆盖好几个主题的长文，被压成单独一条向量的时候，信息被摊平了，超出嵌入窗口的尾巴还会被直接截断。结果就是，你问里头某个很窄的细节，这条被稀释过的向量根本匹配不上；而它一旦被染成"扫过"，精确层也会跳过它。两条路都断了，AI 只能瞎答——实测里它把主题色 token 答成了别的、把数据库答成了另一个库，全是凭空编的。

**Q：那怎么解决？**

办法是**干脆别让大节点存在**。文档一进来，发现它超过尺寸阈值就先标记一下；等系统空闲时，让 LLM 看着它、按句子把它切成几个语义连贯的小块，每个小块独立成节点、各自有自己的向量。这里有个关键的安全设计：**LLM 只负责判断"在哪几句之间切开"，绝不碰文本本身**；真正的拼接是 Rust 按原句精确做的，所以一个字都不会改、不会丢。

效果用真机数据说话：同一篇文档、10 个很窄的问题，切之前 0 个能命中到位(有 2 个甚至完全搜不到)，切之后 10 个全部由第一层 RAG 命中到对应的小块。

## 七、为什么不让 LLM 直接给记忆分类、建目录 【设计哲学 + 已实现】

**Q：组织海量记忆，常见做法是让 LLM 读一遍、提炼标题、建一棵分类树。你为什么偏不这么干？**

因为这里有个根本分歧：**LLM 主动分类是主观的，而从用户查找里长出来的分类是客观的。**

LLM 给你分类，是按"它以为的"主题来分。可它以为的，和你实际怎么找、怎么联想，常常对不上。一旦它分错了，这个错就固化进结构里，以后你每次查找都被它带偏，而且它根本不知道你真正在意的是哪个维度。

GrowBox 的做法反过来：分类不是事先规定的，而是从真实使用里**自己长出来的**。每条指针的键，是一次真实成功的查询；每条边，是一次真实发生过的联想。所以涌现出来的这张网，直接编码的就是你客观的查找习惯——你常走的路被反复加固成最短的捷径，没人走的"分类"既不占成本、也不拖累别的查询。一句话：不替你预设分类，让分类从你怎么用里自己长出来，客观、自适应、越用越准。

(顺带说一句，前面第六节的文档破碎化里也用到了 LLM，但它**只用来定切分的粒度**，不负责给记忆分类——粒度之上"哪块连哪块、先看谁"，照样是由查询涌现出来的，两件事不矛盾。)

## 八、用 KV 缓存进一步加速查找 【探索中，目前还没做】

> 先把话说在前面：**这一节是个新想法，当前代码里还没有。** 现在的"越用越快"靠的是上面第三节那套学习型指针 + 染色 + 上下文置换。下面这层 KV 缓存复用，是想在它之上再加一档更激进的加速。

**Q：你提到 KV 缓存，这是个什么思路？**

大致是这样：把上下文的片段做成"内容 + 一个预置问题"的形式，并且**连同这段内容在模型里算出来的 KV 缓存一起存下来**。检索的时候，拿用户当前的问题去和这些预置问题匹配，匹配上了，就直接复用那个片段算好的 KV 缓存。这样既不像 RAG 那样把内容切碎丢信息，又省掉了"对这段内容重新跑一遍注意力"的计算。

**Q：这跟你现有的 Q/K 指针有啥不一样？**

层次不同。现在的指针，复用的是**"该看哪段记忆"这个检索结果**；而 KV 缓存想再进一步，复用**"这段内容在模型里已经算到一半的中间状态"**。前者帮你省掉找的功夫，后者连算的功夫也省了一截。两者能叠在一起：指针先把你带到那个片段，KV 缓存让这个片段"不用重算"就进场。

**Q：为什么说它能补 RAG 的不精确？**

RAG 是拿你的问题去匹配"内容块"，块可能被切坏、可能选错。这套是拿你的问题去匹配"历史上的问题"，匹配上就把当时完整的 KV 调回来，信息一点不丢。打个比方：RAG 是每次都重新翻一遍书，这套是"记得以前读过、连当时读到哪、理解成什么样，一起想起来"。

**Q：用现在的托管 API(比如 DeepSeek 的接口)能直接做吗？**

不能直接做——这是它最大的门槛。一般的托管聊天 API **不会把模型内部的 KV 缓存暴露给你**，你既拿不到那段算好的 Key/Value，也没法把它塞回下一次请求。所以这套加速对部署形态是有要求的，只有两条路能走通：要么**上游配合**，让模型服务方在 API 层支持 KV 缓存的导出和注入(比"前缀缓存"更进一步，能按需复用任意片段)；要么**用可控的本地模型**，自己把模型握在手里、完全掌控 KV 缓存的读写。说白了，这不是纯应用层能独立交付的东西——要么等上游开口子，要么自己跑模型。这也是它现在只能算"探索中"的根本原因。

**Q：还有哪些没想清楚的？**

KV 缓存怎么存、跨上下文位置复用 KV 在数值上到底有没有效(位置编码和前缀依赖是个坎)、预置问题怎么生成、命中率多少、缓存怎么保持一致——这些都还在概念阶段。好在**当前版本不靠它也能正常跑**，它是未来的加速选项，不是已经交付的能力。

## 九、还有哪些没做好(实话实说)

- **索引暴涨的最优淘汰**：目前就是淘汰长期没命中、也没被打过反例的低价值索引，更好的策略还在试。
- **文档破碎化**：真机只验过结构清晰、断句分明的中文技术文档。换成日志、代码、没有句号的大段连写，切点可能不理想——那种情况只能靠尺寸兜底保证不出错，但保证不了切得漂亮。
- **KV 缓存加速**(第八节)：还是个想法，没落地。
- **前端面板、打包、跨平台**：后端这套记忆机制已经全部落地、`cargo test --workspace` 全绿；但面板接真数据、出包流程、跨平台还在推进。
- 最后一条贯穿全局的原则：**所有这些加速结构都只是"这一次的副产品"，它们失效了不影响正确性，只影响速度**——最坏情况，无非退化成让 LLM 一段段读原文的暴力全量扫描，慢，但一定能找到。

## 十、架构和技术栈，一眼扫过

六个 crate，单向依赖，一条公理贯穿：一切能力都是"执行器"，Agent 循环是唯一的脊柱。

```
app(gui)  Agent 循环脊柱 + 执行器注册表 + Tauri + 前端(SolidJS)
├── memory   分层检索(RAG → 精确层飞轮)+ 上下文置换   ← 本文主角
├── learn    飞轮:收集 → 聚类压缩 + 永久目标调度
├── safety   沙箱 / 路径分级 / 风险 / 三种授权
├── llm      LLM 通信:流式 / reasoning / 工具解析 / 嵌入
└── core     共享类型:结论模型 + 执行器 trait(零依赖)
```

记忆这块用的：**redb** 做单文件持久化(节点、边都是写穿落库)；**arroy + LMDB** 做磁盘原生的向量索引(向量走 mmap，不全堆进内存)；**candle 跑 multilingual-e5-small** 做本地嵌入(另外留了个远程 OpenAI 兼容的口子)。正文和向量两头都做到了磁盘原生，所以内存占用只跟"当前工作集"挂钩，跟你攒了多少总记忆没关系——这正是"记忆可以无限大"能成立的前提。
